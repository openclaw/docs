Официальный плагин `codex` выполняет встроенные агентские ходы OpenAI через Codex
app-server вместо встроенной среды выполнения OpenClaw. Codex управляет
низкоуровневым сеансом агента: нативным возобновлением потока, нативным продолжением инструментов,
нативной Compaction и выполнением в app-server. OpenClaw по-прежнему управляет каналами
чата, файлами сеансов, выбором модели, динамическими инструментами OpenClaw, подтверждениями,
доставкой медиафайлов и видимой копией расшифровки.

Используйте канонические ссылки на модели OpenAI, например `openai/gpt-5.6-sol`. Не настраивайте
устаревшие ссылки Codex GPT; задавайте порядок аутентификации агента OpenAI в `auth.order.openai`.
Устаревшие идентификаторы профилей аутентификации Codex и записи устаревшего порядка аутентификации Codex
исправляются командой `openclaw doctor --fix`.

Если политика среды выполнения провайдера/модели не задана или имеет значение `auto`, один только префикс `openai/*`
никогда не выбирает эту среду выполнения. OpenAI может неявно выбрать Codex только для
точного официального HTTPS-маршрута Platform Responses или ChatGPT Responses без
пользовательского переопределения запроса. См.
[Неявная агентская среда выполнения OpenAI](/ru/providers/openai#implicit-agent-runtime).
Если Codex получает управление аутентификацией до того, как определена маршрутизация Platform или ChatGPT, OpenClaw
по-прежнему требует, чтобы каждый потенциальный маршрут явно заявлял совместимость с Codex. Одно лишь
нативное управление аутентификацией никогда не позволяет обойти эту проверку маршрута.

Когда песочница OpenClaw не активна, OpenClaw запускает потоки Codex app-server
с включённым нативным режимом кода Codex (режим «только код» по умолчанию остаётся выключенным), поэтому
нативные возможности рабочей области и кода остаются доступны наряду с динамическими инструментами
OpenClaw, направляемыми через мост app-server `item/tool/call`. Активная
песочница OpenClaw или ограничительная политика инструментов полностью отключает нативный режим кода,
если вы явно не включили экспериментальный путь exec-server песочницы.

При значении `tools.exec.host: "auto"` по умолчанию и отсутствии активной песочницы OpenClaw
Codex также получает инструменты `node_exec` и `node_process` для выполнения команд на сопряжённых
узлах. Нативная оболочка остаётся на хосте и в рабочей области Codex app-server
(локально на Gateway для развёртывания stdio по умолчанию); `node_exec` выбирает узел по
имени или идентификатору, при этом политика подтверждений узлов OpenClaw продолжает действовать.

Эта нативная функция Codex отличается от
[режима кода OpenClaw](/ru/reference/code-mode) — включаемой по желанию среды выполнения QuickJS-WASI
для обычных запусков OpenClaw с другой формой входных данных `exec`. Общее описание
разделения моделей, провайдеров и сред выполнения см. в разделе
[Среды выполнения агентов](/ru/concepts/agent-runtimes): `openai/gpt-5.6-sol` — ссылка на модель,
`codex` — среда выполнения, а Telegram, Discord, Slack или другой
канал — интерфейс связи.

## Требования

- Установлен официальный плагин `@openclaw/codex`. Добавьте `codex` в
  `plugins.allow`, если в конфигурации используется список разрешённых элементов.
- Codex app-server версии `0.143.0` или новее. По умолчанию плагин управляет совместимым
  исполняемым файлом, поэтому команда `codex` в `PATH` не влияет на обычный
  запуск.
- Аутентификация Codex через `openclaw models auth login --provider openai`, учётная запись
  app-server, уже находящаяся в домашнем каталоге Codex агента, или
  явный профиль аутентификации Codex с ключом API.

Сведения о приоритетах аутентификации, изоляции окружения, пользовательских командах app-server,
обнаружении моделей и полном списке полей конфигурации см. в
[справочнике по среде выполнения Codex](/ru/plugins/codex-harness-reference).

## Быстрый старт

Установите официальный плагин, затем войдите с помощью OAuth Codex:

```bash
openclaw plugins install @openclaw/codex
openclaw models auth login --provider openai
```

Включите плагин `codex` и выберите агентскую модель OpenAI:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
}
```

Если в конфигурации используется `plugins.allow`, также добавьте туда `codex`:

```json5
{
  plugins: {
    allow: ["codex"],
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

После изменения конфигурации плагина перезапустите Gateway. Если у чата уже есть
сеанс, сначала выполните `/new` или `/reset`, чтобы для следующего хода среда выполнения
была определена по текущей конфигурации.

## Совместное использование потоков с Codex Desktop и CLI

Значение `appServer.homeScope: "agent"` по умолчанию изолирует каждого агента OpenClaw от
нативного состояния Codex оператора. Чтобы владелец мог просматривать и управлять
теми же нативными потоками, которые отображаются в Codex Desktop и Codex CLI, включите
использование пользовательского домашнего каталога Codex:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            homeScope: "user",
          },
        },
      },
    },
  },
}
```

Режим пользовательского домашнего каталога поддерживает локальный управляемый процесс stdio или общий транспорт
через Unix-сокет. Он использует `$CODEX_HOME`, если значение задано, и `~/.codex` в противном случае, включая
нативную аутентификацию Codex, конфигурацию, плагины и хранилище потоков этого домашнего каталога. OpenClaw
не внедряет профиль аутентификации OpenClaw в этот app-server.

Ходы владельца получают инструмент `codex_threads`: просмотр списка, поиск, чтение, разветвление, переименование,
архивация и восстановление нативных потоков. Разветвите поток, чтобы продолжить его в
OpenClaw; ветвь присоединяется к текущему сеансу OpenClaw и остаётся
видимой другим нативным клиентам Codex. Для архивации требуется явное
подтверждение того, что поток закрыт в других клиентах. Если также включён
надзор, для полей расшифровки и изменений требуется соответствующее явное включение
`supervision.allowRawTranscripts` или `supervision.allowWriteControls`.

Не возобновляйте и не изменяйте один и тот же поток параллельно через независимые управляемые
App Server с stdio. Codex координирует активных авторов в пределах одного App Server, но не
между отдельными процессами. Для обычных сеансов stdio с пользовательским домашним каталогом
безопасным способом совместной работы является разветвление.

Одного `appServer.homeScope: "user"` недостаточно для включения каталога парка. Используйте
`supervision.enabled: true`, если хотите, чтобы нативные сеансы отображались на
боковой панели OpenClaw. Надзор использует отдельное подключение; при отсутствии
явных параметров подключения `appServer` оно по умолчанию использует управляемый
stdio с пользовательским домашним каталогом, тогда как обычная среда выполнения остаётся привязанной к агенту. Явные
параметры `appServer` учитываются обоими путями. Явно задайте `homeScope: "user"`,
как показано выше, если обычная среда выполнения также должна совместно использовать нативное состояние.

## Надзор за сеансами Codex

Тот же плагин `codex` может отображать неархивированные сеансы Codex с компьютера Gateway
и сопряжённых узлов, для которых это явно разрешено. Для сохранённого или простаивающего локального сеанса Gateway можно
создать привязанный к модели чат, зеркально отражающий его ограниченную сохранённую историю сообщений пользователя и ассистента.
Его закрытая привязка использует подключение надзора для получения нативного
снимка состояния, канонической ветви и последующих ходов, тогда как обычные сеансы Codex остаются
привязанными к агенту. При первом каноническом запуске используются в точности та модель и тот провайдер, которые
Codex возвращает для ветви снимка состояния. При последующих возобновлениях выбор остаётся за нативной
конфигурацией Codex; внешняя модель OpenClaw и цепочка резервных вариантов никогда её не заменяют.
Сохранённые и простаивающие строки можно архивировать после явного подтверждения
отсутствия других исполнителей. Для активных источников нельзя создать ветвь или выполнить архивацию; существующий
чат под надзором всё ещё можно открыть. Сеансы сопряжённых узлов остаются доступными только в виде метаданных.

Сведения о настройке, правилах ветвления, ограничениях сопряжённых узлов, раскрытии метаданных
и устранении неполадок см. в разделе [Надзор за сеансами Codex](/plugins/codex-supervision).

Для порядка «сначала подписка, затем резервный ключ API» предпочитайте `auth.order.openai`.
Существующие устаревшие идентификаторы профилей аутентификации Codex и устаревший порядок аутентификации Codex
являются устаревшим состоянием, обрабатываемым только doctor; не создавайте новые устаревшие ссылки Codex GPT.

```json5
{
  auth: {
    order: {
      openai: ["openai:user@example.com", "openai:api-key-backup"],
    },
  },
}
```

Для эффективного маршрута, совместимого с Codex, оба указанных выше профиля остаются кандидатами
для одного запуска Codex. Порядок профилей выбирает учётные данные, а не среду выполнения.
Изменение порядка аутентификации не делает пользовательский маршрут, маршрут Completions, HTTP-маршрут или
маршрут с переопределённым запросом совместимым с Codex.

### Compaction

Не задавайте `compaction.model` или `compaction.provider` для агентов,
использующих Codex. Codex выполняет Compaction через собственное нативное состояние потока app-server, поэтому
во время выполнения OpenClaw игнорирует эти локальные переопределения суммаризатора, а
`openclaw doctor --fix` удаляет их, когда агент использует Codex.

Lossless по-прежнему поддерживается как механизм контекста для сборки, приёма данных и
обслуживания вокруг ходов Codex и настраивается через
`plugins.slots.contextEngine: "lossless-claw"` и
`plugins.entries.lossless-claw.config.summaryModel`, а не через
`agents.defaults.compaction.provider`. `openclaw doctor --fix` переносит
старую форму `compaction.provider: "lossless-claw"` в слот механизма
контекста Lossless, когда активной средой выполнения является Codex, но Compaction по-прежнему
управляется нативным Codex. Нативная среда выполнения app-server поддерживает механизмы контекста,
которым требуется сборка перед формированием запроса; универсальные серверные части CLI, включая `codex-cli`,
не предоставляют такую возможность хоста.

Для агентов, использующих Codex, `/compact` запускает нативную Compaction Codex app-server
в привязанном потоке. OpenClaw не ожидает завершения,
не устанавливает тайм-аут OpenClaw, не перезапускает общий app-server и не переключается на
механизм контекста или публичный суммаризатор OpenAI. Если нативная привязка потока Codex
отсутствует или устарела, команда завершается с ошибкой вместо незаметного
переключения серверной части Compaction.

Оставшаяся часть этой страницы описывает схему развёртывания, маршрутизацию с завершением при ошибке, политику
подтверждений guardian, нативные плагины Codex и Computer Use. Полные списки
параметров, значений по умолчанию, перечислений, настройки обнаружения, изоляции окружения, тайм-аутов и
полей транспорта app-server см. в
[справочнике по среде выполнения Codex](/ru/plugins/codex-harness-reference).

## Проверка среды выполнения Codex

Используйте `/status` в чате, где ожидается Codex. Для хода агента OpenAI,
использующего Codex, отображается:

```text
Среда выполнения: OpenAI Codex
```

Затем проверьте состояние Codex app-server:

```text
/codex status
/codex models
```

`/codex status` сообщает о подключении к app-server, учетной записи, лимитах запросов, серверах MCP
и навыках. `/codex models` выводит актуальный каталог Codex app-server
для среды выполнения и учетной записи. Если результат `/status` оказался неожиданным, см.
[Устранение неполадок](#troubleshooting).

## Маршрутизация и выбор модели

Не смешивайте ссылки на провайдеров и политику среды выполнения:

- Используйте `openai/gpt-*` для канонического выбора модели OpenAI. Один лишь префикс
  никогда не выбирает Codex.
- Если среда выполнения не задана или указано `auto`, Codex может быть выбран
  неявно только для точного официального HTTPS-маршрута Platform Responses
  или ChatGPT Responses без заданного пользователем переопределения запроса.
- Не используйте устаревшие ссылки Codex GPT в конфигурации; выполните `openclaw doctor --fix`,
  чтобы исправить устаревшие ссылки и закрепленные в сеансах неактуальные маршруты.
- `agentRuntime.id: "codex"` делает Codex обязательным с отказом при ошибке
  для совместимого маршрута. Это не делает несовместимый фактический маршрут совместимым.
- `agentRuntime.id: "openclaw"` явно переводит провайдера или модель на встроенную
  среду выполнения OpenClaw, когда это сделано намеренно.
- `/codex ...` управляет нативными диалогами Codex app-server из чата.
- ACP/acpx — отдельный путь внешней среды выполнения. Используйте его только тогда, когда пользователь
  запрашивает ACP/acpx или адаптер внешней среды выполнения.

| Отказ при недоступности Codex                   | Для провайдера или модели задано `agentRuntime.id: "codex"`                                                                | Ход завершается ошибкой вместо перехода на встроенную среду | Используйте для развертываний только с Codex             |
| Прямой трафик с ключом API OpenAI через OpenClaw | Для провайдера или модели задано `agentRuntime.id: "openclaw"` и используется обычная аутентификация OpenAI                                      | `/status` показывает среду выполнения OpenClaw        | Используйте только при намеренном выборе OpenClaw      |
| Устаревшая конфигурация                         | устаревшие ссылки Codex GPT                                                                                       | `openclaw doctor --fix` перезаписывает их     | Не создавайте новую конфигурацию таким способом           |
| Адаптер Codex для ACP/acpx                      | ACP `sessions_spawn({ runtime: "acp" })`                                                                    | Состояние задачи или сеанса ACP                 | Отдельно от нативной среды выполнения Codex         |

Для `agents.defaults.imageModel` действует то же разделение по префиксам. Используйте `openai/gpt-*`
для обычного маршрута OpenAI, а `codex/gpt-*` — только когда анализ изображений
должен выполняться в рамках ограниченного хода Codex app-server. Doctor перезаписывает устаревшие
ссылки Codex GPT в `openai/gpt-*`.

## Схемы развертывания

### Базовое развертывание Codex

Используйте конфигурацию быстрого старта для модели OpenAI, чей фактический официальный
HTTPS-маршрут допускает неявный выбор Codex:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
}
```

### Развертывание с несколькими провайдерами

Оставьте Claude агентом по умолчанию и добавьте именованный агент Codex:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-6",
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.6-sol",
      },
    ],
  },
}
```

Агент `main` использует свой обычный путь провайдера. Агент `codex` использует Codex
app-server, пока его фактический маршрут OpenAI остается совместимым; добавьте явное
`agentRuntime.id: "codex"` на уровне модели, если это должно быть обязательным
требованием с отказом при ошибке.

### Развертывание Codex с отказом при ошибке

Допустимый точный официальный HTTPS-маршрут OpenAI может разрешаться в Codex, когда
доступен встроенный плагин. Добавьте явную политику среды выполнения для заданного
правила отказа при ошибке:

```json5
{
  models: {
    providers: {
      openai: {
        agentRuntime: {
          id: "codex",
        },
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.6-sol",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

При принудительном использовании Codex OpenClaw сразу завершает работу с ошибкой, если фактический маршрут не объявлен
совместимым с Codex, плагин отключен, app-server слишком стар
или не может запуститься.

## Политика app-server

По умолчанию плагин локально запускает управляемый OpenClaw бинарный файл Codex
с транспортом stdio. Задавайте `appServer.command` только для намеренного запуска
другого исполняемого файла. Используйте транспорт WebSocket, только если app-server
уже запущен в другом месте:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://gateway-host:39175",
            authToken: "${CODEX_APP_SERVER_TOKEN}",
          },
        },
      },
    },
  },
}
```

Локальные сеансы app-server через stdio по умолчанию используют доверенную модель
локального оператора: `approvalPolicy: "never"`, `approvalsReviewer: "user"` и
`sandbox: "danger-full-access"`. Если локальные требования Codex запрещают такую
неявную модель YOLO, OpenClaw вместо нее выбирает разрешения, допускаемые Guardian.
Когда для сеанса активна песочница OpenClaw, OpenClaw
отключает для этого хода нативный Code Mode Codex, пользовательские серверы MCP и выполнение
плагинов на основе приложений, а не полагается на песочницу Codex на стороне хоста.
Доступ к оболочке вместо этого осуществляется через динамические инструменты на базе песочницы OpenClaw,
такие как `sandbox_exec` и `sandbox_process`, когда доступны обычные инструменты exec/process.

Для нативной автоматической проверки Codex используйте нормализованный режим exec OpenClaw до
выхода из песочницы или предоставления дополнительных разрешений:

```json5
{
  tools: {
    exec: {
      mode: "auto",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

Для сеансов Codex app-server `tools.exec.mode: "auto"` сопоставляется с разрешениями,
проверяемыми Codex Guardian: обычно `approvalPolicy: "on-request"`,
`approvalsReviewer: "auto_review"` и `sandbox: "workspace-write"`, когда
локальные требования допускают эти значения. В режиме `tools.exec.mode: "auto"`
OpenClaw не сохраняет устаревшие небезопасные переопределения Codex `approvalPolicy: "never"` или
`sandbox: "danger-full-access"`; используйте `tools.exec.mode: "full"`
для намеренной работы Codex без подтверждений. Устаревшая предустановка
`plugins.entries.codex.config.appServer.mode: "guardian"` по-прежнему
работает, но `tools.exec.mode: "auto"` является нормализованной поверхностью OpenClaw.

Сравнение режимов с подтверждениями выполнения на хосте и разрешениями ACPX
см. в разделе [Режимы разрешений](/ru/tools/permission-modes). Все поля
app-server, порядок аутентификации, изоляция окружения и поведение тайм-аутов
описаны в [справочнике по среде выполнения Codex](/ru/plugins/codex-harness-reference).

## Команды и диагностика

Плагин `codex` регистрирует `/codex` как slash-команду в любом канале,
поддерживающем текстовые команды OpenClaw.

Для нативного выполнения и управления требуется владелец или клиент Gateway с правом `operator.admin`:
привязка или возобновление веток, отправка или остановка ходов,
изменение модели, быстрого режима или состояния разрешений, уплотнение или проверка,
а также отсоединение привязки. Другим авторизованным отправителям доступны только команды
просмотра состояния, справки, учетной записи, модели, ветки, сервера MCP, навыка и привязки.

Распространенные формы:

- `/codex status` проверяет подключение к app-server, модели, учетную запись, лимиты
  запросов, серверы MCP и навыки.
- `/codex models` выводит актуальные модели Codex app-server.
- `/codex threads [filter]` выводит недавние ветки Codex app-server.
- `/codex resume <thread-id>` привязывает текущий сеанс OpenClaw к
  существующей ветке Codex.
- `/codex bind [thread-id] [--cwd <path>] [--model <model>] [--provider <provider>]`
  привязывает текущий чат.
- `/codex detach` (или `/codex unbind`) отсоединяет текущую привязку.
- `/codex binding` описывает текущую привязку.
- `/codex stop` останавливает активный ход; `/codex steer <text>` корректирует его.
- `/codex model <model>`, `/codex fast [on|off|status]` и
  `/codex permissions [default|yolo|status]` изменяют состояние отдельного диалога.
- `/codex compact` запрашивает у Codex app-server уплотнение привязанной ветки.
- `/codex review` запускает нативную проверку Codex для привязанной ветки.
- `/codex diagnostics [note]` запрашивает подтверждение перед отправкой отзыва о Codex
  для привязанной ветки.
- `/codex account` показывает состояние учетной записи и лимитов запросов.
- `/codex mcp` выводит состояние серверов MCP Codex app-server.
- `/codex skills` выводит навыки Codex app-server.
- `/codex plugins list`, `/codex plugins enable <name>` и
  `/codex plugins disable <name>` управляют настроенными нативными плагинами Codex.
- `/codex computer-use [status|install]` управляет Codex Computer Use.
- `/codex help` выводит полное дерево команд.

Для большинства обращений в службу поддержки начните с `/diagnostics [note]` в
диалоге, где произошла ошибка. Эта команда создаёт один диагностический
отчёт Gateway, а для сеансов среды Codex запрашивает разрешение на отправку
соответствующего пакета обратной связи Codex. Модель конфиденциальности и
поведение в групповых чатах описаны в разделе
[Экспорт диагностики](/ru/gateway/diagnostics). Используйте `/codex diagnostics [note]`, только если вам нужен именно
пакет обратной связи Codex для текущей прикреплённой ветки без
полного диагностического пакета Gateway.

### Локальная проверка веток Codex

Зачастую самый быстрый способ исследовать неудачный запуск Codex — напрямую
открыть нативную ветку Codex:

```bash
codex resume <thread-id>
```

Получите идентификатор ветки из завершённого ответа `/diagnostics`, `/codex binding`
или `/codex threads [filter]`.

Механизм отправки и границы диагностики на уровне среды выполнения описаны в разделе
[Среда выполнения Codex harness](/ru/plugins/codex-harness-runtime#codex-feedback-upload).

### Порядок аутентификации

В стандартном домашнем каталоге отдельного агента аутентификация выбирается в следующем порядке:

1. Упорядоченные профили аутентификации OpenAI для агента, предпочтительно в
   `auth.order.openai`. Выполните `openclaw doctor --fix`, чтобы перенести старые устаревшие
   идентификаторы профилей аутентификации Codex и устаревший порядок аутентификации Codex.
2. Существующая учётная запись app-server в домашнем каталоге Codex этого агента.
3. Только для локальных запусков app-server через stdio: `CODEX_API_KEY`, затем
   `OPENAI_API_KEY`, если учётная запись app-server отсутствует, но аутентификация OpenAI
   по-прежнему требуется.

Когда OpenClaw обнаруживает профиль аутентификации Codex с подпиской ChatGPT,
он удаляет `CODEX_API_KEY` и `OPENAI_API_KEY` из порождённого дочернего
процесса Codex. Благодаря этому API-ключи уровня Gateway остаются доступными
для эмбеддингов или прямых моделей OpenAI, но нативные обращения Codex app-server
не тарифицируются случайно через API. Явные профили Codex с API-ключом и
локальный резервный вариант с ключом из окружения для stdio используют вход
через app-server вместо унаследованного окружения дочернего процесса.
Соединения app-server через WebSocket не получают резервный API-ключ
из окружения Gateway; используйте явный профиль аутентификации или собственную
учётную запись удалённого app-server.

Если профиль подписки достигает лимита использования Codex, OpenClaw записывает
время сброса, когда Codex его сообщает, и пробует следующий упорядоченный профиль
аутентификации для того же запуска Codex. После наступления времени сброса профиль
подписки снова становится доступным без изменения выбранной модели `openai/gpt-*`
или среды выполнения Codex.

Если настроены нативные плагины Codex, OpenClaw устанавливает или обновляет
эти плагины через подключённый app-server, прежде чем предоставлять ветке Codex
доступ к приложениям, принадлежащим плагинам. `app/list` остаётся источником
истины для идентификаторов приложений, доступности и метаданных, но решение
о включении для конкретной ветки принимает OpenClaw: если политика разрешает
указанное доступное приложение, OpenClaw отправляет `thread/start.config.apps[appId].enabled = true`, даже если
`app/list` в данный момент сообщает, что приложение отключено.
Этот путь не инициирует установку приложений для неизвестных идентификаторов;
OpenClaw активирует только плагины из маркетплейса с `plugin/install`, а затем
обновляет инвентаризацию.

### Изоляция окружения

При локальных запусках app-server через stdio OpenClaw задаёт для `CODEX_HOME`
отдельный каталог агента, поэтому конфигурация Codex, файлы аутентификации и
учётной записи, кеш и данные плагинов, а также состояние нативных веток по умолчанию
не читаются из личного `~/.codex` оператора и не записываются туда.
OpenClaw сохраняет обычное значение процесса `HOME`; подпроцессы,
запускаемые Codex, по-прежнему могут находить конфигурацию и токены в домашнем
каталоге пользователя, а Codex может обнаруживать общие записи `$HOME/.agents/skills`
и `$HOME/.agents/plugins/marketplace.json`. При использовании `appServer.homeScope: "user"` OpenClaw вместо этого
использует нативный пользовательский домашний каталог Codex и существующую
в нём учётную запись, не внедряя профиль аутентификации OpenClaw.

Если развёртыванию требуется дополнительная изоляция окружения, добавьте эти
переменные в `appServer.clearEnv`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            clearEnv: ["CODEX_API_KEY", "OPENAI_API_KEY"],
          },
        },
      },
    },
  },
}
```

`appServer.clearEnv` влияет только на порождённый дочерний процесс Codex app-server.
При нормализации локального запуска OpenClaw удаляет из этого списка
`CODEX_HOME` и `HOME`: `CODEX_HOME` продолжает указывать
на выбранную область агента или пользователя, а `HOME` наследуется,
чтобы подпроцессы могли использовать обычное состояние домашнего каталога
пользователя.

### Динамические инструменты и веб-поиск

По умолчанию динамические инструменты Codex загружаются через `searchable`.
OpenClaw не предоставляет динамические инструменты, дублирующие нативные операции
Codex с рабочей областью: `read`, `write`, `edit`, `apply_patch`, `exec`, `process`, `update_plan`,
`tool_call`, `tool_describe`, `tool_search` и `tool_search_code`.
Большинство остальных инструментов интеграции OpenClaw, включая обмен сообщениями,
медиа, cron, браузер, узлы, Gateway и `heartbeat_respond`, доступны через поиск
инструментов Codex в пространстве имён `openclaw`, что уменьшает
начальный контекст модели.

Инструменты с пометкой `catalogMode: "direct-only"`, включая инструмент OpenClaw
`computer`, используют вместо этого пространство имён `openclaw_direct`.
Codex рассматривает это пространство имён как `DirectModelOnly`, поэтому такие
инструменты остаются непосредственно видимыми модели в обычных ветках и ветках
только с режимом кода, не проходя через вложенные вызовы Code Mode
`tools.*`.

Если поиск включён и управляемый провайдер не выбран, веб-поиск по умолчанию
использует размещённый инструмент Codex `web_search`. Нативный размещённый
поиск и управляемый динамический инструмент OpenClaw `web_search`
взаимоисключающие, поэтому управляемый поиск не может обойти нативные ограничения
доменов. OpenClaw использует управляемый инструмент, если размещённый поиск
недоступен, явно отключён или заменён выбранным управляемым провайдером.
OpenClaw оставляет отдельное расширение Codex `web.run` отключённым,
поскольку рабочий трафик app-server отклоняет его пользовательское пространство
имён `web`. `tools.web.search.enabled: false` отключает оба пути, как и запуски
только с LLM, в которых инструменты отключены. Codex рассматривает
`"cached"` как предпочтение и преобразует его в действующий внешний
доступ для неограниченных обращений app-server. Автоматический управляемый
резервный вариант завершается отказом, если заданы нативные `allowedDomains`,
чтобы список разрешений нельзя было обойти. При постоянном изменении эффективной
политики поиска привязанная ветка Codex заменяется перед следующим обращением;
временные ограничения для отдельного обращения используют временную ограниченную
ветку и сохраняют существующую привязку для последующего возобновления.

Ответы `sessions_yield` и ответы только с источниками инструмента сообщений
остаются прямыми, поскольку это контракты управления обращением.
`sessions_spawn` остаётся доступным для поиска, чтобы нативный
`spawn_agent` Codex оставался основным механизмом субагентов Codex, а явное
делегирование через OpenClaw или ACP по-прежнему было доступно через пространство
имён динамических инструментов `openclaw`. Инструкции по совместной работе
Heartbeat предписывают Codex искать `heartbeat_respond` перед завершением обращения
Heartbeat, если инструмент ещё не загружен.

Задавайте `codexDynamicToolsLoading: "direct"` только при подключении к пользовательскому
Codex app-server, который не может искать отложенные динамические инструменты,
или при отладке полной полезной нагрузки инструментов.

### Поля конфигурации

Поддерживаемые поля верхнего уровня плагина Codex:

| Поле                       | Значение по умолчанию | Назначение                                                                                |
| -------------------------- | -------------- | ---------------------------------------------------------------------------------------- |
| `codexDynamicToolsLoading` | `"searchable"` | Используйте `"direct"`, чтобы поместить динамические инструменты OpenClaw непосредственно в начальный контекст инструментов Codex. |
| `codexDynamicToolsExclude` | `[]`           | Дополнительные имена динамических инструментов OpenClaw, которые следует исключить из обращений Codex app-server. |
| `codexPlugins`             | отключено       | Поддержка нативных плагинов и приложений Codex для перенесённых курируемых плагинов, установленных из исходного кода. |
| `supervision`              | отключено       | Каталог неархивных нативных сеансов, продолжение локальной ветки и политика инструментов агента. |

Поддерживаемые поля `appServer`:

профиль разрешений мог запустить управляемый сетевой доступ Codex. По умолчанию OpenClaw
создаёт устойчивое к коллизиям имя профиля `openclaw-network-<fingerprint>`
на основе содержимого профиля; используйте `profileName` только тогда, когда требуется
стабильное локальное имя.

```json5
{
  plugins: {
    entries: {
      codex: {
        config: {
          appServer: {
            sandbox: "workspace-write",
            networkProxy: {
              enabled: true,
              domains: {
                "api.openai.com": "allow",
                "blocked.example.com": "deny",
              },
              unixSockets: {
                "/tmp/proxy.sock": "allow",
                "/tmp/blocked.sock": "none",
              },
              allowUpstreamProxy: true,
              proxyUrl: "http://127.0.0.1:3128",
            },
          },
        },
      },
    },
  },
}
```

Если обычной средой выполнения app-server была бы `danger-full-access`, включение
`networkProxy` использует для создаваемого профиля разрешений доступ к
файловой системе в стиле рабочей области: управляемое Codex ограничение сети
реализовано через изолированную сетевую среду, поэтому профиль полного доступа
не защищал бы исходящий трафик. Записи доменов используют `allow` или
`deny`; записи сокетов Unix используют значения Codex
`allow` или `none`.

### Динамические тайм-ауты вызовов инструментов

Вызовы динамических инструментов, принадлежащих OpenClaw, ограничиваются
независимо от `appServer.requestTimeoutMs`: для запросов Codex `item/tool/call` по
умолчанию используется сторожевой таймер OpenClaw на 90 секунд. Положительный
аргумент `timeoutMs` для отдельного вызова увеличивает или уменьшает
бюджет именно этого инструмента, но не более 600000 мс. Инструмент
`image_generate` использует `agents.defaults.imageGenerationModel.timeoutMs`, если вызов инструмента не
задаёт собственный тайм-аут, а в остальных случаях — стандартный тайм-аут
генерации изображений в 120 секунд. Инструмент анализа медиа
`image` использует `tools.media.image.timeoutSeconds` или свой стандартный тайм-аут
для медиа в 60 секунд; при анализе изображений этот тайм-аут применяется к
самому запросу и не сокращается из-за ранее выполненной подготовительной
работы. При истечении тайм-аута OpenClaw прерывает сигнал инструмента, если это
поддерживается, и возвращает Codex ответ о неудачном выполнении динамического
инструмента, чтобы ход мог продолжиться, а сеанс не оставался в
`processing`. Этот сторожевой таймер задаёт внешний динамический бюджет
`item/tool/call`; тайм-ауты запросов конкретных провайдеров действуют внутри
этого вызова и сохраняют собственную семантику.

После того как Codex принимает ход, а также после ответа OpenClaw на относящийся
к ходу запрос app-server, среда ожидает, что Codex продолжит выполнение текущего
хода и в итоге завершит нативный ход с помощью `turn/completed`. Если
app-server не проявляет активности в течение `appServer.turnCompletionIdleTimeoutMs`, OpenClaw по
возможности прерывает ход Codex, регистрирует диагностический тайм-аут и
освобождает полосу сеанса OpenClaw, чтобы последующие сообщения чата не
помещались в очередь за устаревшим нативным ходом. Большинство незавершающих
уведомлений для того же хода отключают этот короткий сторожевой таймер,
поскольку Codex подтвердил, что ход всё ещё активен.

Для передачи управления инструментам используется более длительный бюджет
бездействия после инструмента: после возврата OpenClaw ответа
`item/tool/call`, после завершения нативных элементов инструментов, таких как
`commandExecution`, после завершений необработанного `custom_tool_call_output`, а также
после необработанного прогресса ассистента после инструмента, завершения
рассуждения или прогресса рассуждения. Защитный механизм использует
`appServer.postToolRawAssistantCompletionIdleTimeoutMs`, если он настроен, а иначе по умолчанию — пять минут; тот же
бюджет также продлевает сторожевой таймер прогресса на период безмолвного
формирования ответа до того, как Codex отправит следующее событие текущего хода.
Глобальные уведомления app-server, например обновления ограничений частоты, не
сбрасывают таймер прогресса бездействующего хода. После завершений рассуждения,
завершений комментариев `agentMessage`, а также необработанного рассуждения
перед инструментом или прогресса ассистента может автоматически последовать
окончательный ответ, поэтому для них используется защита ответа после прогресса,
а полоса сеанса не освобождается немедленно.

Только завершённые элементы `agentMessage`, являющиеся окончательными и не
относящиеся к комментариям, а также необработанные завершения ассистента перед
инструментом активируют освобождение по выводу ассистента: если после этого
Codex бездействует без `turn/completed`, OpenClaw по возможности прерывает
нативный ход и освобождает полосу сеанса. Если в гонке за освобождение побеждает
другой сторожевой механизм хода, OpenClaw всё равно принимает завершённый
окончательный элемент ассистента, когда больше нет активных нативных запросов,
элементов или завершений динамических инструментов, освобождение по выводу
ассистента всё ещё относится к последнему завершённому элементу и отсутствует
более позднее завершение элемента. Это позволяет сохранить окончательный ответ
после завершённой работы инструментов без повторного воспроизведения хода.
Частичные порции данных ассистента, устаревшие более ранние ответы и пустые
более поздние завершения не подходят.

Безопасные для повторного воспроизведения сбои app-server через stdio, включая
тайм-ауты бездействия при завершении хода без признаков ответа ассистента,
инструмента, активного элемента или побочных эффектов, повторяются один раз в
новой попытке app-server. При небезопасных тайм-аутах зависший клиент app-server
всё равно выводится из эксплуатации, а полоса сеанса OpenClaw освобождается;
кроме того, устаревшая привязка нативного потока очищается вместо автоматического
повторного воспроизведения. При тайм-аутах контроля завершения отображается
специальный для Codex текст: в безопасных для повторного воспроизведения случаях
указывается, что ответ может быть неполным, а в небезопасных случаях пользователю
предлагается проверить текущее состояние перед повторной попыткой. Публичные
диагностические данные тайм-аутов включают структурные поля, такие как метод
последнего уведомления app-server, идентификатор, тип и роль элемента
необработанного ответа ассистента, количество активных запросов и элементов, а
также состояние активированного сторожевого механизма; если последнее уведомление
является элементом необработанного ответа ассистента, они также включают
ограниченный предварительный фрагмент текста ассистента. Необработанное
содержимое запроса или инструмента в них не включается.

- `OPENCLAW_CODEX_APP_SERVER_ARGS`

- `OPENCLAW_CODEX_APP_SERVER_APPROVAL_POLICY`
- `OPENCLAW_CODEX_APP_SERVER_SANDBOX`

остальные настройки среды Codex.

## Нативные плагины Codex

Поддержка нативных плагинов Codex использует собственные возможности приложений
и плагинов Codex app-server в том же потоке Codex, что и ход среды OpenClaw.
OpenClaw не преобразует плагины Codex в синтетические динамические инструменты
OpenClaw `codex_plugin_*`.

`codexPlugins` влияет только на сеансы, использующие нативную среду Codex.
Она не влияет на запуски встроенной среды, обычные запуски провайдера OpenAI,
привязки диалогов ACP или другие среды.

Минимальная перенесённая конфигурация:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_destructive_actions: true,
            plugins: {
              "google-calendar": {
                enabled: true,
                marketplaceName: "openai-curated",
                pluginName: "google-calendar",
              },
            },
          },
        },
      },
    },
  },
}
```

Конфигурация приложений потока вычисляется, когда OpenClaw устанавливает сеанс
среды Codex или заменяет устаревшую привязку потока Codex; она не вычисляется
заново при каждом ходе. После изменения `codexPlugins` используйте
`/new`, `/reset` или перезапустите Gateway, чтобы будущие
сеансы среды Codex запускались с обновлённым набором приложений.

Сведения о соответствии требованиям миграции, перечне приложений, политике
разрушительных действий, запросах дополнительных данных и диагностике нативных
плагинов см. в разделе
[Нативные плагины Codex](/ru/plugins/codex-native-plugins).

Доступ к приложениям и плагинам на стороне OpenAI контролируется учётной записью
Codex, в которую выполнен вход, а для рабочих областей Business и Enterprise/Edu
— также настройками приложений рабочей области. Обзор управления учётной записью
и рабочей областью OpenAI см. в разделе
[Использование Codex с вашим тарифным планом ChatGPT](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan).

## Управление компьютером

Для управления компьютером предусмотрено отдельное руководство по настройке:
[Управление компьютером с помощью Codex](/ru/plugins/codex-computer-use).

Кратко: OpenClaw не включает приложение управления рабочим столом в свой состав
и не выполняет действия на рабочем столе самостоятельно. Он подготавливает Codex
app-server, проверяет доступность MCP-сервера `computer-use`, а затем
позволяет Codex управлять нативными вызовами инструментов MCP во время ходов в
режиме Codex.

## Границы среды выполнения

Среда Codex изменяет только низкоуровневый встроенный исполнитель агента.

- Динамические инструменты OpenClaw поддерживаются. Codex просит OpenClaw
  выполнить эти инструменты, поэтому OpenClaw остаётся в цепочке выполнения.
- Нативные оболочка, исправления, MCP и инструменты приложений Codex находятся
  под управлением Codex. OpenClaw может наблюдать за выбранными нативными
  событиями или блокировать их через поддерживаемый ретранслятор, но не
  перезаписывает аргументы нативных инструментов.
- Нативная Compaction находится под управлением Codex. OpenClaw сохраняет
  зеркальную копию стенограммы для истории канала, поиска, `/new`,
  `/reset` и будущего переключения модели или среды, но не заменяет
  Compaction Codex механизмом суммаризации OpenClaw или контекстного движка.
- Генерация медиа, анализ медиа, TTS, подтверждения и вывод инструментов
  обмена сообщениями продолжают использовать соответствующие настройки
  провайдера и модели OpenClaw.
- `tool_result_persist` применяется к результатам инструментов стенограммы,
  принадлежащих OpenClaw, а не к нативным записям результатов инструментов
  Codex.

Сведения об уровнях перехватчиков, поддерживаемых поверхностях V1, обработке
нативных разрешений, управлении очередью, механизмах отправки отзывов Codex и
подробностях Compaction см. в разделе

## Устранение неполадок

**Codex не отображается как обычный провайдер `/model`:** это ожидаемо
для новых конфигураций. Выберите модель `openai/gpt-*`, включите
`plugins.entries.codex.enabled` и проверьте, не исключает ли `plugins.allow`
`codex`.

**OpenClaw использует встроенную среду вместо Codex:** убедитесь, что фактический
маршрут в точности является официальным HTTPS-маршрутом Platform Responses или
ChatGPT Responses, не содержит пользовательского переопределения запроса, а
плагин Codex установлен и включён. Одного префикса `openai/gpt-*`
недостаточно. Для строгой проверки при тестировании задайте для провайдера или
модели `agentRuntime.id: "codex"`; при принудительном использовании Codex возникает
ошибка вместо перехода на резервный вариант, если маршрут или среда несовместимы.

**Среда выполнения OpenAI Codex переходит на путь с ключом API:** соберите
отредактированный фрагмент журнала Gateway, в котором указаны модель, среда
выполнения, выбранный провайдер и сбой. Попросите затронутых участников выполнить
эту команду только для чтения на своём хосте OpenClaw:

```bash
(
  pattern='openai/gpt-5\.[45]|openai[-]codex|agentRuntime(\.id)?|harnessRuntime|Runtime: OpenAI Codex|legacy OpenAI Codex prefix|resolveSelectedOpenAIRuntimeProvider|candidateProvider[": ]+openai|status[": ]+401|Incorrect API key|No API key|api-key path|API-key path|OAuth'

  if ls /tmp/openclaw/openclaw-*.log >/dev/null 2>&1; then
    grep -E -i -n "$pattern" /tmp/openclaw/openclaw-*.log 2>/dev/null || true
  else
    journalctl --user -u openclaw-gateway --since today --no-pager 2>/dev/null \
      | grep -E -i "$pattern" || true
  fi
) | sed -E \
    -e 's/(Authorization: Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(api[_ -]?key[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/(OPENAI_API_KEY[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/sk-[A-Za-z0-9_-]{12,}/sk-[REDACTED]/g' \
    -e 's/[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}/[EMAIL-REDACTED]/g' \
  | tail -200
```

Полезные фрагменты обычно содержат `openai/gpt-5.6-sol` или
`openai/gpt-5.6-luna`, `Runtime: OpenAI Codex`, `agentRuntime.id` или
`harnessRuntime`, `candidateProvider: "openai"`, а также результат
`401`, `Incorrect API key` или `No API key`. После
исправления запуск должен показывать путь OAuth OpenAI вместо обычной ошибки
ключа API OpenAI.

**В конфигурации остались устаревшие ссылки на модели Codex:** выполните
`openclaw doctor --fix`. Doctor заменяет устаревшие ссылки на модели на
`openai/*`, удаляет устаревшие закрепления среды выполнения для сеанса и
всего агента и сохраняет существующие переопределения профилей аутентификации.

**App-server отклонён:** используйте Codex app-server `0.143.0` или
новее. Предварительные выпуски той же версии и версии с суффиксом сборки,
например `0.143.0-alpha.2` или `0.143.0+custom`, отклоняются, поскольку
OpenClaw проверяет стабильную минимальную версию протокола
`0.143.0`.

**`/codex status` не удается подключиться:** проверьте, что плагин `codex`
включен, что `plugins.allow` включает его, если настроен список
разрешений, и что все пользовательские `appServer.command`, `url`, `authToken` или
заголовки допустимы.

**Обнаружение моделей выполняется медленно:** уменьшите
`plugins.entries.codex.config.discovery.timeoutMs` или отключите обнаружение.
См. [справочник по среде выполнения Codex](/ru/plugins/codex-harness-reference#model-discovery).

**Транспорт WebSocket сразу завершается с ошибкой:** проверьте `appServer.url`,
`authToken`, заголовки, а также убедитесь, что удаленный сервер приложений использует ту же версию
протокола сервера приложений Codex.

**Нативные инструменты оболочки или применения исправлений заблокированы с ошибкой `Native hook relay
unavailable`:** поток Codex по-прежнему пытается использовать идентификатор
ретранслятора нативных хуков, который больше не зарегистрирован в OpenClaw. Это проблема
транспорта нативных хуков Codex, а не сбой серверной части ACP, провайдера, GitHub или
команды оболочки. Запустите новый сеанс в затронутом чате с помощью `/new` или `/reset`,
затем повторите безопасную команду. Если она сработает один раз, но следующий вызов нативного
инструмента снова завершится с ошибкой, рассматривайте `/new` только как временный обходной путь: скопируйте
запрос в новый сеанс после перезапуска сервера приложений Codex или
OpenClaw Gateway, чтобы старые потоки были удалены, а регистрации нативных хуков
созданы заново.

**Модель, отличная от Codex, использует встроенную среду выполнения:** это ожидаемо, если только политика
среды выполнения провайдера или модели не направляет ее в другую среду выполнения. Обычные ссылки на провайдеров,
отличных от OpenAI, остаются на стандартном пути своего провайдера в режиме `auto`.

**Computer Use установлен, но инструменты не запускаются:** проверьте
`/codex computer-use status` из нового сеанса. Если инструмент сообщает
`Native hook relay unavailable`, воспользуйтесь описанным выше восстановлением ретранслятора нативных хуков.
См. [Codex Computer Use](/ru/plugins/codex-computer-use#troubleshooting).
