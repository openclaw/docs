Встроенный plugin `codex` позволяет OpenClaw выполнять встроенные ходы агента OpenAI
через Codex app-server вместо встроенного harness OpenClaw.

Используйте Codex harness, когда хотите, чтобы Codex владел низкоуровневой сессией агента:
нативным возобновлением треда, нативным продолжением инструментов, нативной Compaction и
выполнением app-server. OpenClaw по-прежнему владеет чат-каналами, файлами сессий, выбором модели, динамическими инструментами OpenClaw, подтверждениями, доставкой медиа и видимым
зеркалом транскрипта.

Обычная настройка использует канонические ссылки на модели OpenAI, такие как `openai/gpt-5.5`.
Не настраивайте устаревшие ссылки Codex GPT. Поместите порядок аутентификации агента OpenAI
в `auth.order.openai`; старые идентификаторы профилей аутентификации Codex и
старые записи порядка аутентификации Codex являются устаревшим состоянием, которое исправляется
`openclaw doctor --fix`.

Когда песочница OpenClaw не активна, OpenClaw запускает треды Codex app-server
с включенным нативным режимом кода Codex, оставляя при этом режим «только код» по умолчанию выключенным.
Это сохраняет доступность нативного рабочего пространства и возможностей кода Codex, пока
динамические инструменты OpenClaw продолжают работать через мост app-server `item/tool/call`.
Активная изоляция OpenClaw в песочнице и ограничительные политики инструментов полностью отключают нативный режим кода,
если вы не включите экспериментальный путь sandbox exec-server.

Эта нативная функция Codex отделена от
[режима кода OpenClaw](/ru/reference/code-mode), который является opt-in runtime QuickJS-WASI
для обычных запусков OpenClaw с другой формой входных данных `exec`.

Для более общего разделения модели, провайдера и runtime начните с
[Runtime агентов](/ru/concepts/agent-runtimes). Краткая версия:
`openai/gpt-5.5` — это ссылка на модель, `codex` — runtime, а Telegram,
Discord, Slack или другой канал остается поверхностью коммуникации.

## Требования

- OpenClaw с доступным встроенным plugin `codex`.
- Если ваша конфигурация использует `plugins.allow`, включите `codex`.
- Codex app-server `0.125.0` или новее. Встроенный plugin по умолчанию управляет совместимым
  бинарным файлом Codex app-server, поэтому локальные команды `codex` в `PATH` не
  влияют на обычный запуск harness.
- Аутентификация Codex доступна через `openclaw models auth login --provider openai`,
  учетную запись app-server в домашнем каталоге Codex агента или явный профиль аутентификации Codex с API-ключом.

О приоритете аутентификации, изоляции окружения, пользовательских командах app-server, обнаружении моделей
и всех полях конфигурации см.
[справочник Codex harness](/ru/plugins/codex-harness-reference).

## Быстрый старт

Большинству пользователей, которым нужен Codex в OpenClaw, подходит этот путь: войдите с
подпиской ChatGPT/Codex, включите встроенный plugin `codex` и используйте
каноническую ссылку на модель `openai/gpt-*`.

Войдите через Codex OAuth:

```bash
openclaw models auth login --provider openai
```

Включите встроенный plugin `codex` и выберите агентную модель OpenAI:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
    },
  },
}
```

Если ваша конфигурация использует `plugins.allow`, добавьте туда и `codex`:

```json5
{
  plugins: {
    allow: ["codex"],
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

Перезапустите Gateway после изменения конфигурации plugin. Если у существующего чата уже
есть сессия, используйте `/new` или `/reset` перед проверкой изменений runtime, чтобы следующий
ход разрешал harness из текущей конфигурации.

## Конфигурация

Конфигурация из быстрого старта — минимально жизнеспособная конфигурация Codex harness. Задавайте параметры Codex
harness в конфигурации OpenClaw, а CLI используйте только для аутентификации Codex:

| Потребность                                           | Установите                                                                        | Где                                |
| -------------------------------------- | -------------------------------------------------------------------------------- | ---------------------------------- |
| Включить harness                     | `plugins.entries.codex.enabled: true`                                            | Конфигурация OpenClaw                    |
| Сохранить установку plugin из списка разрешенных     | Включите `codex` в `plugins.allow`                                               | Конфигурация OpenClaw                    |
| Направлять ходы агента OpenAI через Codex | `agents.defaults.model` или `agents.list[].model` как `openai/gpt-*`               | Конфигурация агента OpenClaw              |
| Войти через ChatGPT/Codex OAuth       | `openclaw models auth login --provider openai`                                   | Профиль аутентификации CLI                   |
| Добавить резервный API-ключ для запусков Codex      | Профиль API-ключа `openai:*`, указанный после аутентификации по подписке в `auth.order.openai` | Профиль аутентификации CLI + конфигурация OpenClaw |
| Завершаться с ошибкой, когда Codex недоступен  | Провайдер или модель `agentRuntime.id: "codex"`                                     | Конфигурация модели/провайдера OpenClaw     |
| Использовать прямой трафик OpenAI API          | Провайдер или модель `agentRuntime.id: "openclaw"` с обычной аутентификацией OpenAI          | Конфигурация модели/провайдера OpenClaw     |
| Настроить поведение app-server               | `plugins.entries.codex.config.appServer.*`                                       | Конфигурация plugin Codex                |
| Включить нативные приложения plugin Codex        | `plugins.entries.codex.config.codexPlugins.*`                                    | Конфигурация plugin Codex                |
| Включить Codex Computer Use              | `plugins.entries.codex.config.computerUse.*`                                     | Конфигурация plugin Codex                |

Используйте ссылки на модели `openai/gpt-*` для ходов агента OpenAI, поддерживаемых Codex. Предпочитайте
`auth.order.openai` для порядка «сначала подписка, затем резервный API-ключ». Существующие
устаревшие идентификаторы профилей аутентификации Codex и устаревший порядок аутентификации Codex являются состоянием legacy только для doctor;
не записывайте новые устаревшие ссылки Codex GPT.

Не задавайте `compaction.model` или `compaction.provider` для агентов, поддерживаемых Codex.
Codex выполняет Compaction через свое нативное состояние треда app-server, поэтому OpenClaw игнорирует
эти локальные переопределения суммаризатора во время выполнения, а `openclaw doctor --fix` удаляет
их, когда агент использует Codex.

Lossless остается поддерживаемым как движок контекста для сборки, приема и
обслуживания вокруг ходов Codex. Настраивайте его через
`plugins.slots.contextEngine: "lossless-claw"` и
`plugins.entries.lossless-claw.config.summaryModel`, а не через
`agents.defaults.compaction.provider`. `openclaw doctor --fix` мигрирует старую
форму `compaction.provider: "lossless-claw"` в слот движка контекста Lossless,
когда активным runtime является Codex, но нативный Codex все равно владеет Compaction.

Нативный Codex app-server harness поддерживает движки контекста, которым требуется
предварительная сборка prompt. Универсальные CLI-бэкенды, включая `codex-cli`, не предоставляют
такую возможность хоста.

Для агентов, поддерживаемых Codex, `/compact` запускает нативную Compaction Codex app-server в
привязанном треде. OpenClaw не ждет завершения, не накладывает тайм-аут OpenClaw,
не перезапускает общий app-server и не переключается резервно на движок контекста или
публичный суммаризатор OpenAI. Если нативная привязка треда Codex отсутствует или
устарела, команда завершается с ошибкой, чтобы оператор видел реальную границу runtime,
а не тихое переключение бэкендов Compaction.

```json5
{
  auth: {
    order: {
      openai: ["openai:user@example.com", "openai:api-key-backup"],
    },
  },
}
```

В такой форме оба профиля по-прежнему выполняются через Codex для ходов агента
`openai/gpt-*`. API-ключ — это только резервный вариант аутентификации, а не запрос на переключение на OpenClaw или
обычные OpenAI Responses.

Остальная часть этой страницы охватывает распространенные варианты, между которыми пользователи должны выбрать:
форму развертывания, маршрутизацию с fail-closed, политику подтверждений guardian, нативные plugin Codex
и Computer Use. Полные списки параметров, значения по умолчанию, перечисления, обнаружение,
изоляцию окружения, тайм-ауты и поля транспорта app-server см. в
[справочнике Codex harness](/ru/plugins/codex-harness-reference).

## Проверка runtime Codex

Используйте `/status` в чате, где ожидаете Codex. Ход агента OpenAI, поддерживаемый Codex,
показывает:

```text
Runtime: OpenAI Codex
```

Затем проверьте состояние Codex app-server:

```text
/codex status
/codex models
```

`/codex status` сообщает о подключении к app-server, учетной записи, лимитах скорости, MCP
серверах и Skills. `/codex models` выводит live-каталог Codex app-server для
harness и учетной записи. Если `/status` выглядит неожиданно, см.
[Устранение неполадок](#troubleshooting).

## Маршрутизация и выбор модели

Разделяйте ссылки на провайдеров и политику runtime:

- Используйте `openai/gpt-*` для ходов агента OpenAI через Codex.
- Не используйте устаревшие ссылки Codex GPT в конфигурации. Запустите `openclaw doctor --fix`, чтобы
  исправить устаревшие ссылки и застарелые закрепления маршрутов сессий.
- `agentRuntime.id: "codex"` необязателен для обычного автоматического режима OpenAI, но полезен,
  когда развертывание должно завершаться с ошибкой, если Codex недоступен.
- `agentRuntime.id: "openclaw"` переводит провайдера или модель на встроенный runtime OpenClaw,
  когда это сделано намеренно.
- `/codex ...` управляет нативными разговорами Codex app-server из чата.
- ACP/acpx — отдельный путь внешнего harness. Используйте его только когда пользователь просит
  ACP/acpx или адаптер внешнего harness.

Распространенная маршрутизация команд:

| Намерение пользователя                                           | Используйте                                                                                                   |
| ----------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| Привязать текущий чат                               | `/codex bind [--cwd <path>]`                                                                          |
| Возобновить существующий тред Codex                       | `/codex resume <thread-id>`                                                                           |
| Перечислить или отфильтровать треды Codex                          | `/codex threads [filter]`                                                                             |
| Перечислить нативные plugin Codex                             | `/codex plugins list`                                                                                 |
| Включить или отключить настроенный нативный plugin Codex    | `/codex plugins enable <name>`, `/codex plugins disable <name>`                                       |
| Подключить существующую сессию Codex CLI на сопряженном узле | `/codex sessions --host <node> [filter]`, затем `/codex resume <session-id> --host <node> --bind here` |
| Отправить только обратную связь Codex                              | `/codex diagnostics [note]`                                                                           |
| Запустить задачу ACP/acpx                                | Команды сессии ACP/acpx, а не `/codex`                                                               |

| Сценарий использования                              | Настройка                                                              | Проверка                                | Примечания                            |
| ---------------------------------------------------- | ---------------------------------------------------------------------- | --------------------------------------- | ------------------------------------- |
| Подписка ChatGPT/Codex с нативной средой выполнения Codex | `openai/gpt-*` плюс включенный плагин `codex`                          | `/status` показывает `Runtime: OpenAI Codex` | Рекомендуемый путь                    |
| Завершение с ошибкой, если Codex недоступен          | Provider или model `agentRuntime.id: "codex"`                          | Ход завершается с ошибкой вместо встроенного резервного варианта | Используйте для развертываний только с Codex |
| Прямой трафик с API-ключом OpenAI через OpenClaw     | Provider или model `agentRuntime.id: "openclaw"` и обычная аутентификация OpenAI | `/status` показывает среду выполнения OpenClaw | Используйте только когда OpenClaw выбран намеренно |
| Устаревшая конфигурация                              | устаревшие ссылки Codex GPT                                            | `openclaw doctor --fix` переписывает ее | Не создавайте новую конфигурацию таким способом |
| Адаптер ACP/acpx Codex                               | ACP `sessions_spawn({ runtime: "acp" })`                               | Статус задачи/сессии ACP                | Отдельно от нативного рантайма Codex  |

`agents.defaults.imageModel` следует тому же разделению по префиксам. Используйте `openai/gpt-*`
для обычного маршрута OpenAI и `codex/gpt-*` только когда понимание изображений
должно выполняться через ограниченный ход сервера приложения Codex. Не используйте
устаревшие ссылки Codex GPT; doctor переписывает этот устаревший префикс в `openai/gpt-*`.

## Шаблоны развертывания

### Базовое развертывание Codex

Используйте конфигурацию quickstart, когда все ходы агента OpenAI должны по
умолчанию использовать Codex.

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
    },
  },
}
```

### Смешанное развертывание провайдеров

Эта форма сохраняет Claude агентом по умолчанию и добавляет именованный агент Codex:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
  agents: {
    defaults: {
      model: "anthropic/claude-opus-4-6",
    },
    list: [
      {
        id: "main",
        default: true,
        model: "anthropic/claude-opus-4-6",
      },
      {
        id: "codex",
        name: "Codex",
        model: "openai/gpt-5.5",
      },
    ],
  },
}
```

С этой конфигурацией агент `main` использует свой обычный путь провайдера, а
агент `codex` использует сервер приложения Codex.

### Развертывание Codex с завершением при недоступности

Для ходов агента OpenAI `openai/gpt-*` уже разрешается в Codex, когда
доступен встроенный плагин. Добавьте явную политику среды выполнения, когда вам нужно
зафиксированное правило завершения при недоступности:

```json5
{
  models: {
    providers: {
      openai: {
        agentRuntime: {
          id: "codex",
        },
      },
    },
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

Когда Codex принудительно задан, OpenClaw завершается с ошибкой на раннем этапе, если плагин Codex отключен, 
сервер приложения слишком старый или сервер приложения не может запуститься.

## Политика сервера приложения

По умолчанию плагин локально запускает управляемый OpenClaw бинарный файл Codex с транспортом stdio.
Задавайте `appServer.command` только когда намеренно хотите запускать
другой исполняемый файл. Используйте транспорт WebSocket только когда сервер приложения уже
запущен в другом месте:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            transport: "websocket",
            url: "ws://gateway-host:39175",
            authToken: "${CODEX_APP_SERVER_TOKEN}",
          },
        },
      },
    },
  },
}
```

Локальные сессии сервера приложения stdio по умолчанию используют доверенную позицию локального оператора:
`approvalPolicy: "never"`, `approvalsReviewer: "user"` и
`sandbox: "danger-full-access"`. Если локальные требования Codex запрещают такую
неявную позицию YOLO, OpenClaw вместо этого выбирает разрешенные guardian-права.
Когда для сессии активна песочница OpenClaw, OpenClaw отключает нативный Code Mode Codex,
пользовательские MCP-серверы и выполнение плагинов, поддерживаемых приложением, для этого
хода вместо того, чтобы полагаться на песочницу на стороне хоста Codex. Доступ к оболочке предоставляется
через динамические инструменты OpenClaw на базе песочницы, такие как `sandbox_exec` и
`sandbox_process`, когда доступны обычные инструменты exec/process.

Используйте нормализованный режим выполнения OpenClaw, когда вам нужен нативный автообзор Codex перед
выходами из песочницы или дополнительными разрешениями:

```json5
{
  tools: {
    exec: {
      mode: "auto",
    },
  },
  plugins: {
    entries: {
      codex: {
        enabled: true,
      },
    },
  },
}
```

Для сессий сервера приложения Codex OpenClaw сопоставляет `tools.exec.mode: "auto"` с проверенными Guardian
подтверждениями Codex, обычно
`approvalPolicy: "on-request"`, `approvalsReviewer: "auto_review"` и
`sandbox: "workspace-write"`, когда локальные требования разрешают эти значения.
В `tools.exec.mode: "auto"` OpenClaw не сохраняет устаревшие небезопасные переопределения Codex
`approvalPolicy: "never"` или `sandbox: "danger-full-access"`; используйте
`tools.exec.mode: "full"` для намеренной позиции Codex без подтверждений. Устаревший
пресет `plugins.entries.codex.config.appServer.mode: "guardian"` все еще
работает, но `tools.exec.mode: "auto"` является нормализованной поверхностью OpenClaw.

Сравнение на уровне режимов с подтверждениями host exec и разрешениями ACPX см.
в [Режимы разрешений](/ru/tools/permission-modes).

Все поля сервера приложения, порядок аутентификации, изоляцию окружения, обнаружение и
поведение тайм-аутов см. в [Справочнике рантайма Codex](/ru/plugins/codex-harness-reference).

## Команды и диагностика

Встроенный плагин регистрирует `/codex` как slash-команду в любом канале, который
поддерживает текстовые команды OpenClaw.

Нативное выполнение и управление требуют владельца или клиента Gateway с `operator.admin`.
Сюда входят привязка или возобновление потоков, отправка или остановка ходов,
изменение модели, fast-mode или состояния разрешений, Compaction или review, а также
отсоединение привязки. Другие авторизованные отправители сохраняют доступ только на чтение к командам
проверки статуса, справки, аккаунта, модели, потока, MCP-сервера, Skills и привязок.

Распространенные формы:

- `/codex status` проверяет подключение к серверу приложения, модели, аккаунт, лимиты частоты,
  MCP-серверы и Skills.
- `/codex models` выводит список живых моделей сервера приложения Codex.
- `/codex threads [filter]` выводит список недавних потоков сервера приложения Codex.
- `/codex resume <thread-id>` подключает текущую сессию OpenClaw к
  существующему потоку Codex.
- `/codex compact` просит сервер приложения Codex выполнить Compaction подключенного потока.
- `/codex review` запускает нативный review Codex для подключенного потока.
- `/codex diagnostics [note]` запрашивает подтверждение перед отправкой feedback Codex для
  подключенного потока.
- `/codex account` показывает статус аккаунта и лимитов частоты.
- `/codex mcp` выводит статус MCP-серверов сервера приложения Codex.
- `/codex skills` выводит Skills сервера приложения Codex.

Для большинства отчетов в поддержку начните с `/diagnostics [note]` в разговоре,
где произошла ошибка. Она создает один диагностический отчет Gateway и, для сессий
рантайма Codex, запрашивает подтверждение на отправку соответствующего пакета feedback Codex.
Модель конфиденциальности и поведение в групповых чатах см. в
[Экспорте диагностики](/ru/gateway/diagnostics).

Используйте `/codex diagnostics [note]` только когда вам нужен именно upload feedback Codex
для текущего подключенного потока без полного диагностического пакета Gateway.

### Локальная проверка потоков Codex

Самый быстрый способ проверить неудачный запуск Codex часто состоит в том, чтобы открыть нативный поток Codex
напрямую:

```bash
codex resume <thread-id>
```

Получите id потока из завершенного ответа `/diagnostics`, `/codex binding` или
`/codex threads [filter]`.

Механику upload и диагностические границы уровня среды выполнения см. в
[Рантайме Codex](/ru/plugins/codex-harness-runtime#codex-feedback-upload).

Аутентификация выбирается в следующем порядке:

1. Упорядоченные профили аутентификации OpenAI для агента, предпочтительно в
   `auth.order.openai`. Запустите `openclaw doctor --fix`, чтобы перенести старые
   устаревшие id профилей аутентификации Codex и устаревший порядок аутентификации Codex.
2. Существующий аккаунт сервера приложения в Codex home этого агента.
3. Только для локальных запусков сервера приложения stdio: `CODEX_API_KEY`, затем
   `OPENAI_API_KEY`, когда аккаунт сервера приложения отсутствует и аутентификация OpenAI
   все еще требуется.

Когда OpenClaw видит профиль аутентификации Codex в стиле подписки ChatGPT, он удаляет
`CODEX_API_KEY` и `OPENAI_API_KEY` из порожденного дочернего процесса Codex. Это
сохраняет API-ключи уровня Gateway доступными для embeddings или прямых моделей OpenAI
и при этом не позволяет нативным ходам сервера приложения Codex случайно тарифицироваться через API.
Явные профили Codex с API-ключом и локальный резервный env-key для stdio используют вход через сервер приложения
вместо унаследованного окружения дочернего процесса. WebSocket-подключения сервера приложения
не получают резервный API-ключ Gateway из окружения; используйте явный профиль аутентификации или
собственный аккаунт удаленного сервера приложения.
Когда настроены нативные плагины Codex, OpenClaw устанавливает или обновляет эти
плагины через подключенный сервер приложения перед тем, как предоставить приложения, принадлежащие плагинам,
потоку Codex. `app/list` остается источником истины для id приложений,
доступности и метаданных, но OpenClaw отвечает за решение о включении на уровне потока:
если политика разрешает указанное доступное приложение, OpenClaw отправляет
`thread/start.config.apps[appId].enabled = true`, даже когда `app/list` сейчас
сообщает, что это приложение отключено. Этот путь не создает установку приложения для
неизвестных id; OpenClaw активирует только marketplace-плагины с помощью `plugin/install`,
а затем обновляет inventory.

Если профиль подписки достигает лимита использования Codex, OpenClaw записывает время сброса,
когда Codex его сообщает, и пробует следующий упорядоченный профиль аутентификации для того же
запуска Codex. Когда время сброса проходит, профиль подписки снова становится доступным
без изменения выбранной модели `openai/gpt-*` или среды выполнения Codex.

Для локальных запусков сервера приложения stdio OpenClaw устанавливает `CODEX_HOME` в
каталог для конкретного агента, чтобы конфигурация Codex, файлы аутентификации/аккаунта, кэш/данные плагинов и нативное
состояние потоков по умолчанию не читали и не записывали личный `~/.codex` оператора.
OpenClaw сохраняет обычный процессный `HOME`; подпроцессы, запущенные Codex,
по-прежнему могут находить пользовательскую конфигурацию и токены, а Codex может обнаруживать общие
записи `$HOME/.agents/skills` и `$HOME/.agents/plugins/marketplace.json`.

Если развертыванию нужна дополнительная изоляция окружения, добавьте эти переменные в
`appServer.clearEnv`:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          appServer: {
            clearEnv: ["CODEX_API_KEY", "OPENAI_API_KEY"],
          },
        },
      },
    },
  },
}
```

`appServer.clearEnv` влияет только на порожденный дочерний процесс сервера приложения Codex.
OpenClaw удаляет `CODEX_HOME` и `HOME` из этого списка во время нормализации локального запуска:
`CODEX_HOME` остается заданным для конкретного агента, а `HOME` остается унаследованным, чтобы
подпроцессы могли использовать обычное состояние пользовательского home.

Динамические инструменты Codex по умолчанию загружаются в режиме `searchable`. OpenClaw не предоставляет
динамические инструменты, дублирующие встроенные операции Codex с рабочей областью: `read`, `write`,
`edit`, `apply_patch`, `exec`, `process` и `update_plan`. Большинство остальных
интеграционных инструментов OpenClaw, таких как обмен сообщениями, медиа, Cron, браузер, узлы,
Gateway и `heartbeat_respond`, доступны через поиск инструментов Codex в пространстве имен
`openclaw`, что уменьшает начальный контекст модели. Веб-поиск
по умолчанию использует размещенный инструмент Codex `web_search`, когда поиск включен и не выбран
управляемый провайдер. Нативный размещенный поиск и управляемый OpenClaw
динамический инструмент `web_search` являются взаимоисключающими, поэтому управляемый поиск не может обходить
нативные ограничения доменов. OpenClaw использует управляемый инструмент, когда размещенный поиск
недоступен, явно отключен или заменен выбранным управляемым провайдером.
OpenClaw оставляет отдельное расширение Codex `web.run` отключенным, поскольку
производственный трафик app-server отклоняет его пользовательское пространство имен `web`.
`tools.web.search.enabled: false` отключает оба пути, как и запуски только для LLM
с отключенными инструментами. Codex рассматривает `"cached"` как предпочтение и преобразует его в живой
внешний доступ для неограниченных обращений app-server. Автоматический управляемый резервный переход
завершается закрыто, когда заданы нативные `allowedDomains`, чтобы список разрешений нельзя было
обойти. Постоянные изменения эффективной политики поиска меняют привязанный поток Codex
перед следующим обращением. Временные ограничения на одно обращение используют временный
ограниченный поток и сохраняют существующую привязку для последующего возобновления.
`sessions_yield` и ответы источника только для инструмента сообщений остаются прямыми, поскольку
это контракты управления обращением. `sessions_spawn` остается доступным для поиска, поэтому нативный
`spawn_agent` Codex остается основной поверхностью подагентов Codex, при этом явное
делегирование OpenClaw или ACP по-прежнему доступно через пространство имен динамических
инструментов `openclaw`. Инструкции по совместной работе Heartbeat предписывают Codex искать
`heartbeat_respond` перед завершением обращения Heartbeat, если инструмент еще не
загружен.

Устанавливайте `codexDynamicToolsLoading: "direct"` только при подключении к пользовательскому Codex
app-server, который не может искать отложенные динамические инструменты, или при отладке полной
полезной нагрузки инструментов.

разрешений мог запускать управляемую сеть Codex. По умолчанию OpenClaw генерирует
устойчивое к коллизиям имя профиля `openclaw-network-<fingerprint>` из тела
профиля; используйте `profileName` только когда требуется стабильное локальное имя.

```js
export default {
  plugins: {
    entries: {
      codex: {
        config: {
          appServer: {
            sandbox: "workspace-write",
            networkProxy: {
              enabled: true,
              domains: {
                "api.openai.com": "allow",
                "blocked.example.com": "deny",
              },
              unixSockets: {
                "/tmp/proxy.sock": "allow",
                "/tmp/blocked.sock": "none",
              },
              allowUpstreamProxy: true,
              proxyUrl: "http://127.0.0.1:3128",
            },
          },
        },
      },
    },
  },
};
```

Если обычная среда выполнения app-server была бы `danger-full-access`, включение
`networkProxy` использует доступ к файловой системе в стиле рабочей области для
сгенерированного профиля разрешений. Управляемое Codex принудительное применение
сетевых правил является сетевой песочницей, поэтому профиль с полным доступом не
защитил бы исходящий трафик.
Записи доменов используют `allow` или `deny`; записи Unix-сокетов используют
значения Codex `allow` или `none`.

Динамические вызовы инструментов, которыми владеет OpenClaw, ограничиваются независимо от
`appServer.requestTimeoutMs`: запросы Codex `item/tool/call` по умолчанию используют 90-секундный
сторожевой таймер OpenClaw. Положительный аргумент `timeoutMs` для отдельного вызова увеличивает
или сокращает бюджет именно этого инструмента. Инструмент `image_generate` использует
`agents.defaults.imageGenerationModel.timeoutMs`, когда вызов инструмента не
задает собственный тайм-аут, либо иначе использует 120-секундный стандартный тайм-аут для генерации изображений.
Инструмент `image` для понимания медиа использует
`tools.media.image.timeoutSeconds` или свой 60-секундный стандартный тайм-аут для медиа. Для понимания изображений
этот тайм-аут применяется к самому запросу и не
сокращается из-за более ранней подготовительной работы. Бюджеты динамических инструментов
ограничены 600000 мс. При тайм-ауте OpenClaw прерывает сигнал инструмента
там, где это поддерживается, и возвращает Codex ответ о сбое динамического инструмента, чтобы ход
мог продолжиться, а не оставлять сессию в состоянии `processing`.
Этот сторожевой таймер является внешним бюджетом динамического `item/tool/call`; тайм-ауты запросов,
специфичные для провайдера, выполняются внутри этого вызова и сохраняют собственную семантику тайм-аутов.

После того как Codex принимает ход, и после того как OpenClaw отвечает на запрос app-server,
ограниченный текущим ходом, обвязка ожидает, что Codex будет продвигать текущий ход и
в итоге завершит нативный ход через `turn/completed`. Если app-server замолкает
на `appServer.turnCompletionIdleTimeoutMs`, OpenClaw по возможности
прерывает ход Codex, записывает диагностический тайм-аут и освобождает
полосу сессии OpenClaw, чтобы последующие сообщения чата не вставали в очередь за устаревшим
нативным ходом. Большинство нетерминальных уведомлений для того же хода снимают этот короткий
сторожевой таймер, потому что Codex доказал, что ход еще активен. Передачи инструментам используют
более длинный бюджет простоя после инструмента: после того как OpenClaw возвращает ответ `item/tool/call`,
после завершения нативных элементов инструментов, таких как `commandExecution`, после завершений необработанных
`custom_tool_call_output`, а также после необработанного прогресса ассистента после инструмента,
завершений рассуждения или прогресса рассуждения. Защита использует
`appServer.postToolRawAssistantCompletionIdleTimeoutMs`, когда он настроен, и
иначе по умолчанию равна пяти минутам. Этот же бюджет после инструмента также продлевает
сторожевой таймер прогресса для окна тихого синтеза перед тем, как Codex выдаст следующее
событие текущего хода. Глобальные уведомления app-server, такие как обновления лимитов частоты,
не сбрасывают прогресс простоя хода. Завершения рассуждения, завершения комментарийных
`agentMessage` и необработанный прогресс рассуждения или ассистента до инструмента могут
сопровождаться автоматическим финальным ответом, поэтому они используют защиту ответа после прогресса
вместо немедленного освобождения полосы сессии. Только
финальные/некомментарийные завершенные элементы `agentMessage` и необработанные
завершения ассистента до инструмента ставят на охрану освобождение по выводу ассистента: если Codex затем замолкает
без `turn/completed`, OpenClaw по возможности прерывает нативный ход и
освобождает полосу сессии. Если другая проверка хода выигрывает гонку за это освобождение,
OpenClaw все равно принимает завершенный финальный элемент ассистента, когда не остается активным ни один
нативный запрос, элемент или завершение динамического инструмента, а
освобождение по выводу ассистента все еще принадлежит последнему завершенному элементу, без
более позднего завершения элемента. Это может сохранить финальный ответ после завершенной работы инструментов
без повторного воспроизведения хода. Частичные дельты ассистента, устаревшие более ранние
ответы и пустые более поздние завершения не подходят. Сбои stdio
app-server, безопасные для повторного воспроизведения,
включая тайм-ауты простоя завершения хода без доказательств ассистента, инструмента, активного элемента
или побочных эффектов, повторяются один раз в новой попытке app-server. Небезопасные
тайм-ауты все равно выводят зависший клиент app-server из использования и освобождают полосу сессии OpenClaw.
Они также очищают устаревшую привязку нативного потока вместо автоматического
повторного воспроизведения. Тайм-ауты контроля завершения показывают специфичный для Codex текст тайм-аута:
в случаях, безопасных для повторного воспроизведения, говорится, что ответ может быть неполным, а небезопасные случаи
сообщают пользователю, что нужно проверить текущее состояние перед повторной попыткой. Публичная диагностика тайм-аутов
включает структурные поля, такие как последний метод уведомления app-server,
id/type/role необработанного элемента ответа ассистента, количество активных запросов/элементов и состояние
поставленных сторожевых проверок. Когда последнее уведомление является необработанным элементом ответа ассистента, она
также включает ограниченный предварительный просмотр текста ассистента. Она не включает необработанный промпт или
содержимое инструмента.

Переопределения окружения остаются доступными для локального тестирования:

- `OPENCLAW_CODEX_APP_SERVER_BIN`
- `OPENCLAW_CODEX_APP_SERVER_ARGS`
- `OPENCLAW_CODEX_APP_SERVER_MODE=yolo|guardian`
- `OPENCLAW_CODEX_APP_SERVER_APPROVAL_POLICY`
- `OPENCLAW_CODEX_APP_SERVER_SANDBOX`

`OPENCLAW_CODEX_APP_SERVER_BIN` обходит управляемый бинарный файл, когда
`appServer.command` не задан.

`OPENCLAW_CODEX_APP_SERVER_GUARDIAN=1` был удален. Вместо него используйте
`plugins.entries.codex.config.appServer.mode: "guardian"` или
`OPENCLAW_CODEX_APP_SERVER_MODE=guardian` для разового локального тестирования. Конфигурация
предпочтительнее для воспроизводимых развертываний, потому что она держит поведение Plugin в
том же проверяемом файле, что и остальная настройка обвязки Codex.

## Нативные плагины Codex

Поддержка нативных плагинов Codex использует собственные возможности приложений и плагинов app-server Codex
в том же потоке Codex, что и ход обвязки OpenClaw. OpenClaw
не переводит плагины Codex в синтетические динамические инструменты OpenClaw
`codex_plugin_*`.

`codexPlugins` влияет только на сессии, которые выбирают нативную обвязку Codex. Он
не влияет на запуски встроенной обвязки, обычные запуски провайдера OpenAI, привязки разговоров ACP
или другие обвязки.

Минимальная перенесенная конфигурация:

```json5
{
  plugins: {
    entries: {
      codex: {
        enabled: true,
        config: {
          codexPlugins: {
            enabled: true,
            allow_destructive_actions: true,
            plugins: {
              "google-calendar": {
                enabled: true,
                marketplaceName: "openai-curated",
                pluginName: "google-calendar",
              },
            },
          },
        },
      },
    },
  },
}
```

Конфигурация приложения потока вычисляется, когда OpenClaw устанавливает сессию обвязки Codex
или заменяет устаревшую привязку потока Codex. Она не пересчитывается на каждом ходе.
После изменения `codexPlugins` используйте `/new`, `/reset` или перезапустите gateway, чтобы
будущие сессии обвязки Codex стартовали с обновленным набором приложений.

Сведения о пригодности к миграции, инвентаре приложений, политике разрушительных действий,
запросах уточнений и диагностике нативных плагинов см. в
[Нативные плагины Codex](/ru/plugins/codex-native-plugins).

Доступ к приложениям и плагинам на стороне OpenAI контролируется вошедшей учетной записью Codex,
а для рабочих пространств Business и Enterprise/Edu также элементами управления приложениями рабочего пространства. См.
[Использование Codex с вашим планом ChatGPT](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan)
для обзора учетных записей и управления рабочими пространствами OpenAI.

## Использование компьютера

Использование компьютера описано в отдельном руководстве по настройке:
[Использование компьютера Codex](/ru/plugins/codex-computer-use).

Кратко: OpenClaw не поставляет встроенно приложение для управления рабочим столом и не выполняет
действия на рабочем столе самостоятельно. Он подготавливает app-server Codex, проверяет, что
MCP-сервер `computer-use` доступен, а затем позволяет Codex владеть нативными вызовами инструментов MCP
во время ходов в режиме Codex.

## Границы выполнения

Обвязка Codex меняет только низкоуровневый встроенный исполнитель агента.

- Динамические инструменты OpenClaw поддерживаются. Codex просит OpenClaw выполнить эти
  инструменты, поэтому OpenClaw остается в пути выполнения.
- Нативные инструменты оболочки, patch, MCP и нативных приложений Codex принадлежат Codex.
  OpenClaw может наблюдать или блокировать выбранные нативные события через поддерживаемый
  relay, но не переписывает аргументы нативных инструментов.
- Codex владеет нативным Compaction. OpenClaw хранит зеркало транскрипта для истории
  каналов, поиска, `/new`, `/reset` и будущего переключения модели или обвязки, но
  не заменяет Compaction Codex суммаризатором OpenClaw или контекстного движка.
- Генерация медиа, понимание медиа, TTS, approvals и вывод messaging-tool
  продолжают проходить через соответствующие настройки провайдера/модели OpenClaw.
- `tool_result_persist` применяется к результатам инструментов транскрипта, которыми владеет OpenClaw, а не
  к записям результатов нативных инструментов Codex.

Сведения о слоях хуков, поддерживаемых поверхностях V1, обработке нативных разрешений, управлении
очередью, механике загрузки обратной связи Codex и деталях Compaction см. в
[Среда выполнения обвязки Codex](/ru/plugins/codex-harness-runtime).

## Устранение неполадок

**Codex не появляется как обычный провайдер `/model`:** это ожидаемо для
новых конфигураций. Выберите модель `openai/gpt-*`, включите
`plugins.entries.codex.enabled` и проверьте, не исключает ли `plugins.allow`
`codex`.

**OpenClaw использует встроенную обвязку вместо Codex:** убедитесь, что ref модели —
`openai/gpt-*` у официального провайдера OpenAI и что Plugin Codex
установлен и включен. Если при тестировании нужны строгие доказательства, задайте у провайдера или
модели `agentRuntime.id: "codex"`. Принудительная среда выполнения Codex завершается с ошибкой вместо
отката к OpenClaw.

**Среда выполнения OpenAI Codex откатывается к пути API-ключа:** соберите отредактированный
фрагмент gateway, который показывает модель, среду выполнения, выбранного провайдера и сбой.
Попросите затронутых коллег выполнить эту read-only команду на своем хосте OpenClaw:

```bash
(
  pattern='openai/gpt-5\.[45]|openai[-]codex|agentRuntime(\.id)?|harnessRuntime|Runtime: OpenAI Codex|legacy OpenAI Codex prefix|resolveSelectedOpenAIRuntimeProvider|candidateProvider[": ]+openai|status[": ]+401|Incorrect API key|No API key|api-key path|API-key path|OAuth'

  if ls /tmp/openclaw/openclaw-*.log >/dev/null 2>&1; then
    grep -E -i -n "$pattern" /tmp/openclaw/openclaw-*.log 2>/dev/null || true
  else
    journalctl --user -u openclaw-gateway --since today --no-pager 2>/dev/null \
      | grep -E -i "$pattern" || true
  fi
) | sed -E \
    -e 's/(Authorization: Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(Bearer )[A-Za-z0-9._~+\/-]+/\1[REDACTED]/Ig' \
    -e 's/(api[_ -]?key[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/(OPENAI_API_KEY[=: ]+)[^ ,}"]+/\1[REDACTED]/Ig' \
    -e 's/sk-[A-Za-z0-9_-]{12,}/sk-[REDACTED]/g' \
    -e 's/[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}/[EMAIL-REDACTED]/g' \
  | tail -200
```

Полезные фрагменты обычно включают `openai/gpt-5.5` или `openai/gpt-5.4`,
`Runtime: OpenAI Codex`, `agentRuntime.id` или `harnessRuntime`,
`candidateProvider: "openai"` и результат `401`, `Incorrect API key` или
`No API key`. Исправленный запуск должен показывать путь OpenAI OAuth
вместо обычного сбоя API-ключа OpenAI.

**Осталась конфигурация устаревших refs моделей Codex:** выполните `openclaw doctor --fix`.
Doctor переписывает устаревшие refs моделей в `openai/*`, удаляет устаревшие закрепления среды выполнения сессии и
всего агента, а также сохраняет существующие переопределения профилей auth.

**app-server отклонен:** используйте app-server Codex `0.125.0` или новее.
Пререлизы той же версии или версии с суффиксом сборки, такие как
`0.125.0-alpha.2` или `0.125.0+custom`, отклоняются, потому что OpenClaw проверяет
стабильный нижний порог протокола `0.125.0`.

**`/codex status` не может подключиться:** проверьте, что встроенный Plugin `codex`
включен, что `plugins.allow` включает его, когда настроен allowlist, и
что любые пользовательские `appServer.command`, `url`, `authToken` или заголовки корректны.

**Обнаружение моделей медленное:** уменьшите
`plugins.entries.codex.config.discovery.timeoutMs` или отключите обнаружение. См.
[Справочник по обвязке Codex](/ru/plugins/codex-harness-reference#model-discovery).

**Транспорт WebSocket сразу завершается с ошибкой:** проверьте `appServer.url`, `authToken`,
заголовки и что удаленный app-server говорит на той же версии протокола app-server Codex.

**Нативные инструменты оболочки или патчей заблокированы с ошибкой `Native hook relay unavailable`:**
поток Codex все еще пытается использовать идентификатор нативного ретранслятора хуков, который OpenClaw
больше не зарегистрировал. Это проблема нативного транспорта хуков Codex, а не сбой ACP
backend, провайдера, GitHub или команды оболочки. Запустите новый сеанс в
затронутом чате с помощью `/new` или `/reset`, затем повторите безвредную команду. Если это
сработает один раз, но следующий вызов нативного инструмента снова завершится ошибкой, считайте `/new` только временным
обходным решением: скопируйте prompt в новый сеанс после перезапуска Codex
app-server или OpenClaw Gateway, чтобы старые потоки были удалены, а регистрации нативных хуков
созданы заново.

**Модель, не являющаяся Codex, использует встроенную обвязку:** это ожидаемо, если
политика runtime провайдера или модели не направляет ее в другую обвязку. Обычные ссылки провайдеров не OpenAI
остаются на своем стандартном пути провайдера в режиме `auto`.

**Computer Use установлен, но инструменты не запускаются:** проверьте
`/codex computer-use status` из нового сеанса. Если инструмент сообщает
`Native hook relay unavailable`, используйте описанное выше восстановление нативного ретранслятора хуков. См.
[Codex Computer Use](/ru/plugins/codex-computer-use#troubleshooting).
