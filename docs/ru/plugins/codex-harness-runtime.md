Контракт среды выполнения для обращений через адаптер Codex. Сведения о настройке и маршрутизации см. в разделе
[Адаптер Codex](/ru/plugins/codex-harness). Поля конфигурации описаны в
[справочнике по адаптеру Codex](/ru/plugins/codex-harness-reference).

## Обзор

Codex управляет нативным циклом модели, нативным возобновлением потока, нативным продолжением работы инструментов и нативным Compaction. OpenClaw управляет маршрутизацией каналов, файлами сеансов, доставкой видимых сообщений, динамическими инструментами OpenClaw, подтверждениями, доставкой медиафайлов и зеркалом стенограммы вокруг этой границы.

Маршрутизация запросов определяется выбранной средой выполнения, а не только строкой поставщика. Нативное обращение Codex получает инструкции разработчика от app-server Codex; при явном использовании маршрута совместимости OpenClaw сохраняется обычный системный запрос OpenClaw, даже если применяется аутентификация или транспорт OpenAI в стиле Codex.

OpenClaw запускает и возобновляет нативные потоки Codex со встроенной индивидуальностью Codex, отключённой с помощью `personality: "none"`, чтобы приоритет сохранялся за файлами индивидуальности рабочей области и идентичностью агента OpenClaw. В остальном нативный Codex сохраняет базовые инструкции и инструкции модели, принадлежащие Codex, а также загрузку проектной документации. Облегчённые запуски OpenClaw (например, Cron) по-прежнему отключают загрузку проектной документации.

Инструкции разработчика OpenClaw охватывают аспекты среды выполнения OpenClaw: доставку в исходный канал, динамические инструменты OpenClaw, делегирование ACP, контекст адаптера и активные файлы профиля рабочей области агента. Каталоги Skills и указатели `MEMORY.md`, доступные через инструменты, проецируются как инструкции разработчика по совместной работе, действующие в пределах обращения. Если инструменты памяти недоступны, активное содержимое `BOOTSTRAP.md` и полный файл `MEMORY.md` вместо этого передаются как обычный входной контекст обращения.

Большинство динамических инструментов OpenClaw используют пространство имён `openclaw` с поддержкой поиска. Инструменты с пометкой `catalogMode: "direct-only"` используют `openclaw_direct`, которое Codex оставляет непосредственно видимым модели как `DirectModelOnly`, не предоставляя его вложенному выполнению в режиме Code Mode.

## Привязки потоков и смена модели

Когда сеанс OpenClaw подключён к существующему потоку Codex, при следующем обращении в app-server повторно передаются выбранная в данный момент модель, политика подтверждений, песочница, проверяющий подтверждения и уровень обслуживания. При переходе с `openai/gpt-5.5` на `openai/gpt-5.2` привязка потока сохраняется, но Codex получает запрос продолжить работу с новой выбранной моделью.

Контролируемые привязки являются исключением. Выбор модели OpenClaw остаётся заблокированным, а при возобновлении не передаются переопределения модели и поставщика, чтобы Codex восстановил сохранённые модель и поставщика канонического потока. Отдельный нативный элемент управления Codex может изменить эту сохранённую пару, а начальный снимок состояния может вызвать обычное предупреждение Codex о различии моделей; внешняя модель OpenClaw и цепочка резервных вариантов никогда не подменяют ни одну из них.

## Контроль и безопасное продолжение

Контроль Codex — это включаемая явно возможность того же Plugin `codex`. Он обнаруживает нативные потоки через отдельное подключение и проецирует в каталог Gateway только неархивированные сеансы. Без явно заданных параметров подключения `appServer` это подключение использует управляемый stdio из домашнего каталога пользователя, тогда как обычный адаптер остаётся привязанным к агенту. Получение списка и чтение метаданных пассивны: они не возобновляют поток, не подписывают OpenClaw на его события в реальном времени и не отвечают на его запросы подтверждения.

Для сохранённого или бездействующего сеанса на компьютере Gateway действие **Продолжить как ветвь** создаёт обычный Chat с заблокированной моделью и зеркалирует ограниченную историю сообщений пользователя и ассистента вплоть до последнего сохранённого завершённого обращения источника. Первое обычное обращение Chat устанавливает реальные обработчики подтверждений и использует временное нативное ответвление, чтобы зафиксировать снимок состояния без переопределения модели или поставщика. Codex App Server использует текущую нативную конфигурацию и возвращает выбранную пару; если эта модель отличается от последней записанной модели источника, он выводит обычное предупреждение. Через то же подключение контроля OpenClaw запускает канонический поток адаптера Codex с источником `appServer` в его рабочем каталоге и с его политикой среды выполнения, используя для этого начального запуска ровно возвращённые модель и поставщика, внедряет ограниченную видимую историю и архивирует временное ответвление. Исходный поток никогда не возобновляется. Канонический поток располагает полным набором инструментов адаптера OpenClaw; рассуждения, вызовы инструментов и их результаты из источника в него не клонируются. Закрытая область подключения сохраняется для ожидающих и зафиксированных состояний привязки, поэтому каждое последующее обращение остаётся в этом подключении с нативной аутентификацией и конфигурацией поставщика. При отключённом контроле или расхождении привязки либо подключения операция завершается с отказом, а не переключается на обычный адаптер из домашнего каталога агента.

Исходный поток CLI или VS Code по-прежнему может отображаться в обоих каталогах. Каноническая ветвь является нативным потоком Codex, но тип её источника — `appServer`; нативные клиенты могут отфильтровывать этот тип источника, поэтому её отображение в Codex Desktop не гарантируется.

Для активных источников нельзя создать новую ветвь или выполнить архивацию; существующий контролируемый Chat по-прежнему можно открыть. `notLoaded` означает, что активность неизвестна, а не что сеанс бездействует; OpenClaw разрешает архивировать локальную строку со статусом `idle` или `notLoaded` только после явного подтверждения отсутствия другого процесса выполнения и нового чтения статуса в рамках текущего процесса. Codex сериализует изменения потоков внутри одного процесса App Server, но не предоставляет эксклюзивную межпроцессную аренду исполнителя или владельца подтверждений, поэтому такое чтение не может доказать, что другой процесс не использует поток. OpenClaw блокирует известного активного владельца привязки для точной цели или любого неархивированного порождённого потомка, возвращённого постраничным запросом потомков Codex. Ошибки перечисления, циклы и исчерпание предела безопасности приводят к отказу. Нативная архивация всё ещё может вступить в гонку с новым обращением в другом процессе, поэтому подтверждение охватывает неизвестные клиенты и промежуток между чтением статуса и архивацией. Контролируемый Chat с заблокированной моделью нельзя удалить, пока он защищает нативную привязку.

Каталоги сопряжённых Node в первоначальном выпуске содержат только метаданные. Текущая граница вызова Node работает по схеме запрос/ответ и не может передавать долгоживущие события обращения, запросы подтверждений или потоковый вывод, необходимые для настоящей привязки адаптера Codex. Поэтому удалённые действия **Продолжить** и **Архивировать** остаются недоступными, даже если строка бездействует.

Настройка для операторов и видимое поведение интерфейса управления описаны в разделе [Контроль Codex](/ru/plugins/codex-supervision).

## Видимые ответы и Heartbeat

Для обращений прямого или исходного чата через адаптер Codex по умолчанию включена автоматическая доставка окончательного ответа ассистента во внутренних интерфейсах WebChat в соответствии с контрактом адаптера Pi: агент отвечает обычным образом, а OpenClaw публикует окончательный текст в исходной беседе. Установите `messages.visibleReplies: "message_tool"`, чтобы окончательный текст ассистента оставался скрытым, если агент не вызовет `message(action="send")`.

При обращениях Heartbeat Codex по умолчанию получает `heartbeat_respond` в доступном для поиска каталоге инструментов OpenClaw, чтобы агент мог указать, должно ли пробуждение пройти без уведомления или сообщить о себе. Рекомендации по инициативности Heartbeat передаются как инструкция разработчика режима совместной работы Codex, действующая в пределах обращения Heartbeat; обычные обращения чата остаются в режиме Codex Default. Если `HEARTBEAT.md` не пуст, инструкции Heartbeat указывают Codex на этот файл, а не встраивают его содержимое.

## Границы хуков

| Уровень                               | Владелец                 | Назначение                                                                    |
| ------------------------------------- | ------------------------ | ----------------------------------------------------------------------------- |
| Хуки Plugin OpenClaw                  | OpenClaw                 | Совместимость продукта и Plugin между адаптерами OpenClaw и Codex.            |
| ПО промежуточного слоя расширений app-server Codex | Встроенные плагины OpenClaw | Поведение адаптера для каждого обращения вокруг динамических инструментов OpenClaw. |
| Нативные хуки Codex                   | Codex                    | Низкоуровневый жизненный цикл Codex и политика нативных инструментов из конфигурации Codex. |

OpenClaw не использует проектные или глобальные файлы Codex `hooks.json` для маршрутизации поведения Plugin. Для моста нативных инструментов и разрешений OpenClaw внедряет в каждый поток конфигурацию Codex для `PreToolUse`, `PostToolUse`, `PermissionRequest` и `Stop`.

Когда подтверждения app-server Codex включены (`approvalPolicy` не равно `"never"`), внедряемая по умолчанию конфигурация нативных хуков не включает `PermissionRequest`, чтобы проверяющий app-server Codex и мост подтверждений OpenClaw обрабатывали реальные запросы повышения привилегий после проверки. Добавьте `permission_request` в `nativeHookRelay.events`, чтобы в любом случае принудительно использовать ретранслятор совместимости. Другие хуки Codex, такие как `SessionStart` и `UserPromptSubmit`, остаются элементами управления уровня Codex; в контракте v1 они не предоставляются как хуки Plugin OpenClaw.

Для динамических инструментов OpenClaw выполнение инструмента происходит в OpenClaw после того, как Codex запрашивает вызов, поэтому поведение Plugin и промежуточного слоя реализуется в адаптере. Для нативных инструментов Codex канонической записью инструмента управляет Codex; OpenClaw может зеркалировать выбранные события, но не может изменять нативный поток, если Codex не предоставляет такую возможность через app-server или обратные вызовы нативных хуков.

События `PreToolUse` app-server Codex в режиме отчёта откладывают подтверждение Plugin до соответствующего подтверждения app-server. Если хук OpenClaw `before_tool_call` возвращает `requireApproval`, а нативная полезная нагрузка задаёт `openclaw_approval_mode: "report"`, ретранслятор нативных хуков регистрирует требование подтверждения Plugin и не возвращает нативного решения. Когда Codex позже отправляет запрос подтверждения app-server для того же использования инструмента, OpenClaw открывает запрос подтверждения Plugin и сопоставляет решение с Codex. События Codex `PermissionRequest` используют отдельный путь подтверждения и при соответствующей настройке моста по-прежнему могут направляться через подтверждения OpenClaw.

Уведомления элементов app-server Codex также предоставляют асинхронные наблюдения `after_tool_call` за завершениями нативных инструментов, которые ещё не были охвачены ретранслятором нативного `PostToolUse`. Они предназначены только для телеметрии и совместимости и не могут блокировать, задерживать или изменять вызов нативного инструмента.

Проекции Compaction и жизненного цикла LLM формируются из уведомлений app-server Codex и состояния адаптера OpenClaw, а не из команд нативных хуков Codex. `before_compaction`, `after_compaction`, `llm_input` и `llm_output` — это наблюдения уровня адаптера, а не побайтовые копии внутренних запросов Codex или полезных нагрузок Compaction.

Нативные уведомления app-server Codex `hook/started` и `hook/completed` проецируются как события агента `codex_app_server.hook` для отслеживания траектории и отладки. Они не вызывают хуки Plugin OpenClaw.

| Вмешательство в Compaction                          | OpenClaw не позволяет плагинам или движкам контекста запрещать, изменять или заменять нативную Compaction Codex.                              | Если плагинам необходимо запрещать или изменять нативную Compaction, следует добавить хуки Codex до и после Compaction. |
| Побайтовый захват запроса к API модели               | OpenClaw может захватывать запросы и уведомления app-server, но ядро Codex формирует окончательный запрос к API OpenAI внутренними средствами. | Требуется событие трассировки запроса модели Codex или отладочный API.                      |

## Нативные разрешения и запросы MCP

Для `PermissionRequest` OpenClaw возвращает явное решение о разрешении или
отказе только тогда, когда его определяет политика. Отсутствие решения не
означает разрешение: Codex рассматривает его как отсутствие решения хука и
переходит к собственному защитному механизму или пути подтверждения
пользователем.

В режимах подтверждения Codex app-server этот нативный хук по умолчанию
не используется. Исключение действует, если `permission_request` явно включён
в `nativeHookRelay.events` или его устанавливает среда выполнения
совместимости.

Когда оператор выбирает `allow-always` для запроса нативного разрешения
Codex, OpenClaw запоминает точный отпечаток входных данных
поставщика/сеанса/инструмента/cwd в пределах ограниченного временного окна
сеанса. Запомненное решение намеренно применяется только при точном
совпадении: изменённая команда, аргументы, полезная нагрузка инструмента или
cwd создают новый запрос подтверждения.

Запросы подтверждения инструментов Codex MCP направляются через поток
подтверждения плагина OpenClaw, когда Codex помечает
`_meta.codex_approval_kind` значением `"mcp_tool_call"`. Промпты Codex
`request_user_input` отправляются обратно в исходный чат, а следующее
поставленное в очередь последующее сообщение отвечает на этот нативный запрос
сервера, а не направляется как дополнительный контекст. Прочие запросы MCP
закрываются с отказом.

Общий поток подтверждения плагина, передающий эти промпты, описан в разделе
[Запросы разрешений плагина](/ru/plugins/plugin-permission-requests).

## Управление очередью

Управление очередью активного запуска сопоставляется с `turn/steer` в Codex
app-server. При значении по умолчанию `messages.queue.mode: "steer"` OpenClaw
объединяет сообщения чата в режиме управления за настроенный период тишины и
отправляет их одним запросом `turn/steer` в порядке поступления.

Проверка Codex и ходы ручной Compaction могут отклонять корректировку в том же ходе. В
таком случае OpenClaw ожидает завершения активного запуска, прежде чем начать
обработку запроса. Используйте `/queue followup` или `/queue collect`, если сообщения по
умолчанию должны ставиться в очередь, а не корректировать текущий ход. См. [Очередь корректировок](/ru/concepts/queue-steering).

## Отправка отзывов Codex

Когда `/diagnostics [note]` одобрена для сеанса в нативной
среде Codex, OpenClaw также вызывает `feedback/upload` сервера приложений Codex для связанных
веток Codex, включая журналы каждой указанной ветки и порождённых
дочерних веток Codex, когда они доступны.

Отправка выполняется по стандартному каналу обратной связи Codex на серверы OpenAI. Если
обратная связь Codex отключена на этом сервере приложений, команда возвращает
ошибку сервера приложений. Ответ о завершённой диагностике содержит каналы,
идентификаторы сеансов OpenClaw, идентификаторы веток Codex и локальные команды
`codex resume <thread-id>` для отправленных веток.

Если вы отклоните или проигнорируете запрос на одобрение, OpenClaw не выведет эти идентификаторы Codex
и не отправит отзыв Codex. Отправка не заменяет локальный
экспорт диагностики Gateway. Сведения об одобрении, конфиденциальности, локальном пакете
и поведении в групповых чатах см. в разделе [Экспорт диагностики](/ru/gateway/diagnostics).

Используйте `/codex diagnostics [note]`, только если хотите отправить отзыв Codex
для текущей подключённой ветки без полного пакета диагностики
Gateway.

## Compaction и зеркало журнала диалога

Когда выбранная модель использует среду Codex, нативная Compaction ветки
выполняется сервером приложений Codex. OpenClaw не выполняет предварительную Compaction для
ходов Codex, не заменяет Compaction Codex на Compaction механизма контекста и не
переключается на суммаризацию OpenClaw или общедоступного OpenAI, если нативную Compaction
не удаётся запустить. OpenClaw хранит зеркало журнала диалога для истории канала, поиска,
`/new`, `/reset` и последующего переключения модели или среды.

Явные запросы на Compaction, такие как `/compact` или запрошенная плагином ручная
операция Compaction, запускают нативную Compaction Codex с помощью `thread/compact/start`.
OpenClaw сохраняет запрос и аренду общего клиента открытыми, пока Codex не выдаст
соответствующий элемент завершения `contextCompaction`, после чего сообщает о завершении
хода Compaction. Если этот завершающий ход превышает настроенный тайм-аут Compaction,
OpenClaw запрашивает нативное прерывание хода. Аренда и блокировка Compaction
для конкретной ветки остаются активными, пока Codex не сообщит о конечном состоянии или не подтвердит
RPC прерывания. Если Codex не подтвердит его в течение льготного периода
прерывания, OpenClaw выведет соединение из эксплуатации до снятия блокировки. Для удалённых
соединений также отсоединяется привязка соответствующей ветки, чтобы последующая работа не могла
пересечься с неподтверждённым удалённым ходом. Другие ходы в выведенном из эксплуатации соединении завершаются
ошибкой и могут быть повторены с новым клиентом. Закрытие клиента, отмена запроса или
неудачный ход Compaction возвращают ошибку операции. Автоматическая Compaction при нехватке
контекстного окна выполняется Codex; OpenClaw запускает нативную Compaction только для запрошенных
вручную триггеров.

Когда механизм контекста запрашивает проекцию начального состояния ветки Codex, OpenClaw
переносит в новую ветку Codex имена и идентификаторы вызовов инструментов, структуры входных данных
и отредактированное содержимое результатов инструментов. Необработанные значения аргументов вызовов
инструментов в эту проекцию не копируются.

Зеркало включает запрос пользователя, окончательный текст ассистента и упрощённые
записи рассуждений или плана Codex, когда сервер приложений их выдаёт. OpenClaw
записывает начало нативной Compaction и конечный статус, но не
предоставляет удобочитаемую сводку Compaction или проверяемый список записей,
сохранённых Codex после Compaction.

Поскольку Codex управляет канонической нативной веткой, `tool_result_persist` не
перезаписывает нативные записи результатов инструментов Codex. Он применяется, только когда OpenClaw
записывает результат инструмента в журнал диалога сеанса, принадлежащий OpenClaw.

## Мультимедиа и доставка

OpenClaw продолжает управлять доставкой мультимедиа и выбором поставщика мультимедиа. Для изображений,
видео, музыки, PDF, TTS и анализа мультимедиа используются соответствующие настройки
поставщика и модели, такие как `agents.defaults.imageGenerationModel`,
`videoGenerationModel`, `pdfModel` и `messages.tts`.

Текст, изображения, видео, музыка, TTS, одобрения и вывод инструментов обмена сообщениями по-прежнему
доставляются по стандартному пути OpenClaw; для генерации мультимедиа
устаревшая среда выполнения не требуется. Когда Codex выдаёт нативный элемент генерации изображения с
`savedPath`, OpenClaw пересылает именно этот файл по стандартному пути мультимедиа
ответа, даже если ход Codex не содержит текста ассистента.
