Эта страница документирует контракт среды выполнения для ходов обвязки Codex. Для настройки и
маршрутизации начните с [обвязки Codex](/ru/plugins/codex-harness). Поля конфигурации
см. в [справочнике обвязки Codex](/ru/plugins/codex-harness-reference).

## Обзор

Режим Codex — это не OpenClaw с другим вызовом модели внутри. Codex владеет большей частью
нативного цикла модели, а OpenClaw адаптирует свои поверхности Plugin, инструментов, сессий и
диагностики вокруг этой границы.

OpenClaw по-прежнему владеет маршрутизацией каналов, файлами сессий, доставкой видимых сообщений,
динамическими инструментами OpenClaw, подтверждениями, доставкой медиа и зеркалом транскрипта.
Codex владеет каноническим нативным потоком, нативным циклом модели, нативным продолжением
инструментов и нативным Compaction.

Маршрутизация промптов следует выбранной среде выполнения, а не только строке провайдера.
Нативный ход Codex получает developer-инструкции app-server Codex, а явный маршрут совместимости
OpenClaw сохраняет обычный системный промпт OpenClaw, даже когда использует OpenAI-аутентификацию
или транспорт в стиле Codex.

Нативный Codex сохраняет базовые инструкции и инструкции модели, принадлежащие Codex, а также
поведение проектной документации согласно активной конфигурации потока Codex. OpenClaw запускает
и возобновляет нативные потоки Codex со встроенной персональностью Codex в отключенном состоянии,
чтобы файлы персональности рабочей области и идентичность агента OpenClaw оставались
авторитетными. Облегченные запуски OpenClaw по-прежнему сохраняют существующее подавление
проектной документации. Developer-инструкции OpenClaw покрывают аспекты среды выполнения OpenClaw,
такие как доставка в исходный канал, динамические инструменты OpenClaw, делегирование ACP,
контекст адаптера и активные файлы профиля рабочей области агента. Каталоги Skills OpenClaw и
маршрутизируемые через инструменты указатели `MEMORY.md` проецируются как ограниченные ходом
developer-инструкции для совместной работы в нативном Codex. Активное содержимое `BOOTSTRAP.md`
и полная резервная инъекция `MEMORY.md` по-прежнему используют ссылочный контекст входных данных
хода.

## Привязки потоков и изменения модели

Когда сессия OpenClaw присоединена к существующему потоку Codex, следующий ход снова отправляет
текущую выбранную модель OpenAI, политику подтверждений, песочницу и уровень сервиса в app-server.
Переключение с `openai/gpt-5.5` на
`openai/gpt-5.2` сохраняет привязку потока, но просит Codex продолжить с
новой выбранной моделью.

## Видимые ответы и Heartbeat

Когда прямой ход или ход исходного чата выполняется через обвязку Codex, видимые ответы
по умолчанию используют автоматическую доставку финального ответа ассистента для внутренних
поверхностей WebChat. Это выравнивает Codex с контрактом промпта обвязки Pi: агенты отвечают
обычным образом, а OpenClaw публикует финальный текст в исходную беседу. Установите
`messages.visibleReplies: "message_tool"`, когда прямой чат или исходный чат должен
намеренно оставлять финальный текст ассистента приватным, если агент не вызывает
`message(action="send")`.

Ходы Heartbeat Codex также по умолчанию получают `heartbeat_respond` в доступном для поиска
каталоге инструментов OpenClaw, чтобы агент мог записать, должен ли пробуждающий сигнал
оставаться тихим или отправить уведомление, не кодируя этот поток управления в финальном тексте.

Специальные указания инициативы для Heartbeat отправляются как developer-инструкция режима
совместной работы Codex непосредственно на ходе Heartbeat. Обычные ходы чата восстанавливают
режим Codex Default вместо того, чтобы нести философию Heartbeat в своем обычном промпте
среды выполнения. Когда существует непустой `HEARTBEAT.md`, инструкции режима совместной работы
Heartbeat указывают Codex на файл вместо встраивания его содержимого.

## Границы хуков

Обвязка Codex имеет три слоя хуков:

| Слой                                  | Владелец                 | Назначение                                                          |
| ------------------------------------- | ------------------------ | ------------------------------------------------------------------- |
| Хуки Plugin OpenClaw                  | OpenClaw                 | Совместимость продукта и Plugin между обвязками OpenClaw и Codex.   |
| Middleware расширений app-server Codex | Встроенные Plugin OpenClaw | Поведение адаптера на каждый ход вокруг динамических инструментов OpenClaw. |
| Нативные хуки Codex                   | Codex                    | Низкоуровневый жизненный цикл Codex и политика нативных инструментов из конфигурации Codex. |

OpenClaw не использует проектные или глобальные файлы Codex `hooks.json` для маршрутизации
поведения Plugin OpenClaw. Для поддерживаемого моста нативных инструментов и разрешений
OpenClaw внедряет конфигурацию Codex на поток для `PreToolUse`, `PostToolUse`,
`PermissionRequest` и `Stop`.

Когда подтверждения app-server Codex включены, то есть `approvalPolicy` не равно
`"never"`, внедряемая по умолчанию конфигурация нативных хуков опускает `PermissionRequest`, чтобы
рецензент app-server Codex и мост подтверждений OpenClaw обрабатывали реальные повышения прав
после проверки. Операторы могут явно добавить `permission_request` в
`nativeHookRelay.events`, когда им нужен relay совместимости.

Другие хуки Codex, такие как `SessionStart` и `UserPromptSubmit`, остаются
средствами управления уровня Codex. Они не раскрываются как хуки Plugin OpenClaw в контракте v1.

Для динамических инструментов OpenClaw OpenClaw выполняет инструмент после того, как Codex
запрашивает вызов, поэтому OpenClaw запускает поведение Plugin и middleware, которым он владеет,
в адаптере обвязки. Для нативных инструментов Codex Codex владеет канонической записью
инструмента. OpenClaw может зеркалировать выбранные события, но не может переписать нативный
поток Codex, если Codex не раскрывает эту операцию через app-server или callback нативных хуков.

События `PreToolUse` в report-режиме app-server Codex откладывают запросы подтверждения Plugin
до соответствующего подтверждения app-server. Если хук OpenClaw `before_tool_call`
возвращает `requireApproval`, пока нативная полезная нагрузка задает режим подтверждения report
(`openclaw_approval_mode` равно `"report"`), relay нативных хуков записывает требование
подтверждения Plugin и не возвращает нативного решения. Когда Codex отправляет запрос
подтверждения app-server для того же использования инструмента, OpenClaw открывает запрос
подтверждения Plugin и сопоставляет решение обратно с Codex. События Codex `PermissionRequest`
являются отдельным путем подтверждения и все еще могут маршрутизироваться через подтверждения
OpenClaw, когда среда выполнения настроена для этого моста.

Уведомления элементов app-server Codex также предоставляют асинхронные наблюдения
`after_tool_call` для завершений нативных инструментов, которые еще не покрыты relay
нативного `PostToolUse`. Эти наблюдения предназначены только для телеметрии и совместимости
Plugin; они не могут блокировать, задерживать или изменять нативный вызов инструмента.

Проекции жизненного цикла Compaction и LLM поступают из уведомлений app-server Codex
и состояния адаптера OpenClaw, а не из команд нативных хуков Codex.
События OpenClaw `before_compaction`, `after_compaction`, `llm_input` и
`llm_output` являются наблюдениями уровня адаптера, а не побайтовыми снимками
внутреннего запроса Codex или полезных нагрузок Compaction.

Нативные уведомления app-server Codex `hook/started` и `hook/completed`
проецируются как события агента `codex_app_server.hook` для траектории и отладки.
Они не вызывают хуки Plugin OpenClaw.

## Нативные разрешения и MCP-запросы

Для `PermissionRequest` OpenClaw возвращает только явные решения разрешить или
запретить, когда это решает политика. Результат без решения не является
разрешением. Codex рассматривает его как отсутствие решения хука и переходит к
собственному пути guardian или пользовательского одобрения.

Режимы одобрения app-server Codex по умолчанию пропускают этот нативный хук. Это
поведение применяется, когда `permission_request` явно включен в
`nativeHookRelay.events` или когда его устанавливает совместимая среда
выполнения.

Когда оператор выбирает `allow-always` для запроса нативного разрешения Codex,
OpenClaw запоминает точный отпечаток provider/session/tool input/cwd для
ограниченного окна сессии. Запомненное решение намеренно требует точного
совпадения: измененная команда, аргументы, полезная нагрузка инструмента или cwd
создают новое одобрение.

Запросы одобрения инструментов MCP Codex маршрутизируются через поток одобрения
Plugin OpenClaw, когда Codex помечает `_meta.codex_approval_kind` как
`"mcp_tool_call"`. Промпты Codex `request_user_input` отправляются обратно в
исходный чат, а следующее поставленное в очередь последующее сообщение отвечает
на этот нативный серверный запрос вместо направления как дополнительный
контекст. Другие запросы MCP завершаются закрыто.

Об общем потоке одобрения Plugin, который переносит эти промпты, см.
[Запросы разрешений Plugin](/ru/plugins/plugin-permission-requests).

## Управление очередью

Управление очередью активного запуска сопоставляется с `turn/steer` app-server
Codex. При значении по умолчанию `messages.queue.mode: "steer"` OpenClaw
пакетирует сообщения чата в режиме steer за настроенное тихое окно и отправляет
их одним запросом `turn/steer` в порядке поступления.

Проверка Codex и ходы ручной Compaction могут отклонять управление в том же ходе. В таком
случае OpenClaw ждет завершения активного запуска перед запуском промпта.
Используйте `/queue followup` или `/queue collect`, когда сообщения должны по умолчанию
попадать в очередь, а не управлять ходом. См. [Очередь управления](/ru/concepts/queue-steering).

## Загрузка отзыва Codex

Когда `/diagnostics [note]` одобрена для сеанса, использующего нативную обвязку Codex,
OpenClaw также вызывает `feedback/upload` app-server Codex для соответствующих
потоков Codex. Загрузка просит app-server включить журналы для каждого указанного потока
и порожденных подпотоков Codex, когда они доступны.

Загрузка проходит через обычный путь отзывов Codex на серверы OpenAI. Если отзывы Codex
отключены в этом app-server, команда возвращает ошибку app-server.
Завершенный ответ диагностики перечисляет каналы, идентификаторы сеансов OpenClaw,
идентификаторы потоков Codex и локальные команды `codex resume <thread-id>` для потоков,
которые были отправлены.

Если вы отклоняете или игнорируете одобрение, OpenClaw не выводит эти идентификаторы Codex и
не отправляет отзыв Codex. Загрузка не заменяет локальный диагностический экспорт Gateway.
См. [Экспорт диагностики](/ru/gateway/diagnostics) для поведения одобрения, приватности,
локального пакета и групповых чатов.

Используйте `/codex diagnostics [note]` только когда вам конкретно нужна загрузка
отзыва Codex для текущего подключенного потока без полного диагностического пакета Gateway.

## Compaction и зеркало транскрипта

Когда выбранная модель использует обвязку Codex, нативная Compaction потока принадлежит
app-server Codex. OpenClaw не запускает предварительную Compaction для ходов Codex,
не заменяет Compaction Codex на Compaction движка контекста и не
откатывается к суммаризации OpenClaw или публичной OpenAI, когда нативную Compaction Codex
нельзя запустить. OpenClaw хранит зеркало транскрипта для истории каналов,
поиска, `/new`, `/reset` и будущего переключения модели или обвязки.

Явные запросы Compaction, такие как `/compact` или запрошенная Plugin операция ручной
Compaction, запускают нативную Compaction Codex с `thread/compact/start`.
OpenClaw держит запрос и аренду общего клиента открытыми, пока Codex не выдаст
соответствующий элемент завершения `contextCompaction`, а затем сообщает ход Compaction
как завершенный. Если этот терминальный ход превышает настроенный тайм-аут Compaction,
OpenClaw запрашивает нативное прерывание хода. Аренда и барьер Compaction для каждого потока
остаются удержанными, пока Codex не сообщит терминальное состояние или не подтвердит RPC прерывания.
Если Codex не подтверждает в течение льготного периода прерывания, OpenClaw выводит
соединение из эксплуатации перед освобождением барьера. Удаленные соединения также отсоединяют
соответствующую привязку потока, чтобы последующая работа не могла пересечься с неподтвержденным удаленным
ходом. Другие ходы на выведенном из эксплуатации соединении завершаются ошибкой и могут повториться на новом клиенте.
Закрытие клиента, отмена запроса или неудачный ход Compaction возвращает
неудачную операцию.

Когда движок контекста запрашивает проекцию начальной загрузки потока Codex, OpenClaw
проецирует имена и идентификаторы вызовов инструментов, формы входных данных и отредактированное содержимое результатов инструментов
в новый поток Codex. Он не копирует сырые значения аргументов вызовов инструментов в
эту проекцию.

Зеркало включает пользовательский промпт, итоговый текст ассистента и облегченные записи
рассуждений или плана Codex, когда app-server их выдает. OpenClaw записывает
старт нативной Compaction и терминальный статус, но не предоставляет
человекочитаемую сводку Compaction или проверяемый список записей, которые Codex
сохранил после Compaction.

Поскольку Codex владеет каноническим нативным потоком, `tool_result_persist` сейчас не
перезаписывает нативные записи результатов инструментов Codex. Он применяется только когда
OpenClaw записывает результат инструмента транскрипта сеанса, принадлежащего OpenClaw.

## Медиа и доставка

OpenClaw продолжает владеть доставкой медиа и выбором поставщика медиа. Изображения,
видео, музыка, PDF, TTS и понимание медиа используют соответствующие настройки поставщика/модели,
такие как `agents.defaults.imageGenerationModel`, `videoGenerationModel`,
`pdfModel` и `messages.tts`.

Текст, изображения, видео, музыка, TTS, одобрения и вывод инструментов сообщений продолжают
проходить через обычный путь доставки OpenClaw. Генерация медиа не требует устаревшей среды выполнения.
Когда Codex выдает нативный элемент генерации изображения с `savedPath`, OpenClaw
пересылает этот точный файл через обычный путь медиа-ответа, даже если ход Codex
не содержит текста ассистента.
