Плагин Workboard добавляет необязательную доску в стиле канбан в
[интерфейс управления](/ru/web/control-ui): рабочие карточки для агентов, назначение агентам
и ссылку на связанные с карточкой задачу, запуск и сеанс панели управления.

Workboard намеренно сделан компактным: он отслеживает локальную операционную работу для одного
OpenClaw Gateway. Он не заменяет GitHub Issues, Linear, Jira или
другие командные системы управления проектами.

## Включение

Workboard входит в комплект поставки, но по умолчанию отключён:

1. Откройте **Плагины** в интерфейсе управления или перейдите по пути `/settings/plugins` относительно
   настроенного базового пути интерфейса управления. Например, при базовом пути `/openclaw`
   используется `/openclaw/settings/plugins`.
2. Найдите **Workboard** и выберите **Включить**. Поскольку Workboard входит в состав
   OpenClaw, действие **Установить** не требуется.
3. Если интерфейс сообщает, что требуется перезапуск, перезапустите Gateway.

Вкладка Workboard появляется в навигации панели управления после загрузки среды выполнения плагина.
Пока плагин отключён, вкладка не отображается в навигации. При прямом открытии
маршрута `/workboard`, когда плагин отключён или заблокирован
`plugins.allow`/`plugins.deny`, вместо данных карточек отображается состояние
недоступности плагина.

Эквивалентная последовательность действий в CLI:

```bash
openclaw plugins enable workboard
openclaw gateway restart
openclaw dashboard
```

## Конфигурация

У Workboard нет конфигурации, специфичной для плагина. Включайте и отключайте его с помощью стандартной
записи плагина:

```json5
{
  plugins: {
    entries: {
      workboard: {
        enabled: true,
        config: {},
      },
    },
  },
}
```

```bash
openclaw plugins disable workboard
openclaw gateway restart
```

## Поля карточки

| Поле          | Значения                                                                                                                     |
| ------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| `status`      | `triage`, `backlog`, `todo`, `scheduled`, `ready`, `running`, `review`, `blocked`, `done`                                    |
| `priority`    | `low`, `normal`, `high`, `urgent`                                                                                            |
| `labels`      | строки произвольного формата                                                                                                 |
| `agentId`     | необязательный назначенный агент                                                                                             |

Карточки также содержат компактные метаданные попыток, комментариев, ссылок, подтверждений,
артефактов, настроек автоматизации, вложений, журналов исполнителей, состояния протокола
исполнителя, заявок, диагностики, уведомлений, идентификатора шаблона, состояния архива и
обнаружения устаревших сеансов, а также список недавних событий (`created`, `edited`,
`moved`, `linked`, `specified`, `decomposed`, `claimed`, `heartbeat`,
`execution_updated`, `attempt_started`, `attempt_updated`, `comment_added`,
`link_added`, `proof_added`, `artifact_added`, `attachment_added`,
`diagnostic`, `notification`, `dispatch`, `orchestration`,
`protocol_violation`, `archived`, `unarchived`, `stale`). Эти метаданные позволяют
оператору увидеть, как карточка перемещалась по доске, не открывая связанный
сеанс; это локальный операционный контекст, а не замена расшифровкам
сеансов или истории задач GitHub.

Карточки хранятся в собственном состоянии Gateway плагина и перемещаются вместе с остальным
состоянием OpenClaw этого Gateway (см. [Хранилище](#storage)).

## Запуск работы из карточки

Из несвязанных карточек можно запускать работу напрямую:

- **Запустить Codex** / **Запустить Claude** запускает отслеживаемый как задача запуск агента с
  явно указанным движком, отправляет запрос карточки и присваивает карточке статус `running`. Запуски Codex
  используют `openai/gpt-5.6-sol`; запуски Claude используют `anthropic/claude-sonnet-4-6`.
- **Открыть Codex** / **Открыть Claude** создаёт связанный сеанс панели управления без
  отправки запроса карточки или её перемещения — для ручной работы, которая остаётся
  прикреплённой к доске.

Автономные запуски используют путь Gateway для запусков агентов, отслеживаемых как задачи (агент
и модель по умолчанию, если Codex/Claude не выбран явно); затем Workboard связывает
полученную задачу, идентификатор запуска и ключ сеанса с карточкой. Для каждого связанного
выполнения также записывается сводка попытки (движок, режим, модель, идентификатор запуска,
временные метки, статус, накопительное количество сбоев), чтобы повторяющиеся сбои оставались заметными.

Панель управления обновляет статус задачи из журнала задач Gateway, сопоставляя
задачи с карточками по идентификатору задачи, идентификатору запуска или ключу связанного сеанса. Задача
в очереди или в процессе выполнения сохраняет активным жизненный цикл карточки; завершённая, неудачная,
просроченная или отменённая задача переводит карточку в `review` или `blocked` по тому же правилу
синхронизации, что и связанные сеансы (см. [Синхронизация жизненного цикла сеанса](#session-lifecycle-sync)).

от других агентов, если вызывающая сторона не предоставляет токен назначения,
возвращённый `workboard_claim`. В каждой карточке, возвращаемой инструментом
агента или вызовом Gateway RPC, значение `metadata.claim.token` скрывается как
`[redacted]` (сам токен возвращается только один раз, на верхнем уровне и
только из `workboard_claim`), поэтому операторы панели управления и другие
агенты могут просматривать состояние назначения, не получая пригодный для
использования токен. Восстановление выполняется через
`workboard_promote`/`workboard_reassign`/`workboard_reclaim`, для которых
токен не требуется.

## Диспетчеризация

Диспетчеризация выполняется локально в Gateway: она не запускает произвольные
процессы ОС. Выполнением по-прежнему управляют обычные сеансы субагентов
OpenClaw. Один проход диспетчеризации:

1. Продвигает карточки, зависимости которых готовы.
2. Записывает метаданные диспетчеризации в готовые карточки.
3. Блокирует назначения с истёкшим сроком и выполнения, превысившие лимит времени.
4. Помечает настроенные на доске карточки сортировки как кандидатов на оркестрацию.
5. Назначает небольшой пакет готовых карточек и запускает выполнение исполнителей
   через среду выполнения субагентов Gateway.

Исполнители получают ограниченный контекст карточки и токен назначения,
необходимый для отправки Heartbeat, завершения или блокировки карточки через
инструменты Workboard.

### Выбор исполнителей

Каждый проход по умолчанию запускает **не более 3 исполнителей**. Готовые
карточки упорядочиваются сначала по приоритету, затем по позиции, затем по
времени создания. За один проход запускается только одна карточка на каждого
владельца/агента; владельцы, у которых на доске уже есть выполняемая или
проверяемая работа, пропускаются. Архивные карточки, карточки с активным
назначением и карточки не в состоянии `ready` никогда не выбираются для
запуска исполнителей (но на них всё ещё может влиять работа диспетчеризации
с данными: очистка устаревших назначений, продвижение зависимостей и очистка
по истечении лимита времени).

Ключи сеансов детерминированы для каждой пары доска/карточка, поэтому повторная
диспетчеризация направляет работу обратно в ту же линию исполнителя, а не
создаёт несвязанные сеансы:

- Назначенные карточки: `agent:<agentId>:subagent:workboard-<boardId>-<cardId>`
- Неназначенные карточки: `subagent:workboard-<boardId>-<cardId>` (Gateway
  определяет настроенного агента по умолчанию)

Если после назначения карточки исполнителя запустить не удаётся, Workboard
блокирует карточку, снимает назначение, записывает ошибку запуска выполнения
и добавляет строку в журнал исполнителя — она отображается в панели
управления, JSON-выводе CLI, инструментах агента и диагностике карточки.

### Точки входа

- Действие диспетчеризации в панели управления
- `openclaw workboard dispatch`
- `/workboard dispatch` в канале с поддержкой команд

Все три используют среду выполнения субагентов Gateway, когда Gateway
доступен. В CLI предусмотрен один резервный режим для оператора: если вызов
Gateway завершается ошибкой подключения или недоступности (либо ошибкой
`unknown method` для старых версий Gateway), при этом не указана явная цель
`--url`/`--token` и не настроен удалённый Gateway (`OPENCLAW_GATEWAY_URL` или
`gateway.mode: remote`), CLI выполняет диспетчеризацию только данных в
локальном состоянии SQLite — он может продвигать зависимости, очищать
устаревшие назначения и блокировать выполнения, превысившие лимит времени,
но не может запускать исполнителей. Ошибки аутентификации, разрешений и
проверки данных от доступного Gateway не считаются недоступностью: они
выводятся как ошибки команды, как и любая ошибка Gateway, если была указана
явная цель `--url`/`--token`.

В метаданных доски можно задать `autoDecompose`,
`autoDecomposePerDispatch`, `defaultAssignee` и `orchestratorProfile`.
OpenClaw записывает это намерение и предоставляет его в контексте исполнителя;
фактическое уточнение и разбиение по-прежнему выполняются через обычные
инструменты Workboard.

## CLI и слеш-команда

```bash
openclaw workboard list [--board <id>] [--status <status>] [--include-archived] [--json]
openclaw workboard create "Fix stale card lifecycle" --priority high --labels bug,workboard
openclaw workboard show <card-id> [--json]
openclaw workboard dispatch [--board <id>] [--json]
```

Текстовый вывод `list` по умолчанию скрывает архивные карточки
(`--include-archived` переопределяет это поведение); `--json` всегда включает
архивные карточки в соответствии с контрактом полной карточки, используемым
существующими скриптами. `show` принимает однозначный префикс идентификатора.
`list`, `create` и `show` всегда напрямую читают и записывают локальное
состояние Plugin. Только `dispatch` вызывает работающий Gateway, используя
описанный выше резервный режим.

Полное описание флагов, JSON-вывода, резервного поведения Gateway, обработки
префиксов идентификаторов, правил выбора при диспетчеризации и устранения
неполадок см. в разделе [CLI Workboard](/ru/cli/workboard).

`/workboard list`, `/workboard show <card-id>`, `/workboard create <title>`
и `/workboard dispatch` соответствуют командам CLI. Просмотр списка и
карточки — операции чтения, доступные любому авторизованному отправителю
команд. Для создания и диспетчеризации на поверхностях чата требуется статус
владельца, а для клиента Gateway — `operator.write`/`operator.admin`.

## Синхронизация жизненного цикла сеанса

Карточки могут быть связаны с существующим сеансом панели управления или с сеансом, созданным при запуске работы из карточки. В связанных карточках жизненный цикл сеанса отображается непосредственно в карточке: выполняется, устарел, связан и бездействует, завершён, завершился с ошибкой или отсутствует. Существующий сеанс также можно добавить на вкладке «Сеансы» с помощью кнопки **Добавить на рабочую доску**; карточка связывается с этим сеансом, использует метку сеанса или недавний запрос пользователя в качестве заголовка и, если возможно, заполняет заметки недавним запросом пользователя и последним ответом ассистента.

Если связанный сеанс исчезает, карточка сохраняет связь для контекста и по-прежнему предоставляет элементы управления запуском, позволяющие начать работу заново в новом сеансе. Если активный связанный сеанс перестаёт сообщать о недавней активности, рабочая доска помечает карточку как `stale` и сохраняет это значение в метаданных, пока жизненный цикл не удалит его.

Пока карточка находится в активном рабочем состоянии, рабочая доска отслеживает связанный сеанс:

| Состояние связанного сеанса                | Статус карточки |
| ------------------------------------------ | --------------- |
| активен                                    | `running`       |
| завершён                                   | `review`        |
| ошибка, принудительная остановка, истечение времени или прерывание | `blocked`       |

**Состояния ручной проверки имеют приоритет.** Перемещение карточки в `review`, `blocked` или `done` прекращает её автоматическую синхронизацию, пока вы не вернёте её в `todo` или `running`.

При запуске карточки используются обычные сеансы Gateway; рабочая доска хранит только метаданные карточки и связи. Стенограмма разговора, выбор модели и жизненный цикл выполнения остаются под управлением обычной системы сеансов. Используйте **Остановить** в активной связанной карточке, чтобы прервать текущее выполнение, — рабочая доска пометит карточку как `blocked`, чтобы она оставалась видимой для последующей работы.

Новые карточки можно создавать на основе шаблонов рабочей доски (`bugfix`, `docs`, `release`, `pr_review`, `plugin`). Шаблоны предварительно заполняют заголовок, заметки, метки и приоритет; идентификатор шаблона сохраняется в метаданных карточки.

## Рабочий процесс панели управления

1. Откройте вкладку «Рабочая доска» в интерфейсе управления.
2. Создайте карточку, указав заголовок, заметки, приоритет, метки, необязательного агента и необязательный связанный сеанс, — либо откройте «Сеансы» и выберите **Добавить на рабочую доску** для существующего сеанса.
3. Перетаскивайте карточку между столбцами либо переведите фокус на её компактный элемент управления статусом и используйте меню или ArrowLeft/ArrowRight.
4. Запустите работу из карточки, чтобы создать или повторно использовать сеанс панели управления.
5. Пока агент работает, открывайте связанный сеанс из карточки.
6. Позвольте синхронизации жизненного цикла переместить выполняемую работу в `review`/`blocked`, а после принятия вручную переместите карточку в `done`.

## Диагностика

Результаты диагностики вычисляются на основе локальных метаданных карточек. Встроенные проверки выявляют:

| Тип                         | Условие                                                                        |
| --------------------------- | ------------------------------------------------------------------------------ |
| `stranded_ready`            | Назначенная карточка в состоянии `todo`/`backlog`/`ready` не обновлялась более 1 часа. |
| `running_without_heartbeat` | Карточка в состоянии `running` не получала Heartbeat заявки или обновлений выполнения более 20 минут. |
| `blocked_too_long`          | Карточка в состоянии `blocked` не обновлялась более 24 часов.                  |
| `repeated_failures`         | Отслеживаемое число сбоев карточки достигло 2 или более.                       |
| `missing_proof`             | У карточки в состоянии `done` нет подтверждений, артефактов или вложений.      |
| `orphaned_session`          | У карточки в состоянии `running` есть `sessionKey`, но отсутствуют метаданные `execution`. |

## Разрешения

Методы RPC Gateway находятся в пространстве имён `workboard.*`:

| Область доступа   | Методы                                                                                                                                                                                                                                                                                                                                                                            |
| ----------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `operator.read`   | `cards.list`, `cards.export`, `cards.diagnostics`, получение списка и содержимого вложений, чтение событий уведомлений, `boards.list`, `cards.stats`, `cards.runs`                                                                                                                                                                                                                 |
| `operator.write`  | `cards.diagnostics.refresh`, создание/обновление/перемещение/удаление/комментирование/связывание/связывание зависимостей/подтверждение/добавление артефактов, добавление/удаление вложений, журнал исполнителя, нарушение протокола, заявка/Heartbeat/освобождение/повышение/переназначение/повторная заявка/завершение/блокировка/разблокировка, `cards.dispatch`, `cards.bulk`, архивирование, `boards.upsert`/`archive`/`delete`, `cards.specify`/`decompose`, подписка на уведомления/удаление/продвижение |

Ни один метод RPC не требует `operator.admin`. Браузеры, подключённые с доступом оператора только для чтения, могут просматривать доску, но не могут изменять карточки.

## Хранилище

Рабочая доска хранит постоянные данные в принадлежащей Plugin реляционной базе данных SQLite в каталоге состояния OpenClaw: доски, карточки, метки, события жизненного цикла, попытки выполнения, комментарии, связи зависимостей, подтверждения, ссылки на артефакты, метаданные и двоичные данные вложений, результаты диагностики, уведомления, журналы исполнителей, состояние протокола и подписки находятся в таблицах рабочей доски, а не в записях хранилища «ключ — значение» Plugin. Экспорт карточки сохраняет историю доски, не включая в неё содержимое двоичных данных вложений.

В установках, где рабочая доска использовалась в выпуске `.28`, можно выполнить `openclaw doctor --fix`, чтобы перенести выпущенные устаревшие пространства имён состояния Plugin (`workboard.cards`, `workboard.boards`, `workboard.notify` и, при наличии, `workboard.attachments`) в реляционную базу данных.

## Устранение неполадок

**На вкладке указано, что рабочая доска недоступна**

```bash
openclaw plugins inspect workboard --runtime --json
```

Если настроен параметр `plugins.allow`, добавьте в него `workboard`. Если `plugins.deny` содержит `workboard`, удалите его оттуда перед включением Plugin.

**Карточки не сохраняются**

Убедитесь, что подключение браузера имеет доступ `operator.write`. Сеансы оператора только для чтения могут выводить список карточек, но не могут создавать, редактировать, перемещать или удалять их.

**При запуске карточки не открывается ожидаемый сеанс**

Проверьте идентификатор агента и связанный сеанс карточки, затем откройте «Сеансы» или «Чат», чтобы проверить фактическое состояние выполнения.

**Распределение не запускает исполнителя**

Убедитесь, что существует хотя бы одна карточка в состоянии `ready` без активной заявки:

```bash
openclaw workboard list --status ready
```

Если CLI сообщает о распределении только данных, запустите или перезапустите Gateway и повторите попытку: распределение только данных обновляет локальное состояние доски, но не может запускать сеансы исполнителей-субагентов. Карточки также могут быть пропущены, если другая карточка того же владельца или агента уже выполняется либо ожидает проверки; завершите, заблокируйте или освободите эту активную работу, прежде чем распределять дополнительные задачи тому же владельцу.
