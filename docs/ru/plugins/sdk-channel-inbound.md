Пути получения сообщений каналом следуют единому потоку:

```text
событие платформы -> входные факты/контекст -> ответ агента -> доставка сообщения
```

Используйте `openclaw/plugin-sdk/channel-inbound` для нормализации входящих событий,
форматирования, корневых каталогов и оркестрации. Используйте
`openclaw/plugin-sdk/channel-outbound` для нативной отправки, квитанций, гарантированной
доставки и предварительного просмотра в реальном времени.

## Основные вспомогательные функции

```ts
import {
  buildChannelInboundEventContext,
  runChannelInboundEvent,
  dispatchChannelInboundReply,
} from "openclaw/plugin-sdk/channel-inbound";
```

- `buildChannelInboundEventContext(...)`: преобразует нормализованные данные канала
  в контекст запроса/сеанса. Передавайте метаданные отправителя и чата, принадлежащие каналу,
  через `channelContext`, который обработчики Plugin получают как `ctx.channelContext`.
  Расширяйте `PluginHookChannelSenderContext` или `PluginHookChannelChatContext`
  из этого подпути полями, специфичными для канала.
- `runChannelInboundEvent(...)`: выполняет прием, классификацию, предварительную проверку, разрешение,
  запись, диспетчеризацию и завершение обработки одного входящего события платформы.
- `dispatchChannelInboundReply(...)`: записывает и диспетчеризирует уже
  сформированный входящий ответ с помощью адаптера доставки.

Встроенные/нативные каналы, которые уже получают внедренный объект среды выполнения Plugin,
могут вызывать те же вспомогательные функции через `runtime.channel.inbound.*` вместо
прямого импорта этого подпути:

```ts
await runtime.channel.inbound.run({
  channel: "demo",
  accountId,
  raw: platformEvent,
  adapter: {
    ingest: normalizePlatformEvent,
    resolveTurn: resolveInboundReply,
  },
});
```

Формируйте входные данные для `dispatchChannelInboundReply(...)` в диспетчерах
совместимости, которые оставляют доставку платформе в адаптере доставки. Новые пути
отправки должны вместо этого использовать адаптеры сообщений и вспомогательные функции
гарантированной обработки сообщений из `channel-outbound`.

## Миграция

Псевдонимы среды выполнения `runtime.channel.turn.*` удалены. Используйте:

- `runtime.channel.inbound.run(...)` для необработанных входящих событий.
- `runtime.channel.inbound.dispatchReply(...)` для сформированных контекстов ответа.
- `runtime.channel.inbound.buildContext(...)` для полезной нагрузки входящего контекста.
- `runtime.channel.inbound.runPreparedReply(...)`, устаревший метод, только для
  принадлежащих каналу путей диспетчеризации подготовленных ответов, которые уже формируют
  собственное замыкание диспетчеризации.

Новый код Plugin не должен вводить API каналов с `turn` в названии. Используйте терминологию
цикла модели или агента только в коде агента/провайдера; в Plugin каналов применяйте термины
«входящие данные», «сообщение», «доставка» и «ответ».
