# Plugin политик

Добавляет основанные на политиках проверки doctor для контроля соответствия рабочего пространства.

## Распространение

- Пакет: `@openclaw/policy`
- Способ установки: входит в состав OpenClaw

## Поверхность

Plugin

## Поведение

Plugin политик добавляет проверки состояния doctor для настроек OpenClaw, управляемых политиками, и регламентируемых деклараций рабочего пространства. В настоящее время политики охватывают соответствие каналов, метаданные регламентируемых инструментов, состояние серверов MCP, состояние поставщиков моделей, состояние доступа к частной сети, состояние доступности Gateway, состояние рабочего пространства и инструментов агента, состояние настроенных глобальных и индивидуальных для агентов инструментов, состояние настроенной среды выполнения песочницы, состояние входящего доступа и доступа к каналам, состояние обработки данных, а также состояние поставщика секретов и профиля аутентификации в конфигурации OpenClaw.

Политики хранят заданные требования в `policy.jsonc`, используют существующие настройки OpenClaw и декларации рабочего пространства как свидетельства и сообщают об отклонениях с помощью `openclaw policy check` и `openclaw doctor --lint`. Успешная проверка политики выводит хеши политики, свидетельств, обнаруженных проблем и аттестации, которые операторы могут сохранять для аудита.

`openclaw policy compare --baseline <file>` сравнивает один файл политики с другим. Это проверка соответствия только на уровне конфигурации: она использует метаданные правил политики, чтобы убедиться, что проверяемая политика не содержит меньше требований и не является менее строгой, чем заданная базовая политика, и не проверяет состояние среды выполнения, учетные данные или значения секретов.

Правила состояния инструментов могут требовать утвержденные профили, ограничение инструментов файловой системы только рабочим пространством, ограниченные настройки безопасности, подтверждения и хоста для выполнения команд, отключенный привилегированный режим, точное соответствие записей `alsoAllow` и наличие обязательных запрещающих записей для инструментов. В свидетельствах фиксируются дополнительные записи `alsoAllow`, поскольку они могут расширять фактические разрешения для инструментов. Эти проверки контролируют только соответствие конфигурации; они не считывают состояние подтверждений среды выполнения и не добавляют принудительное применение правил во время выполнения.

Правила состояния песочницы могут требовать утвержденные режимы и серверные механизмы песочницы, запрещать сетевое взаимодействие контейнера через хост, присоединение к пространствам имен контейнеров, монтирование в контейнеры не в режиме только для чтения, монтирование сокетов среды выполнения контейнеров и использование неограниченных профилей контейнеров, а также требовать диапазоны исходных адресов CDP для браузера в песочнице.
Эти проверки контролируют только соответствие конфигурации; они не считывают состояние подтверждений среды выполнения, не проверяют работающие контейнеры и не добавляют принудительное применение правил во время выполнения.

Правила обработки данных могут требовать редактирования конфиденциальной информации в журналах, запрещать сбор содержимого в телеметрии, требовать обслуживания срока хранения сеансов и запрещать индексирование расшифровок сеансов в памяти. Эти проверки контролируют только соответствие конфигурации; они не проверяют необработанные журналы, экспортированные данные телеметрии, расшифровки, файлы памяти, секреты или персональные данные.

Именованные области политик в `scopes.<scopeName>` могут добавлять более строгие стандартные разделы политик для указанных в них селекторов. `agentIds` поддерживает `tools`, `agents.workspace`, `sandbox` и `dataHandling.memory`; `channelIds` поддерживает `ingress.channels`.
Идентификаторы агентов среды выполнения, которые не указаны явно в `agents.list[]`, проверяются на соответствие унаследованным глобальным настройкам или настройкам по умолчанию, а не пропускаются без свидетельств. Каждая область в `policy.jsonc` должна быть допустимой и обеспечивать принудительное применение правил для своего селектора. Правила наложения являются дополнительными требованиями, поэтому они не ослабляют политику верхнего уровня и могут формировать собственные обнаруженные проблемы, если одна и та же наблюдаемая конфигурация нарушает обе области.
