Внешний plugin `@openclaw/copilot` выполняет встроенные агентские циклы Copilot по подписке через GitHub Copilot CLI (`@github/copilot-sdk`) вместо встроенной среды выполнения OpenClaw. Сеанс Copilot CLI управляет низкоуровневым агентским циклом: нативным выполнением инструментов, нативной Compaction (`infiniteSessions`) и состоянием ветки, которым CLI управляет в каталоге `copilotHome`. OpenClaw по-прежнему управляет каналами чата, файлами сеансов, выбором модели, динамическими инструментами (через мост), подтверждениями, доставкой медиафайлов, видимой зеркальной копией стенограммы, побочными вопросами `/btw` (см. [Побочные вопросы (`/btw`)](#side-questions-btw)) и `openclaw doctor`.

Общее описание разделения моделей, провайдеров и сред выполнения см. в разделе
[Среды выполнения агентов](/ru/concepts/agent-runtimes).

## Требования

- OpenClaw с установленным plugin `@openclaw/copilot`.
- Если в конфигурации используется `plugins.allow`, добавьте `copilot` (идентификатор, объявленный в манифесте plugin). Запись имени npm-пакета `@openclaw/copilot` в списке разрешений не совпадёт с идентификатором, поэтому plugin останется заблокированным даже при заданном `agentRuntime.id: "copilot"`.
- Подписка GitHub Copilot, позволяющая использовать Copilot CLI, либо переменная окружения `gitHubToken` / запись профиля аутентификации для автономных запусков или запусков Cron.
- Доступный для записи каталог `copilotHome`. По умолчанию используется `<agentDir>/copilot`, если OpenClaw предоставляет каталог агента, иначе — `~/.openclaw/agents/<agentId>/copilot`.

`openclaw doctor` выполняет [контракт doctor](#doctor) plugin для управления состоянием сеанса и будущих миграций конфигурации. Среду Copilot CLI он не проверяет.

## Установка

Среда выполнения Copilot поставляется как внешний plugin, поэтому основной пакет `openclaw` не содержит `@github/copilot-sdk` и зависящий от платформы бинарный файл CLI `@github/copilot-<platform>-<arch>` (вместе около 260 МБ). Устанавливайте его только для агентов, которые используют эту среду выполнения:

```bash
openclaw plugins install @openclaw/copilot
```

Мастер настройки автоматически устанавливает plugin при первом выборе модели `github-copilot/*`, **если** конфигурация направляет эту модель (или её провайдер) в среду выполнения Copilot через `agentRuntime: { id: "copilot" }`; см. [Быстрый старт](#quickstart). Без такого явного выбора OpenClaw использует встроенный провайдер GitHub Copilot и не устанавливает этот plugin.

Среда выполнения разрешает SDK в следующем порядке:

1. `import("@github/copilot-sdk")` из установленного пакета `@openclaw/copilot`.
2. Резервный каталог `~/.openclaw/npm-runtime/copilot/` (устаревшее расположение для установки по требованию).

Если SDK отсутствует, возвращается одна ошибка с кодом `COPILOT_SDK_MISSING` и приведённой выше командой переустановки.

## Быстрый старт

Закрепите одну модель (или одного провайдера) за средой выполнения:

```json5
{
  agents: {
    defaults: {
      model: "github-copilot/auto",
      models: {
        "github-copilot/auto": {
          agentRuntime: { id: "copilot" },
        },
      },
    },
  },
}
```

Задайте `agentRuntime.id` в записи одной модели, чтобы направить через эту среду выполнения только её, либо в записи провайдера, чтобы направить через неё все модели этого провайдера.

`github-copilot/auto` — переносимая отправная точка. Доступность именованных моделей Copilot зависит от учётной записи и политики организации; перед закреплением модели убедитесь, что аутентифицированный Copilot CLI действительно предоставляет её.

## Поддерживаемые провайдеры

Среда выполнения поддерживает канонический провайдер `github-copilot` (принадлежащий `extensions/github-copilot`), а также пользовательские записи `models.providers`, если у модели задан непустой `baseUrl` и одна из следующих форм `api`:

- `anthropic-messages`
- `azure-openai-responses`
- `ollama` (OpenAI-совместимые дополнения)
- `openai-completions`
- `openai-responses`

Нативные идентификаторы провайдеров (`openai`, `anthropic`, `google`, `ollama`) остаются в ведении соответствующих нативных сред выполнения. Чтобы направить конечную точку через Copilot BYOK, используйте отдельный идентификатор пользовательского провайдера.

Конечные точки Copilot BYOK должны быть общедоступными URL-адресами HTTPS. Среда выполнения предоставляет Copilot SDK отдельный local loopback-прокси для каждой попытки, а затем перенаправляет трафик провайдера через защищённый путь запросов OpenClaw, чтобы закрепление DNS и политика SSRF оставались под управлением OpenClaw. Для локальных серверов моделей Ollama, LM Studio или серверов в локальной сети используйте нативную среду выполнения OpenClaw.

## BYOK

Copilot BYOK использует контракт пользовательского провайдера SDK на уровне сеанса. OpenClaw передаёт разрешённую конечную точку модели, ключ API, режим токена носителя, заголовки, идентификатор модели и ограничения контекста/вывода; транспортная логика провайдера остаётся в SDK, а не в основном коде.

```json5
{
  agents: {
    defaults: {
      model: "custom-proxy/llama-3.1-8b",
      models: {
        "custom-proxy/llama-3.1-8b": {
          agentRuntime: { id: "copilot" },
        },
      },
    },
  },
  models: {
    mode: "merge",
    providers: {
      "custom-proxy": {
        baseUrl: "https://api.example.com/v1",
        apiKey: "${CUSTOM_PROXY_API_KEY}",
        api: "openai-responses",
        authHeader: true,
        models: [{ id: "llama-3.1-8b", name: "Llama 3.1 8B" }],
      },
    },
  },
}
```

Сеансы BYOK получают отдельные ключи относительно сеансов подписки и других конечных точек или учётных данных BYOK. Смена ключа, заголовков, модели или конечной точки запускает новый сеанс Copilot SDK вместо возобновления несовместимого состояния.

## Аутентификация

Приоритет, применяемый для каждого агента во время `runCopilotAttempt`:

1. **Явное `useLoggedInUser: true`** во входных данных попытки — используется вошедший в Copilot CLI пользователь из каталога `copilotHome` агента.
2. **Явный `gitHubToken`** во входных данных попытки (требуются `profileId` и `profileVersion`). Предназначен для прямых вызовов CLI и тестов, которым нужно обойти разрешение профиля аутентификации.
3. **Разрешённые по контракту `resolvedApiKey` и `authProfileId`** — основной рабочий путь. Перед вызовом среды выполнения основной код разрешает настроенный для агента профиль аутентификации `github-copilot` (`src/infra/provider-usage.auth.ts:resolveProviderAuths`), поэтому профиль аутентификации `github-copilot:<profile>` полностью работает для автономных запусков, Cron и конфигураций с несколькими профилями без переменных окружения.
4. **Резервное использование переменных окружения**, проверяемых в следующем порядке (используется первое непустое значение; пустые строки считаются отсутствующими; порядок соответствует поставляемому провайдеру `github-copilot` в `extensions/github-copilot/auth.ts`):
   1. `OPENCLAW_GITHUB_TOKEN` — переопределение для среды выполнения; позволяет закрепить токен за средой выполнения OpenClaw, не изменяя общесистемную конфигурацию `gh` / Copilot CLI.
   2. `COPILOT_GITHUB_TOKEN` — стандартная переменная окружения Copilot SDK / CLI.
   3. `GH_TOKEN` — стандартная переменная окружения CLI `gh`.
   4. `GITHUB_TOKEN` — резервный универсальный токен GitHub.

   Синтезированный идентификатор профиля пула имеет вид `env:<NAME>`; версия профиля представляет собой необратимый отпечаток токена sha256, поэтому смена значения переменной окружения корректно сбрасывает пул клиентов.

5. **`useLoggedInUser` по умолчанию**, если сигнал токена отсутствует.

Каждый агент получает собственный каталог `copilotHome`, поэтому токены, сеансы и конфигурация Copilot CLI никогда не передаются между агентами на одном компьютере. По умолчанию используется `<agentDir>/copilot` (состояние SDK хранится отдельно от `models.json` / `auth-profiles.json` OpenClaw) или `~/.openclaw/agents/<agentId>/copilot`, если каталог агента не предоставлен. Чтобы указать другое расположение (например, общий подключённый том для миграции), задайте `copilotHome: <path>` во входных данных попытки.

Интерактивные тесты среды выполнения используют `OPENCLAW_COPILOT_AGENT_LIVE_TOKEN` для прямой передачи токена. Общая настройка интерактивных тестов очищает `COPILOT_GITHUB_TOKEN`, `GH_TOKEN` и `GITHUB_TOKEN` после размещения реальных профилей аутентификации в изолированном домашнем каталоге тестов, поэтому значение `gh auth token`, переданное через специальную переменную, предотвращает ложные пропуски и не попадает в несвязанные наборы тестов.

## Параметры конфигурации

Среда выполнения читает конфигурацию из входных данных каждой попытки (`runCopilotAttempt({...})`) и небольшого набора значений переменных окружения по умолчанию внутри `extensions/copilot/src/`:

| Поле                     | Назначение                                                                                                                                                                                                                                                                                                                                                                 |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `copilotHome`            | Каталог состояния CLI для отдельного агента (значения по умолчанию приведены выше).                                                                                                                                                                                                                                                                                         |
| `model`                  | Строка или `{ provider, id, api?, baseUrl?, headers?, authHeader? }`. Не указывайте, чтобы использовать обычный выбор модели агента; среда выполнения проверяет, что разрешённый провайдер поддерживается.                                                                                                                                                                     |
| `reasoningEffort`        | `"low" \| "medium" \| "high" \| "xhigh"`. Сопоставляется с результатом разрешения `ThinkLevel` / `ReasoningLevel` OpenClaw в `auto-reply/thinking.ts`.                                                                                                                                                                                                                       |
| `infiniteSessionConfig`  | Необязательное переопределение блока SDK `infiniteSessions`, управляемого `harness.compact`. Можно безопасно оставить без изменений.                                                                                                                                                                                                                                         |
| `hooksConfig`            | Необязательная нативная конфигурация Copilot SDK `SessionHooks` для обратных вызовов инструментов/MCP, пользовательских запросов, сеанса и ошибок. Не связана с переносимыми хуками жизненного цикла OpenClaw.                                                                                                                                                                  |
| `permissionPolicy`       | Необязательное переопределение обработчика SDK `onPermissionRequest` для встроенных типов инструментов SDK (`shell`, `write`, `read`, `url`, `mcp`, `memory`, `hook`). В качестве защитного механизма по умолчанию используется `rejectAllPolicy`; почему он фактически никогда не вызывается, см. в разделе [Разрешения и ask_user](#permissions-and-ask_user). |
| `enableSessionTelemetry` | Необязательный флаг телеметрии сеанса SDK.                                                                                                                                                                                                                                                                                                                                  |

Для хуков plugin OpenClaw не нужна конфигурация попытки, специфичная для Copilot. Среда выполнения запускает `before_prompt_build` (и устаревший хук совместимости `before_agent_start`), `llm_input`, `llm_output` и `agent_end` через стандартные вспомогательные функции среды выполнения. При успешной Compaction SDK также запускаются `before_compaction` и `after_compaction`. Для инструментов OpenClaw, подключённых через мост, запускается `before_tool_call` и передаётся результат в `after_tool_call`; `hooksConfig` остаётся предназначенным для нативных обратных вызовов только SDK, не имеющих переносимого эквивалента.

Остальным компонентам OpenClaw ничего не нужно знать об этих полях. Другие plugins, каналы и основной код видят только стандартную структуру `AgentHarnessAttemptParams` / `AgentHarnessAttemptResult`.

## Compaction

При выполнении `harness.compact` среда выполнения Copilot SDK:

1. Возобновляет отслеживаемый сеанс SDK, не продолжая ожидающую работу.
2. Вызывает RPC сжатия истории в SDK на уровне сеанса.
3. Возвращает результат Compaction SDK, не записывая файлы маркеров совместимости в рабочую область.

Зеркальная копия стенограммы на стороне OpenClaw (см. ниже) продолжает получать сообщения после Compaction, поэтому видимая пользователю история чата остаётся согласованной.

## Зеркалирование стенограммы

`runCopilotAttempt` параллельно записывает отражаемые сообщения каждого цикла в аудиторскую стенограмму OpenClaw через `extensions/copilot/src/dual-write-transcripts.ts`. Зеркальная копия ограничена отдельным сеансом (`copilot:${sessionId}`), а ключ каждого сообщения имеет вид `${role}:${sha256_16(role,content)}`, поэтому повторно отправленные записи предыдущих циклов совпадают с существующими ключами на диске и не дублируются.

Два уровня локализации сбоев оборачивают зеркало, поэтому ошибка записи транскрипта
никогда не приводит к сбою попытки: внутренняя обёртка с выполнением по мере
возможности и дополнительный `.catch(...)` на уровне попытки. Ошибки записываются
в журнал, но не передаются наружу.

## Дополнительные вопросы (`/btw`)

`/btw` **не** является встроенной возможностью этого средства запуска. `createCopilotAgentHarness()`
намеренно оставляет `harness.runSideQuestion` неопределённым
(это проверяется в `extensions/copilot/harness.test.ts`, `describe("runSideQuestion")`),
поэтому диспетчер `/btw` в OpenClaw (`src/agents/btw.ts`) переходит на тот же
путь, который используется для каждой среды выполнения, отличной от Codex:
настроенный поставщик модели вызывается напрямую с коротким запросом
дополнительного вопроса, а ответ передаётся в потоковом режиме через
`streamSimple` (без сеанса CLI и без дополнительного слота в пуле).

Это резервирует сеансы Copilot CLI для основного цикла хода агента и
обеспечивает для `/btw` поведение, идентичное другим средам выполнения,
отличным от Codex.

## Doctor

`extensions/copilot/doctor-contract-api.ts` автоматически загружается
`src/plugins/doctor-contract-registry.ts`. Он предоставляет:

- Пустой `legacyConfigRules` (устаревших полей пока нет).
- Ничего не выполняющий `normalizeCompatibilityConfig` (сохранён, чтобы будущие
  выводы полей из эксплуатации имели стабильное место в дереве исходного кода).
- Одну запись `sessionRouteStateOwners`: поставщик `github-copilot`, среда
  выполнения `copilot`, ключ сеанса CLI `copilot`, префикс профиля аутентификации
  `github-copilot:`.

## Ограничения

- Средство запуска заявляет `github-copilot`, а также пользовательские
  идентификаторы поставщиков BYOK без владельца. Идентификаторы встроенных
  поставщиков, принадлежащие манифестам, остаются в среде выполнения своего
  владельца, даже когда для `agentRuntime.id` принудительно задано `copilot`.
- Интерфейс TUI отсутствует; TUI среды PI остаётся резервным вариантом для сред
  выполнения без собственного интерфейса.
- Состояние сеанса PI не переносится, когда агент переключается на `copilot`.
  Выбор выполняется для каждой попытки; существующие сеансы PI остаются
  действительными.
- `ask_user` использует тот же механизм запросов и ответов OpenClaw, что и
  средство запуска Codex: когда Copilot SDK запрашивает ввод пользователя,
  OpenClaw отправляет блокирующий запрос в активный канал или TUI, а следующее
  сообщение пользователя в очереди разрешает запрос SDK.

## Разрешения и ask_user

Проверка разрешений для инструментов OpenClaw, подключённых через мост,
выполняется **внутри обёртки инструмента**, а не через обратный вызов SDK
`onPermissionRequest`. Тот же `wrapToolWithBeforeToolCallHook`, который
использует PI (`src/agents/agent-tools.before-tool-call.ts`), применяется
`createOpenClawCodingTools` к каждому инструменту программирования: обнаружение
циклов, политики доверенных плагинов, перехватчики перед вызовом инструмента и
двухэтапные подтверждения плагинов через Gateway (`plugin.approval.request`) —
всё выполняется по тому же пути кода, что и во встроенных попытках PI.

Инструмент SDK, возвращаемый `convertOpenClawToolToSdkTool`, помечается
следующими параметрами:

- `overridesBuiltInTool: true` — заменяет встроенный инструмент Copilot CLI
  с тем же именем (редактирование, чтение, запись, bash, ...) так, чтобы каждый
  вызов инструмента направлялся обратно в OpenClaw.
- `skipPermission: true` — указывает SDK не вызывать
  `onPermissionRequest({kind: "custom-tool"})` перед запуском инструмента.
  Обёрнутый `execute()` уже выполняет более содержательную проверку политик
  OpenClaw; запрос на уровне SDK либо обходил бы проверку OpenClaw
  (разрешать всё), либо блокировал бы каждый вызов инструмента (отклонять всё) —
  ни один из вариантов не обеспечивает соответствие PI.

Входящее в дерево исходного кода средство запуска Codex использует то же
разделение: подключённые через мост инструменты OpenClaw оборачиваются
(`extensions/codex/src/app-server/dynamic-tools.ts`), а собственные встроенные
виды подтверждений codex-app-server
(`item/commandExecution/requestApproval`, `item/fileChange/requestApproval`,
`item/permissions/requestApproval`) направляются через `plugin.approval.request`
(`extensions/codex/src/app-server/approval-bridge.ts`). Эквивалент в Copilot SDK —
закрытая по умолчанию политика `rejectAllPolicy` для любого вида, отличного от
`custom-tool`, который когда-либо достигает `onPermissionRequest`, — служит той
же страховочной мерой и на практике никогда не срабатывает, поскольку
`overridesBuiltInTool: true` вытесняет каждый встроенный инструмент.

Чтобы уровень обёрнутых инструментов мог принимать решения по политикам,
эквивалентные PI, средство запуска передаёт в `createOpenClawCodingTools` полный
контекст инструментов попытки PI: идентификационные данные (`senderIsOwner`,
`memberRoleIds`, `ownerOnlyToolAllowlist`, ...), канал и маршрутизацию (`groupId`,
`currentChannelId`, `replyToMode`, переключатели инструментов сообщений),
аутентификацию (`authProfileStore`), идентификацию запуска (`sessionKey` /
`runSessionKey`, полученные из `sandboxSessionKey`, `runId`), контекст модели
(`modelApi`, `modelContextWindowTokens`, `modelCompat`, `modelHasVision`) и
перехватчики запуска (`onToolOutcome`, `onYield`). Без этих полей списки
разрешений только для владельца по умолчанию незаметно запрещают доступ,
политики доверия к плагинам не могут определить правильную область действия,
а `session_status: "current"` разрешается в устаревший ключ песочницы.
Построитель моста находится в `extensions/copilot/src/tool-bridge.ts` и
повторяет эталонный вызов PI из
`src/agents/embedded-agent-runner/run/attempt.ts:1262`.
`runAttempt` определяет контекст песочницы через общий механизм
`resolveSandboxContext`, передаёт SDK фактический рабочий каталог и передаёт
`sandbox` вместе с рабочей областью создания подагента в мост инструментов.
Мост также передаёт ограниченные параметры построения инструментов, соблюдение
которых он может обеспечить на границе SDK: `includeCoreTools`, список
разрешённых инструментов среды выполнения и `toolConstructionPlan`.

Для соответствия PI мост также использует общий вспомогательный механизм
поверхности инструментов средства запуска из
`openclaw/plugin-sdk/agent-harness-tool-runtime`. Когда включён поиск
инструментов, SDK получает компактные управляющие инструменты и скрытый
исполнитель каталога вместо схем каждого инструмента OpenClaw. Когда включён
режим кода, вспомогательный механизм создаёт ту же управляющую поверхность
режима кода и тот же жизненный цикл каталога, которые используются другими
средствами запуска агентов. Облегчённые значения по умолчанию для локальных
моделей, совместимая со средой выполнения фильтрация схем, заполнение каталогов
и очистка каталога остаются в общем вспомогательном механизме, чтобы средства
запуска Copilot и смежные с Codex средства запуска не расходились в поведении.

### Токен GitHub уровня сеанса

Контракт Copilot SDK различает токен GitHub **уровня клиента**
(`CopilotClientOptions.gitHubToken`, аутентифицирует сам процесс CLI) и токен
**уровня сеанса** (`SessionConfig.gitHubToken`, определяет исключение содержимого,
маршрутизацию моделей и квоту для этого сеанса; учитывается как при
`createSession`, так и при `resumeSession`). Средство запуска однократно
определяет аутентификацию через `resolveCopilotAuth` и задаёт оба поля, когда
режим аутентификации равен `gitHubToken` (явный `auth.gitHubToken` или
определённый контрактом `resolvedApiKey` из настроенного профиля аутентификации
`github-copilot`). Когда определён режим `useLoggedInUser`, поле уровня сеанса
опускается, чтобы SDK продолжал получать идентификационные данные от вошедшего
в систему пользователя.

`ask_user` использует `SessionConfig.onUserInputRequest`. Мост принимает индексы
или метки вариантов для запросов с фиксированным выбором, принимает ответы в
свободной форме, когда запрос SDK их допускает, и отменяет ожидающий запрос,
когда попытка OpenClaw прерывается.
