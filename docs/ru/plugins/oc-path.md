Встроенный plugin `oc-path` добавляет CLI [`openclaw path`](/ru/cli/path) для
схемы адресации файлов рабочего пространства `oc://`. Он поставляется в
репозитории OpenClaw в каталоге `extensions/oc-path/`, но требует явного
включения: после установки или сборки он остаётся неактивным, пока вы его не
включите.

Адреса `oc://` указывают на один конечный элемент (или набор конечных элементов
по шаблону) внутри файла рабочего пространства. Plugin поддерживает четыре
типа файлов:

- **markdown** (`.md`): frontmatter, разделы, элементы, поля
- **jsonc** (`.jsonc`, `.json`): с сохранением комментариев и форматирования
- **jsonl** (`.jsonl`, `.ndjson`): записи, ориентированные на строки
- **yaml** (`.yaml`, `.yml`, `.lobster`): узлы отображений, последовательностей
  и скалярных значений через API `Document` пакета `yaml`

Пользователи собственных развёртываний и расширения редакторов используют CLI,
чтобы читать или записывать отдельный конечный элемент без непосредственного
написания скриптов для SDK; агенты и хуки используют его как детерминированную
основу, чтобы точные до байта циклы чтения и записи и защита сигнального
значения редактирования единообразно применялись ко всем типам файлов. Полную
грамматику, список флагов для каждой команды и практические примеры для каждого
типа файлов см. в [справочнике по CLI](/ru/cli/path); на этой странице объясняется,
зачем и как включать plugin.

## Зачем его включать

Включите `oc-path`, если скриптам, хукам или локальным агентским инструментам
нужно обращаться к конкретной части состояния рабочего пространства без
отдельного специализированного парсера для каждой структуры файла. Один адрес
`oc://` может обозначать ключ frontmatter в Markdown, элемент раздела, конечный
элемент конфигурации JSONC, поле события JSONL или шаг рабочего процесса YAML.

Это важно для рабочих процессов сопровождающих, в которых изменение должно
оставаться небольшим, проверяемым и воспроизводимым: просмотреть одно значение,
найти соответствующие записи, выполнить пробную запись, а затем применить
изменение только к этому конечному элементу, не затрагивая комментарии,
окончания строк и соседнее форматирование.

Распространённые причины включить его:

- **Локальная автоматизация**: скрипты оболочки разрешают или обновляют одно
  значение рабочего пространства с помощью `openclaw path … --json`, не
  включая отдельный код разбора Markdown, JSONC, JSONL и YAML.
- **Изменения, видимые агенту**: перед записью агент показывает различия
  пробного изменения одного адресуемого конечного элемента, что проще
  проверить, чем произвольную перезапись файла.
- **Интеграции с редакторами**: редактор сопоставляет
  `oc://AGENTS.md/tools/gh` с точным узлом Markdown и номером строки, не
  пытаясь определить их по тексту заголовка.
- **Диагностика**: `emit` пропускает файл через парсер и генератор вывода,
  чтобы можно было проверить побайтовую стабильность типа файла, прежде чем
  полагаться на автоматические изменения.

```bash
# Включён ли plugin GitHub в этой конфигурации?
openclaw path resolve 'oc://config.jsonc/plugins/github/enabled' --json

# Какие имена вызовов инструментов встречаются в этом журнале сеанса?
openclaw path find 'oc://session.jsonl/[event=tool_call]/name' --json

# Какие байты будут записаны при этом небольшом изменении конфигурации?
openclaw path set 'oc://config.jsonc/plugins/github/enabled' 'true' --dry-run
```

`oc-path` намеренно не отвечает за семантику более высокого уровня. Plugins
памяти по-прежнему отвечают за запись памяти, команды конфигурации — за полное
управление конфигурацией, а восстановление последней заведомо исправной
конфигурации (LKG) — за восстановление и продвижение. `oc-path` представляет
собой узкий слой адресации и файловых операций с сохранением байтов, вокруг
которого могут строиться эти инструменты более высокого уровня.

## Где он выполняется

Plugin выполняется **внутри процесса CLI `openclaw`** на узле, где вы вызываете
команду. Для него не требуется работающий Gateway, и он не открывает сетевые
сокеты; каждая команда представляет собой чистое преобразование указанного
вами файла.

Метаданные plugin находятся в `extensions/oc-path/openclaw.plugin.json`:

```json
{
  "id": "oc-path",
  "name": "OC Path",
  "activation": {
    "onStartup": false,
    "onCommands": ["path"]
  },
  "commandAliases": [{ "name": "path", "kind": "cli" }]
}
```

`onStartup: false` исключает plugin из пути запуска Gateway.
`commandAliases` и `activation.onCommands` указывают CLI лениво загрузить
plugin при первом выполнении `openclaw path …`, поэтому установки, в которых
эта команда не используется, не несут дополнительных затрат.

## Включение

```bash
openclaw plugins enable oc-path
```

Перезапустите Gateway (если вы его используете), чтобы снимок манифеста
учитывал новое состояние. Отдельные вызовы `openclaw path` сразу работают на
том же узле; CLI загружает plugin по запросу.

Отключение:

```bash
openclaw plugins disable oc-path
```

## Зависимости

Все зависимости парсеров являются локальными для plugin; включение `oc-path`
не добавляет новые пакеты в основную среду выполнения:

| Зависимость    | Назначение                                                               |
| -------------- | ------------------------------------------------------------------------ |
| `commander`    | Подключение подкоманд `resolve`, `find`, `set`, `validate`, `emit`.      |
| `jsonc-parser` | Разбор JSONC и изменение конечных элементов с сохранением комментариев и завершающих запятых. |
| `markdown-it`  | Токенизация Markdown для модели разделов, элементов и полей.             |
| `yaml`         | Разбор, вывод и изменение `Document` YAML с сохранением комментариев и потокового стиля. |

JSONL по-прежнему обрабатывается вручную: построчный разбор проще любой
зависимости, а разбор каждой строки уже выполняется через `jsonc-parser`.

## Предоставляемые возможности

| Поверхность                     | Предоставляется                                            |
| ------------------------------- | ---------------------------------------------------------- |
| CLI `openclaw path`             | `extensions/oc-path/cli-registration.ts`                   |
| Парсер и форматтер `oc://`      | `extensions/oc-path/src/oc-path/oc-path.ts`                |
| Разбор, вывод и изменение по типам | `extensions/oc-path/src/oc-path/{md,jsonc,jsonl,yaml}`  |
| Универсальные resolve / find / set | `extensions/oc-path/src/oc-path/{resolve,find,edit}.ts` |
| Защита сигнального значения редактирования | `extensions/oc-path/src/oc-path/sentinel.ts`     |

На данный момент CLI — единственная общедоступная поверхность. Команды
базового слоя являются внутренними для plugin; потребители используют CLI
(или создают собственный plugin на основе SDK).

## Связь с другими plugins

- **`memory-*`**: запись памяти выполняется через plugins памяти, а не через
  `oc-path`. `oc-path` — универсальный файловый базовый слой; plugins памяти
  добавляют поверх него собственную семантику.
- **LKG**: `path` ничего не знает о восстановлении последней заведомо исправной
  конфигурации. Если файл, изменяемый через `path`, также отслеживается LKG,
  следующий цикл наблюдения за конфигурацией решает, продвигать его или
  восстанавливать; рассматривайте изменение через `path` так же, как любую
  другую прямую запись в этот файл.

## Безопасность

`set` записывает необработанные байты через путь вывода базового слоя, который
автоматически применяет защиту сигнального значения редактирования. Запись
конечного элемента, содержащего `__OPENCLAW_REDACTED__` (дословно или как
подстроку), отклоняется с ошибкой `OC_EMIT_SENTINEL`. CLI также удаляет
буквальное сигнальное значение из любого выводимого текста или JSON, заменяя
его на `[REDACTED]`, чтобы снимки терминала и конвейеры никогда не раскрывали
этот маркер.
