OpenClaw выполняет работу с зависимостями Plugin во время установки/обновления. Загрузка во время выполнения
не запускает менеджеры пакетов, не чинит деревья зависимостей и не изменяет каталог
пакета OpenClaw.

## Разделение ответственности

Пакеты Plugin сами отвечают за свой граф зависимостей:

- зависимости времени выполнения находятся в `dependencies` или
  `optionalDependencies` пакета Plugin
- импорты SDK/ядра являются peer-зависимостями или импортами, предоставляемыми OpenClaw
- локальные Plugin для разработки приносят собственные уже установленные зависимости
- npm- и git-Plugin устанавливаются в корни пакетов, принадлежащие OpenClaw

OpenClaw отвечает только за жизненный цикл Plugin:

- обнаружить источник Plugin
- установить или обновить пакет по явному запросу
- записать метаданные установки
- загрузить точку входа Plugin
- завершиться с полезной ошибкой, когда зависимости отсутствуют

## Корни установки

OpenClaw использует стабильные корни для каждого источника:

- npm-пакеты устанавливаются в проекты для отдельных Plugin в
  `~/.openclaw/npm/projects/<encoded-package>`
- git-пакеты клонируются в `~/.openclaw/git`
- локальные установки/установки из пути/архива копируются или подключаются по ссылке без исправления зависимостей

npm-установки выполняются в корне такого проекта для отдельного Plugin с:

```bash
cd ~/.openclaw/npm/projects/<encoded-package>
npm install --omit=dev --omit=peer --legacy-peer-deps --ignore-scripts --no-audit --no-fund
```

`openclaw plugins install npm-pack:<path.tgz>` использует тот же корень npm-проекта
для отдельного Plugin для локального tarball npm-pack. OpenClaw читает npm-метаданные
tarball, добавляет его в управляемый проект как скопированную зависимость `file:`, выполняет
обычную npm-установку, а затем проверяет метаданные установленного lockfile, прежде чем
доверять Plugin.
Это предназначено для подтверждения приемки пакета и релиз-кандидата, когда
локальный артефакт pack должен вести себя как имитируемый им артефакт реестра.

Используйте `npm-pack:` при тестировании официальных или внешних пакетов Plugin перед
публикацией. Установка из необработанного архива или пути полезна для локальной отладки, но она
не подтверждает тот же путь зависимостей, что установленный npm- или ClawHub-пакет.
`npm-pack:` подтверждает форму установки управляемого пакета; само по себе это
не является доказательством, что Plugin является официальным содержимым, связанным с каталогом.

Когда поведение зависит от статуса встроенного Plugin или доверенного официального Plugin, сочетайте
подтверждение локального пакета с официальной установкой, подкрепленной каталогом, или с опубликованным
путем пакета, который записывает официальное доверие. Доступ к привилегированным помощникам и
обработка доверенной официальной области должны проверяться на этом доверенном пути установки,
а не выводиться из установки локального tarball.

Если Plugin во время выполнения завершается с отсутствующим импортом, исправьте манифест пакета,
а не чините управляемый проект вручную. Импорты времени выполнения должны находиться в
`dependencies` или `optionalDependencies` пакета Plugin; `devDependencies` не
устанавливаются для управляемых проектов времени выполнения. Локальный `npm install` внутри
`~/.openclaw/npm/projects/<encoded-package>` может временно разблокировать диагностику,
но он не является подтверждением приемки пакета, потому что следующая установка или обновление
пересоздаст проект из метаданных пакета.

npm может поднимать транзитивные зависимости в `node_modules` проекта для отдельного Plugin
рядом с пакетом Plugin. OpenClaw сканирует корень управляемого проекта
перед тем, как доверять установке, и удаляет этот проект во время удаления, поэтому
поднятые зависимости времени выполнения остаются внутри границы очистки этого Plugin.

Опубликованные npm-пакеты Plugin могут поставлять `npm-shrinkwrap.json`. npm использует этот
публикуемый lockfile во время установки, а корень управляемого npm-проекта OpenClaw
поддерживает его через обычный путь npm-установки. Принадлежащие OpenClaw публикуемые
пакеты Plugin должны включать локальный для пакета shrinkwrap, сгенерированный из
публикуемого графа зависимостей этого пакета Plugin:

```bash
pnpm deps:shrinkwrap:generate
pnpm deps:shrinkwrap:check
```

Генератор удаляет `devDependencies` Plugin, применяет политику переопределений workspace
и записывает `extensions/<id>/npm-shrinkwrap.json` для каждого Plugin
`publishToNpm`. Сторонние пакеты Plugin также могут поставлять shrinkwrap;
OpenClaw не требует его для пакетов сообщества, но npm будет учитывать его
при наличии.

Перед тем как считать локальный пакет подтверждением релиз-кандидата, проверьте tarball,
который будет установлен:

```bash
npm pack --pack-destination /tmp
tar -xOf /tmp/<plugin-package>.tgz package/package.json
tar -tf /tmp/<plugin-package>.tgz | grep '^package/dist/'
```

Для изменений зависимостей также проверьте, что производственная установка может разрешить
пакеты времени выполнения без dev-зависимостей:

```bash
tmpdir=$(mktemp -d)
(
  cd "$tmpdir"
  npm init -y >/dev/null
  npm install --package-lock-only --omit=dev --omit=peer --legacy-peer-deps --ignore-scripts /tmp/<plugin-package>.tgz
)
rm -rf "$tmpdir"
```

Принадлежащие OpenClaw npm-пакеты Plugin также могут публиковаться с явными
`bundledDependencies`. Путь публикации npm накладывает список имен зависимостей времени выполнения,
удаляет dev-only метаданные workspace из манифеста публикуемого пакета,
запускает npm-установку без скриптов для локальных для пакета зависимостей времени выполнения,
а затем упаковывает или публикует tarball Plugin с включенными файлами этих зависимостей. Пакеты
с большим количеством нативного кода, включая среды выполнения Codex и ACP, отключают это
с помощью `openclaw.release.bundleRuntimeDependencies: false`; эти пакеты все равно
поставляют свой shrinkwrap, но npm разрешает зависимости времени выполнения во время установки
вместо встраивания каждого платформенного бинарного файла в tarball Plugin. Корневой
пакет `openclaw` не включает в себя все дерево зависимостей.

Plugin, импортирующие `openclaw/plugin-sdk/*`, объявляют `openclaw` peer-зависимостью.
OpenClaw не позволяет npm установить отдельную копию пакета хоста из реестра
в управляемый проект, потому что устаревшие пакеты хоста могут влиять на npm
peer-разрешение внутри этого Plugin. Управляемые npm-установки пропускают npm
peer-разрешение/материализацию, а OpenClaw после установки или обновления повторно закрепляет
локальные для Plugin ссылки `node_modules/openclaw` для установленных пакетов, которые объявляют peer хоста.

git-установки клонируют или обновляют репозиторий, затем выполняют:

```bash
npm install --omit=dev --ignore-scripts --no-audit --no-fund
```

Затем установленный Plugin загружается из этого каталога пакета, поэтому разрешение
локальных для пакета и родительских `node_modules` работает так же, как для обычного
пакета Node.

## Локальные Plugin

Локальные Plugin рассматриваются как каталоги, контролируемые разработчиком. OpenClaw не
запускает для них `npm install`, `pnpm install` или исправление зависимостей. Если у локального
Plugin есть зависимости, установите их в этом Plugin перед загрузкой.

Сторонние локальные Plugin на TypeScript могут использовать аварийный путь Jiti. Упакованные
Plugin на JavaScript и встроенные внутренние Plugin загружаются через нативные
import/require вместо Jiti.

## Запуск и перезагрузка

Запуск Gateway и перезагрузка конфигурации никогда не устанавливают зависимости Plugin. Они читают
записи установки Plugin, вычисляют точку входа и загружают ее.

Если зависимость отсутствует во время выполнения, Plugin не загружается, а ошибка
должна указывать оператору на явное исправление:

```bash
openclaw plugins update <id>
openclaw plugins install <source>
openclaw doctor --fix
```

`doctor --fix` может очищать устаревшее состояние зависимостей, сгенерированное OpenClaw, и восстанавливать
загружаемые Plugin, отсутствующие в локальных записях установки, когда конфигурация
ссылается на них. Doctor не исправляет зависимости для уже установленного
локального Plugin.

## Встроенные Plugin

Легковесные и критически важные для ядра встроенные Plugin поставляются как часть OpenClaw.
У них либо не должно быть тяжелого дерева зависимостей времени выполнения, либо их следует вынести в
загружаемый пакет на ClawHub/npm.

Текущий сгенерированный список Plugin, которые поставляются в пакете ядра, устанавливаются
внешне или остаются только исходным кодом, см. в [Инвентаре Plugin](/ru/plugins/plugin-inventory).

Манифесты встроенных Plugin не должны запрашивать подготовку зависимостей. Крупная или опциональная
функциональность Plugin должна быть упакована как обычный Plugin и установлена через
тот же путь npm/git/ClawHub, что и сторонние Plugin.

В исходных checkout OpenClaw рассматривает репозиторий как pnpm-монорепозиторий. После
`pnpm install` встроенные Plugin загружаются из `extensions/<id>`, поэтому локальные для пакета
workspace-зависимости доступны, а изменения подхватываются напрямую. Разработка в исходном
checkout поддерживает только pnpm; обычный `npm install` в корне репозитория
не является поддерживаемым способом подготовить зависимости встроенных Plugin.

| Форма установки                 | Расположение встроенного Plugin       | Владелец зависимостей                                                |
| -------------------------------- | ------------------------------------- | -------------------------------------------------------------------- |
| `npm install -g openclaw`        | Собранное дерево времени выполнения внутри пакета | Пакет OpenClaw и явные потоки установки/обновления/doctor Plugin     |
| Git checkout плюс `pnpm install` | Workspace-пакеты `extensions/<id>`    | Workspace pnpm, включая собственные зависимости каждого пакета Plugin |
| `openclaw plugins install ...`   | Корень управляемого npm-проекта/git/ClawHub | Поток установки/обновления Plugin                                    |

## Очистка legacy

Старые версии OpenClaw генерировали корни зависимостей встроенных Plugin при запуске или
во время исправления doctor. Текущая очистка doctor удаляет эти устаревшие каталоги и
символические ссылки при использовании `--fix`, включая старые корни `plugin-runtime-deps`, глобальные
символические ссылки пакетов Node-prefix, указывающие на удаленные цели `plugin-runtime-deps`,
манифесты `.openclaw-runtime-deps*`, сгенерированные `node_modules` Plugin, каталоги
этапа установки и локальные для пакета хранилища pnpm. Упакованный postinstall также
удаляет эти глобальные символические ссылки перед удалением устаревших целевых корней, чтобы обновления
не оставляли оборванные ESM-импорты пакетов.

Старые npm-установки также использовали общий корень `~/.openclaw/npm/node_modules`.
Текущие потоки установки, обновления, удаления и doctor по-прежнему распознают этот устаревший
плоский корень только для восстановления и очистки. Новые npm-установки должны создавать
корни проектов для отдельных Plugin.
