# 组织和命名空间声明

ClawHub 将所有者用户名、组织用户名、技能 slug、插件包名称和
包作用域用作公共命名空间。如果某个命名空间看起来属于现实中的
项目、品牌、包生态系统或组织，但已在 ClawHub 上被占用、保留、存在误导性或争议，请要求工作人员通过
[组织/命名空间声明 issue 表单](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
进行审核。

此途径适用于公开的非敏感所有权审核。请勿使用产品内
举报或账户申诉表单提出命名空间声明。

## 何时提出声明

如果你认为 ClawHub 工作人员应根据现实中的所有权，审核某个
命名空间是否应予以保留、转让、重命名、隐藏、隔离、设置别名
或进行其他更改，请提出命名空间声明。

示例包括：

- 与 GitHub 组织、项目、公司或社区相匹配的组织用户名
- 类似 `@example-org/*` 的包作用域，仅应由
  相匹配的 ClawHub 所有者发布
- 疑似冒充某个项目的技能 slug 或插件包名称
- 涉及品牌、商标、项目重命名或包历史记录的争议
- 已删除、不活跃或无法联系的所有者，阻碍了合法的命名空间
  所有者

如果该条目除了所有权争议外还存在不安全、恶意或误导性问题，
还应遵循相关的审核或安全指南。命名空间声明
表单用于所有权审核，而非紧急漏洞披露。

## 提交前

首先确认你正在使用与命名空间相匹配的所有者进行发布。
对于插件包，类似 `@example-org/example-plugin` 的作用域名称必须
以相匹配的 `example-org` 所有者身份发布。

如果你可以管理当前所有者，请通过发布、重命名、转让、隐藏
或删除受影响的资源，直接修正命名空间。如果你无法管理
当前所有者，或需要工作人员解决争议，请提出声明。

## 应提供的证据

请使用公开的非敏感证据。有用的证明包括：

- GitHub 组织、仓库、发布版本或维护者历史记录
- 明确命名该命名空间的官方项目文档
- 域名或官方电子邮件域名证明
- 对 npm、PyPI、crates.io 或其他包注册表作用域的控制权
- 可安全公开讨论的商标、品牌或项目所有权证据
- 源代码仓库历史记录、包历史记录或公开的重命名公告
- 指向存在争议的 ClawHub 所有者、技能、插件、包或 issue 的链接

请说明每个链接能证明什么。工作人员应能理解其中的
关联，而无需获取私密凭据或机密信息。

## 不应提供的内容

请勿在公开的 GitHub issue 中发布机密信息或私密证明。请勿包含：

- API 令牌、签名密钥或凭据
- DNS 质询令牌
- 私密法律文件或合同
- 个人身份证明文件
- 私人邮件、私密安全报告或机密客户数据

声明表单会询问敏感证据是否需要通过工作人员的私密渠道提交。
请使用该选项，而不要公开发布敏感材料。

## 可能的处理结果

根据证据和风险，ClawHub 工作人员可能会保留命名空间、
转让所有权、重命名资源、隐藏或隔离现有条目、
添加别名或重定向、要求提供更多证明，或拒绝请求。

命名空间审核并不保证所有匹配的名称都会被转让。
工作人员会权衡公开证据、现有使用情况、安全风险和用户影响。
