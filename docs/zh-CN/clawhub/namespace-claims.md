# 组织和命名空间权属声明

ClawHub 使用所有者句柄、组织句柄、技能 slug、插件包名和
包作用域作为公共命名空间。如果某个命名空间看起来属于
现实世界中的项目、品牌、包生态系统或组织，但已在 ClawHub 上
被认领、保留、具有误导性或存在争议，请通过
[组织 / 命名空间权属声明 issue 表单](https://github.com/openclaw/clawhub/issues/new?template=org-namespace-claim.yml)
请求工作人员审核。

此路径用于公开、非敏感的所有权审核。不要将产品内
报告或账号申诉表单用于命名空间权属声明。

## 何时提交权属声明

当你认为 ClawHub 工作人员应审核某个命名空间是否应因现实世界中的所有权而被保留、转移、重命名、隐藏、隔离、设置别名或以其他方式变更时，请提交命名空间权属声明。

示例包括：

- 与你的 GitHub 组织、项目、公司或社区匹配的组织句柄
- 类似 `@example-org/*` 的包作用域，只应由匹配的
  ClawHub 所有者发布
- 看起来冒充某个项目的技能 slug 或插件包名
- 品牌、商标、项目重命名或包历史争议
- 已删除、不活跃或无法联系的所有者阻挡了合法的命名空间
  所有者

如果列表除所有权争议外还存在不安全、恶意或误导性问题，
也请遵循相关的审核或安全指南。命名空间权属声明
表单用于所有权审核，而不是紧急漏洞披露。

## 提交前

首先确认你正在使用与命名空间匹配的所有者发布。
对于插件包，类似 `@example-org/example-plugin` 的作用域名称必须作为匹配的 `example-org` 所有者发布。

如果你可以管理当前所有者，请通过发布、重命名、转移、隐藏或删除受影响资源来直接修复命名空间。当你无法管理当前所有者，或需要工作人员解决
争议时，再使用权属声明。

## 应包含的证据

使用公开、非敏感的证据。有帮助的证明包括：

- GitHub 组织、仓库、发布或维护者历史
- 标明该命名空间的官方项目文档
- 域名或官方邮箱域名证明
- npm、PyPI、crates.io 或其他包注册表作用域控制权
- 可以公开讨论的商标、品牌或项目所有权证据
- 源仓库历史、包历史或公开的重命名通知
- 指向存在争议的 ClawHub 所有者、技能、插件、包或 issue 的链接

请说明每个链接证明了什么。工作人员应能够在不需要私有凭据或密钥的情况下理解其中的
关系。

## 不应包含的内容

不要在公开的 GitHub issue 中放入密钥或私有证明。不要包含：

- API 令牌、签名密钥或凭据
- DNS 质询令牌
- 私有法律文件或合同
- 个人身份证明文件
- 私人邮件、私有安全报告或机密客户数据

权属声明表单会询问敏感证据是否需要私有工作人员渠道。
请使用该选项，而不是公开发布敏感材料。

## 可能结果

根据证据和风险，ClawHub 工作人员可能会保留某个命名空间、
转移所有权、重命名资源、隐藏或隔离现有列表、
添加别名或重定向、要求提供更多证明，或拒绝请求。

命名空间审核不保证每个匹配名称都会被转移。
工作人员会权衡公开证据、现有使用情况、安全风险和用户影响。
