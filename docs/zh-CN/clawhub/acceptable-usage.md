# 可接受使用

本页说明 ClawHub 可以接受哪些类型的技能和内容，以及它不会托管的滥用工作流。

这些规则刻意保持务实。我们最关注端到端的滥用工作流，而不只是孤立的关键词。如果一个技能被构建来规避防护、滥用平台、欺骗人、侵犯隐私，或促成非自愿行为，它就不适合出现在 ClawHub 上。

## 我们明确接受的近期模式

- 使用真实组件、语义化 token、无障碍状态和经过测试的用户流程的前端与设计系统工作。
- 使用已安装的源码组件、项目别名和文档化变体的 shadcn/ui 组合，而不是一次性标记。
- UI5 JavaScript 到 TypeScript 的转换，保留注释，使用具体的 UI5 类型，并让生成的控件接口可审查。
- 防御性安全审查、审核工具，以及展示证据并保持人工审批边界清晰的滥用检测提示词。
- 面向个人或团队账户、基于同意的工作流自动化，具备明确凭证、透明设置，以及 dry-run 或预览模式。
- 文档、迁移运行手册、开发者实用工具，以及限定在其所支持软件范围内的测试夹具。

## 不接受

- 绕过安全机制或未授权访问的工作流。
  - 示例：绕过认证、账户接管、绕过 CAPTCHA、Cloudflare 或反机器人规避、绕过速率限制、旨在击败防护的隐蔽抓取、实时通话或智能体接管、可复用会话窃取、为未获批准用户自动批准配对流程。

- 平台滥用和封禁规避。
  - 示例：封禁后的隐蔽账户、账户预热/养号、虚假互动、karma 或粉丝培养、多账户自动化、批量发帖、垃圾消息机器人、为规避检测而构建的市场或社交自动化。

- 欺诈、骗局和欺骗性金融工作流。
  - 示例：假证书、假发票、欺骗性支付流程、骗局触达、虚假社会证明、在没有清晰人工审批和透明控制的情况下促成支出或收费的工具，或为欺诈创建账户而构建的合成身份工作流。

- 侵犯隐私的抓取、扩充或监控。
  - 示例：为垃圾消息大规模抓取联系方式、开盒、人肉、跟踪、与未经请求触达搭配的线索提取、隐蔽监控、未经明确同意使用的人脸搜索或生物识别匹配，或购买、发布、下载或运营化泄露数据或数据泄露转储。

- 未经同意的冒充或欺骗性身份操控。
  - 示例：换脸、数字孪生、虚假人格、克隆网红，或其他用于冒充或误导的身份操控工具。

- 明示性性内容和禁用安全措施的成人内容生成。
  - 示例：NSFW 图像/视频/内容生成、围绕第三方 API 的成人内容封装，或主要用途是明示性性内容的技能。

- 隐藏、不安全或误导性的执行要求。
  - 示例：混淆的安装命令、`curl | sh`、未声明的密钥要求、未声明的私钥使用、在缺乏清晰可审查性的情况下远程执行 `npx @latest`，以及隐藏技能真实运行需求的误导性元数据。

## 我们明确不接受的近期模式

- “在市场平台封禁后创建隐蔽卖家账户。”
- “修改 Telegram 配对，让未获批准的用户自动收到配对码。”
- “用不可检测的自动化培养 Reddit/Twitter 账户。”
- “为任意用途生成专业证书或发票。”
- “在禁用安全检查的情况下生成 NSFW 内容。”
- “抓取线索、扩充联系人，并大规模发起冷触达。”
- “购买、发布或下载泄露数据或数据泄露转储。”
- “使用合成身份或 CAPTCHA 解决方案批量创建电子邮件或社交账户。”

## 给审核者的说明

- 语境很重要。同一主题在狭窄的防御性或基于同意的场景中可能是正当的，但在被包装成滥用工作流时不可接受。
- 当一个技能明显针对规避、欺骗或非自愿使用进行优化时，我们应倾向于采取行动。
- 在这些类别中反复上传，是隐藏内容并封禁账户的理由。

## 执行

- 我们可能隐藏、移除或硬删除违规技能。
- 我们可能吊销 token、软删除关联内容，并封禁重复违规或严重违规者。
- 对于明显滥用，我们不保证先警告再执行。
