本页记录了 2026 年 5 月 OpenClaw 性能、包大小、依赖项和 shrinkwrap 清理背后的证据。它是公开博客文章的技术配套说明。

这里合并了两项审计：

- **发布性能扫查：** 从 `v2026.5.28` 回溯到稳定版 `v2026.4.23` 的 GitHub Releases，使用 `OpenClaw Performance` 工作流、`profile=smoke`、mock-provider 线路。大多数标签行是单个样本；`v2026.5.27` 和 `v2026.5.28` 行使用最新的 repeat-3 发布分支工件。
- **更早的 4 月上下文：** 从 `v2026.4.1` 到 `v2026.5.2` 已发布的 `clawgrit-reports` mock-provider 基线，仅用于避免把 4 月下旬损坏的发布版本当作公开性能基线。
- **安装占用扫查：** 使用全新的 `npm install --ignore-scripts` 安装到临时包中，用 `du -sk node_modules` 统计大小，并遍历 `node_modules` 统计包实例数量。
- **npm 包大小扫查：** 对已发布版本运行 `npm pack openclaw@<version> --dry-run --json`，记录压缩 tarball 大小、解包大小和文件数量。

<Warning>
主性能扫查对每个标签使用一个 smoke 样本，但 `v2026.5.27` 和 `v2026.5.28` 行使用最新的 repeat-3 发布分支工件。更早的 4 月上下文使用来自 `clawgrit-reports` 的已发布 repeat-3 中位数。请把这些数字视为趋势证据和回归排查信号，而不是发布门禁统计数据。
</Warning>

### Shrinkwrap 边界

`2026.5.20` 发布时没有根 shrinkwrap，也没有大型嵌套 OpenClaw 依赖树。`2026.5.22` 引入了根 shrinkwrap，并在嵌套的 `openclaw/node_modules` 下安装了 911.8MB。`2026.5.28` 保留 shrinkwrap，并且仍会在嵌套的 `openclaw/node_modules` 下安装 259.7MiB，但在本地全新安装审计中不再安装任何 `@napi-rs/canvas` 软件包。

已发布 tarball 检查验证了这个边界：

| 版本        | 已发布稳定版？ | 根 `npm-shrinkwrap.json` | 备注                                  |
| ----------- | -------------- | ------------------------ | ------------------------------------- |
| `2026.5.20` | 是             | 否                       | shrinkwrap 之前的最后一个稳定版本     |
| `2026.5.21` | 否             | 不适用                   | 没有稳定 npm 版本                     |
| `2026.5.22` | 是             | 是                       | 引入 shrinkwrap                       |
| `2026.5.23` | 否             | 不适用                   | 没有稳定 npm 版本                     |
| `2026.5.24` | 否             | 不适用                   | 没有稳定 npm 版本                     |
| `2026.5.25` | 否             | 不适用                   | 没有稳定 npm 版本                     |
| `2026.5.26` | 是             | 是                       | 嵌套依赖树仍然存在                    |
| `2026.5.27` | 是             | 是                       | 嵌套依赖树仍然存在                    |
| `2026.5.28` | 是             | 是                       | 嵌套依赖树小得多                      |

重要区别是：**shrinkwrap 本身不是问题**。`v2026.5.28` 仍然发布根 shrinkwrap。问题在于软件包形态导致 npm 生成了大型嵌套 OpenClaw 依赖树以及全部 12 个 `@napi-rs/canvas` 平台软件包。`v2026.5.28` 中的嵌套树更小，并且 canvas 平台扇出不再出现在本地审计中。

有关 shrinkwrap 和维护者级软件包检查的通俗说明，请参阅 [npm shrinkwrap](/zh-CN/gateway/security/shrinkwrap)。

## 供应链解读

依赖数量是一项操作安全指标，而不仅仅是安装大小指标。每个软件包都会扩大操作员必须信任的维护者、tarball、传递更新、可选原生二进制文件和安装时行为的集合。

清理方向是：

- 将大型和可选能力保留在默认核心安装之外
- 让插件软件包拥有自己的运行时依赖图
- 避免在 Gateway 网关启动期间执行运行时软件包管理器修复
- 保持确定性安装，同时避免所有平台原生软件包物化
- 在软件包验收和测量路径中保持禁用安装脚本
- 在发布前捕获嵌套依赖树和原生可选依赖爆炸
