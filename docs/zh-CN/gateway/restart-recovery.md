重启 Gateway 网关不会丢失智能体状态。对话、转录记录、
定时作业、后台任务记录和排队中的出站消息都存储
在磁盘上；在轮次中途被中断的工作会被检测到，并在
Gateway 网关恢复运行后自动继续。无需人工干预，
也无需进行任何配置：恢复功能始终开启。

本页介绍重启后会保留哪些内容、如何检测被中断的工作，
以及自动继续的具体过程。

## 重启后会保留哪些内容

| 状态                          | 存储位置                                    | 重启期间的行为                                                          |
| ----------------------------- | ------------------------------------------- | ----------------------------------------------------------------------- |
| 对话历史                      | 每智能体 SQLite 数据库                      | 保持不变；会话从已存储的转录记录继续                                   |
| 被中断的主会话轮次            | 每智能体 SQLite 会话行和转录记录            | 启动几秒后自动继续或协调处理                                            |
| 子智能体运行                  | SQLite（共享状态数据库）                    | 启动时恢复注册表；继续被中断的运行                                     |
| 后台任务                      | SQLite（共享状态数据库）                    | 启动时协调处理；恢复孤立的运行或将其标记为丢失                         |
| 排队中的出站投递              | SQLite 投递队列                             | 重启后清空队列；重试尚未投递的回复                                     |
| 定时（cron）作业              | SQLite cron 存储                            | 调度计划持久保留；启动时调度器重新设定任务                             |
| 重启后续任务                  | SQLite 重启哨兵                             | 向请求重启的会话分派一次性后续轮次                                     |

## 优雅重启会先等待任务完成

请求重启（`openclaw gateway restart`、需要
重启的配置更改或 Gateway 网关更新）不会立即终止正在进行的工作。
Gateway 网关会停止接受新工作，然后等待活动的智能体轮次和
后台任务完成，最长不超过排空时限（默认为 5 分钟）。因此，大多数
重启根本不会中断任何工作。

只有无法在排空时限内完成的工作（或被强制重启或崩溃
中断的任何运行）才会被终止——在此之前，每个
受影响的会话都会被标记为待恢复。

## 如何检测被中断的工作

三种互补机制会标记轮次尚未完成的会话：

- **轮次准入时：**对于现有主会话中的普通文本轮次，
  Gateway 网关会在模型或
  `before_agent_reply` 钩子执行之前，通过一次 SQLite 事务追加用户消息、
  将会话标记为正在运行，并记录其恢复投递声明。Control UI 会在返回
  `started` 确认之前执行此操作；渠道分派则在准备好的轮次
  采用智能体运行时执行此操作。
  命令、附件、单轮次覆盖设置、待处理投递、先前的终止
  提示、插件所有的会话以及带有执行钩子的轮次会保留其
  专用准入路径。
  如果安装了 `before_agent_reply` 钩子，准入还会记录其阶段。
  恢复绝不会重放在调用中途被中断的钩子。未处理的钩子
  完成后，其检查点会记录该结果，但只要该钩子仍处于活动状态，
  恢复仍会采用失败关闭策略：检查点无法证明重启后加载的是同一
  插件代码和配置。已处理文本结果和静默结果会分别写入检查点，
  以确保确定性结算。
  旧版本写入的持久恢复声明没有来源所有权
  标记，因此升级期间也会对它们执行相同的失败关闭钩子检查。
- **关闭时：**在重启排空期间，任何存在活动运行的会话
  都会在运行终止前，在会话存储中写入恢复标记。
- **启动时：**Gateway 网关会扫描会话存储，查找仍
  声明正在运行、但在新进程中没有活动所有者的会话。这样可以捕获
  未执行任何关闭代码的严重崩溃和进程终止。同时还会清理过期的转录记录锁
  文件。

## 自动继续

启动几秒后，Gateway 网关会为每个已标记的会话重新分派任务，
并附上一条合成的系统消息，告知智能体其上一个轮次因重启而
中断，并要求它基于现有转录记录继续。如果最终回复
已经生成但尚未投递，其文本也会包含在内，
让智能体可以直接投递回复，而不是重新执行工作。恢复操作最多重试
3 次，并采用指数退避。每次重试都会复用同一个持久分派
标识符，因此即使连接失败的结果不明确，也不会重复启动同一恢复任务。
已完成和无法继续的 Control UI 轮次也会保留有界的持久
幂等墓碑，使重新连接的发件箱能够将它们清除，而不必
重新执行请求。

仅使用消息工具的回复会使用第二个持久关联。在同一对话中的终结性
发送到达渠道之前，Gateway 网关会在确切的会话和来源轮次上记录一项未解决的
投递意图。确认提供商成功后，该意图会解析为持久的已投递回执；
确认失败则会将其清除。恢复操作可以直接完成已投递回执，而无需重新运行工具。
如果崩溃导致提供商结果未知，恢复会采用失败关闭策略，而不会重放
外部效果。

已投递的回复也会连同其来源
消息 ID 镜像到转录记录中。终结性镜像使用独立的回执键，因此使用
相同提供商幂等键的进度发送无法掩盖终结标记。较早轮次的进度
发送和回执无法完成当前轮次。只有
持久的渠道入口声明才能恢复消息操作权限。恢复后的
运行会保留原始来源投递模式和来源关联，包括
请求者身份以及任何同渠道/线程限制，因此即使恢复期间再次发生重启，
同一回执仍具有权威性。无法重建渠道权限的
仅使用消息工具的轮次会采用失败关闭策略，并收到一次性的重新发送通知。

继续之前，Gateway 网关会检查转录记录尾部是否适合
继续。如果不适合（例如轮次结束于过期的待处理
审批），则不会盲目重新运行会话；智能体会改为发布一条简短
通知，要求用户重新发送最后一个请求。对于 WebChat，该通知会
直接写入会话历史记录，以便重新连接后仍然可见。

OpenClaw 还可以重建被中断的只读 [代码模式](/zh-CN/reference/code-mode)
工作。代码模式会将这些运行标记为重启安全，并在具有副作用的
目录工具或插件命名空间执行前拒绝它们。如果重启发生在
`wait` 控制处，新 Gateway 网关会根据转录记录重建轮次，
并强制重建后的执行保持重启安全，即使
模型省略或清除了该标志。主机会将整个重建后的
轮次限制为经过审计的只读核心工具和明确可安全重放的插件工具，
即使重启后禁用了代码模式也是如此。具有副作用的工作
仍由重新发送通知保护，避免重复写入的风险。

### 子智能体

子智能体运行会持久保存到共享 SQLite 状态数据库中，因此
子智能体注册表可以跨进程保留。启动时会恢复注册表，并使用原始任务上下文
继续被中断的子智能体会话。
有两项安全机制：

- 超过 2 小时前被中断的运行会被结束，而不是继续，以免
  停机一整夜的 Gateway 网关重新启动过期的工作。
- 反复恢复失败的会话会被设置墓碑并标记为卡死，以免
  恢复操作无限循环。

### 后台任务

[后台任务注册表](/zh-CN/automation/tasks)由 SQLite 支持，并会在启动时
及按固定时间间隔进行协调处理：恢复已完成运行所记录的持久结果，
而所属进程已消失的运行会在宽限期后被
标记为丢失，而不是永久挂起。

### 智能体请求的重启

当智能体本身触发重启（应用配置更改、更新
Gateway 网关或显式请求重启）时，会在进程退出前向
SQLite 写入一个重启哨兵。启动后，Gateway 网关会将结果发回
发起请求的聊天，并分派一个一次性后续轮次，使
智能体在同一渠道和线程中从中断处准确继续。

## 安全机制和可观测性

- **崩溃循环断路器：**5 分钟内发生 3 次非正常启动会触发断路器，
  在下一次启动时阻止自动启动辅助服务，避免崩溃的 Gateway 网关
  放大问题。非正常启动窗口清空后，它会自动恢复。
- **指标：**恢复活动通过
  [Prometheus](/zh-CN/gateway/prometheus) 导出为 `openclaw_session_recovery_total` 和
  `openclaw_session_recovery_age_seconds`。
- **日志：**恢复决策记录在
  `main-session-restart-recovery` 和 `subagent-interrupted-resume`
  子系统下。

## 不会继续的内容

- 因已有其他所有者负责处理而排除在主会话恢复之外的
  会话：子智能体会话（由子智能体恢复处理）、cron 会话（调度器
  按计划重新运行）以及 ACP 管理的会话（由已连接的 IDE
  或客户端负责继续）。
- 转录记录尾部无法安全继续的会话；这些会话会收到
  上述重新发送通知，而不是在无提示的情况下重新运行。
- 从未获得准入的工作：在排空窗口期间到达的消息会
  收到明确的重启错误并被拒绝，而不会被静默加入即将终止的进程
  队列中。
