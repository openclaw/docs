Os exemplos abaixo estão alinhados com o schema atual de configuração. Para a referência exaustiva e observações por campo, consulte [Configuração](/pt-BR/gateway/configuration).

## Início rápido

### Mínimo absoluto

```json5
{
  agent: { workspace: "~/.openclaw/workspace" },
  channels: { whatsapp: { allowFrom: ["+15555550123"] } },
}
```

Salve em `~/.openclaw/openclaw.json` e você poderá enviar DM ao bot a partir desse número.

### Configuração inicial recomendada

```json5
{
  identity: {
    name: "Clawd",
    theme: "assistente prestativo",
    emoji: "🦞",
  },
  agent: {
    workspace: "~/.openclaw/workspace",
    model: { primary: "anthropic/claude-sonnet-4-6" },
  },
  channels: {
    whatsapp: {
      allowFrom: ["+15555550123"],
      groups: { "*": { requireMention: true } },
    },
  },
}
```

## Padrões comuns

### Linha de base compartilhada de Skill com uma substituição

```json5
{
  agents: {
    defaults: {
      workspace: "~/.openclaw/workspace",
      skills: ["github", "weather"],
    },
    list: [
      { id: "main", default: true },
      { id: "docs", workspace: "~/.openclaw/workspace-docs", skills: ["docs-search"] },
    ],
  },
}
```

- `agents.defaults.skills` é a linha de base compartilhada.
- `agents.list[].skills` substitui essa linha de base para um agente.
- Use `skills: []` quando um agente não deve ver nenhuma Skill.

### Configuração multiplataforma

```json5
{
  agent: { workspace: "~/.openclaw/workspace" },
  channels: {
    whatsapp: { allowFrom: ["+15555550123"] },
    telegram: {
      enabled: true,
      botToken: "YOUR_TOKEN",
      allowFrom: ["123456789"],
    },
    discord: {
      enabled: true,
      token: "YOUR_TOKEN",
      dm: { allowFrom: ["123456789012345678"] },
    },
  },
}
```

### Aprovação automática em rede de Nodes confiáveis

Mantenha o pareamento de dispositivos manual, a menos que você controle o caminho da rede. Para um
laboratório dedicado ou sub-rede tailnet, você pode ativar a aprovação automática de dispositivo Node na primeira vez
com CIDRs ou IPs exatos:

```json5
{
  gateway: {
    nodes: {
      pairing: {
        autoApproveCidrs: ["192.168.1.0/24", "fd00:1234:5678::/64"],
      },
    },
  },
}
```

Isso permanece desativado quando não definido. Aplica-se apenas a pareamento novo de `role: node` sem
escopos solicitados. Clientes de operador/navegador e upgrades de função, escopo, metadados ou chave pública ainda exigem aprovação manual.

### Modo DM seguro (caixa de entrada compartilhada / DMs com vários usuários)

Se mais de uma pessoa puder enviar DM ao seu bot (múltiplas entradas em `allowFrom`, aprovações de pareamento para várias pessoas ou `dmPolicy: "open"`), ative o **modo DM seguro** para que DMs de remetentes diferentes não compartilhem um único contexto por padrão:

```json5
{
  // Modo DM seguro (recomendado para agentes de DM com vários usuários ou sensíveis)
  session: { dmScope: "per-channel-peer" },

  channels: {
    // Exemplo: caixa de entrada multiusuário do WhatsApp
    whatsapp: {
      dmPolicy: "allowlist",
      allowFrom: ["+15555550123", "+15555550124"],
    },

    // Exemplo: caixa de entrada multiusuário do Discord
    discord: {
      enabled: true,
      token: "YOUR_DISCORD_BOT_TOKEN",
      dm: { enabled: true, allowFrom: ["123456789012345678", "987654321098765432"] },
    },
  },
}
```

Para Discord/Slack/Google Chat/Microsoft Teams/Mattermost/IRC, a autorização do remetente é por ID primeiro, por padrão.
Ative a correspondência mutável direta por nome/email/apelido com `dangerouslyAllowNameMatching: true` de cada canal somente se você aceitar explicitamente esse risco.

## Dicas

- Se você definir `dmPolicy: "open"`, a lista `allowFrom` correspondente deve incluir `"*"`.
- Os IDs dos providers variam (números de telefone, IDs de usuário, IDs de canal). Use a documentação do provider para confirmar o formato.
- Seções opcionais para adicionar depois: `web`, `browser`, `ui`, `discovery`, `canvasHost`, `talk`, `signal`, `imessage`.
- Consulte [Providers](/pt-BR/providers) e [Solução de problemas](/pt-BR/gateway/troubleshooting) para observações de configuração mais detalhadas.
